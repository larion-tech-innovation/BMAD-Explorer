This skill generates production-ready visual assets from specifications using AI-powered creative tools. This is where specifications become pixels. Phase 4 told you precisely what each page and component should be; Phase 6 produces the artifacts — wireframes, full page compositions, UI elements, icons, imagery, video, and the strategic copy that fills them. The menu is non-linear: pick whichever activity you need, in any order, as many times as the work demands.

It reads four inputs: Phase 4 page specifications, Phase 7 design tokens (if available), visual direction from Phase 1 (style references, brand), and reference images provided by you. Outputs span wireframes, page design compositions, UI element assets (buttons, cards, forms), icon sets, photography / illustrations / backgrounds, video and motion content, written content (copy), and Figma files when the Figma export activity is used.

pattern Shared Generation Pattern 7 steps

Every visual activity (W, P, U, I, M, V) walks the same seven-step pattern. It is what keeps generation grounded: spec, style, and references converge into a prompt before a single pixel is produced.

1. Load Context     → Read relevant page specs, design system, visual direction
2. Asset Inventory  → Identify all assets needed (single or batch)
3. Select Style     → Choose design style + content style
4. Reference Images → Attach reference images for visual consistency
5. Craft Prompts    → Translate spec + style + references into generation prompts
6. Select Service   → Route to MCP service (Stitch, Nano Banana) or export prompts
7. Generate & Review → Execute, review results, iterate

§ WHY Why Asset Generation Works This Way — The reasoning behind each design choice

why-references Why Reference Images Are Critical

Without explicit reference images, AI generation drifts from your brand — it free-associates toward generic, statistically average output rather than your specific visual intent. References are the anchor. Collect 3–5 reference images before generating so every prompt has concrete visual ground to match. This is why step 4 of the shared pattern exists as its own step rather than being folded into prompt-crafting: references are non-negotiable, not optional polish.

why-batch Why Batch Mode?

When you generate many assets of the same type — say 20 icons — batch mode keeps the style consistent across the whole set. Generating them one at a time invites drift: each individual prompt subtly diverges, and the set ends up looking like it came from different hands. Batching holds style, weight, and treatment constant so the assets read as a coherent family.

why-after-specs Why This Phase Comes After Specs

Asset Generation is Phase 6 by design — it runs after specifications exist. Specs + style + references combine into grounded prompts rather than free-association. Generating before the spec is settled means producing pixels for decisions that have not been made yet, then regenerating once they are. By the time you reach this phase, Phase 4 has already decided what each page and component should be; Phase 6 only has to render it.

why-mcp Why Route Through MCP Services?

Stitch and Nano Banana are the primary integrated generation services, accessed via MCP in your AI coding environment. Routing through MCP keeps generation inside the same environment where the specs and design log live — the AI can read context and call the service directly, rather than asking you to copy prompts out to a separate tool and paste results back. When a service is unavailable, the pattern still produces craftable prompts you can export and run elsewhere.

§ HOW Workflows — Eight sub-workflows, one for each activity

Phase 6 is a multi-workflow phase. Each activity code (W, P, U, I, M, V, C, E) routes to its own dedicated workflow with its own step sequence. Stitch is a separate utility coordinator used across visual workflows. All workflows share the same facilitation discipline: read the complete step file, wait for user input at every menu, never skip steps.

workflow-wireframes Wireframes 5 steps

Generates structural outline wireframes that visualize page layouts from UX specifications. Use this activity to establish layout logic and information hierarchy before committing to full visual designs. Fidelity is selectable — Low-Fi (boxes and labels), Mid-Fi (recognizable components), or High-Fi (near-realistic with placeholder content) — and annotations can be switched on or off.

Step 1 · Load Context

Reads page specifications from the Scenarios folder, loads layout tokens from the design system (grid, gutters, spacing, breakpoints), and scans for any previously generated wireframes. Presents a context summary showing the pages to wireframe, grid configuration, and existing wireframe count before proceeding.

Step 2 · Asset Inventory

Lists all pages and views (desktop, mobile) identified in the specs, cross-references them against existing wireframes to flag what already exists and what is outdated, and presents scope options: generate all, select specific, or missing only. Waits for user scope selection before proceeding.

Step 3 · Select Style

Presents fidelity level options and loads the chosen design style from the style library to extract layout principles and spacing feel. Also confirms annotation preferences and output dimensions (desktop and mobile widths). Documents the complete wireframe style configuration before generation begins.

Step 4 · Generate Wireframes

Crafts a base prompt from context and style (fidelity, grid, content zones, style keywords, dimensions, grayscale palette), then customizes per page. Offers MCP generation or prompt export for external tools. Generates one wireframe at a time with per-page approval before continuing; chains approved results as references for batch consistency. Tracks progress per page and view.

Step 5 · Review and Iterate

Presents all wireframes grouped by page with desktop and mobile side by side. Runs consistency checks across the set: grid alignment, navigation placement, typography hierarchy, spacing uniformity, and annotation labels. Offers approve all, regenerate specific, style adjust, or edit annotations. Saves the approved set to E-Assets/wireframes/ and updates the design log.

workflow-page-designs Page Designs 5 steps

Generates complete full-page design compositions that bring UX specifications to life with real color, typography, imagery, and components. Use after wireframes are approved and the design system and visual direction are established. Requires specs, a complete design system, visual direction, and ideally approved wireframes as structural reference.

Step 1 · Load Context

Loads all four context sources: page specifications (full structure, user journeys, content, image placement), the complete design system (color, typography, components, spacing, elevation tokens), visual direction (brand guidelines, mood board, photography direction), and approved wireframes from the wireframes folder. Presents a context summary with pages to design, token count, wireframe availability, and content readiness per page.

Step 2 · Asset Inventory

Lists all pages with readiness assessment: wireframe approved, real copy available, source images available, all components defined. Flags pages blocked by missing assets and suggests the relevant activity (Images, Icons) to run first. Presents ready count, blocked count, and already-designed count alongside scope selection options.

Step 3 · Select Style

Presents available design styles and content styles from the style libraries, with suggestions matched to the project brand direction. Merges the selected style mood with design system color tokens, spacing feel with spacing tokens, and typography approach with the project's font stack. Confirms design style, content style (or photography-only mode), and output dimensions.

Step 4 · Generate Page Designs

Builds comprehensive prompts per page — layout (from wireframe or spec), hex colors from the design system, typography, style keywords, real headlines and body text, components, image areas, and dimensions. Attaches the approved wireframe as structural reference when available. Generates desktop first, presents for approval, then uses the approved desktop as reference for mobile. Chains approved pages for batch visual consistency across the set.

Step 5 · Review and Iterate

Shows all designs grouped by page as desktop + mobile pairs. Runs two review passes: design system compliance (colors, typography, spacing, components, responsive layout match tokens) and cross-page consistency (navigation identical, footer consistent, color usage and visual rhythm unified). Offers approve, regenerate, style adjust, detail edit, or side-by-side compare. Saves the approved set to E-Assets/page-designs/.

workflow-ui-elements UI Elements 5 steps

Generates UI component assets — buttons, cards, form inputs, navigation, feedback components — consistent with the design system. Covers all variants and states (default, hover, focus, active, disabled) for each component type. Use when you need design-system-compliant component visuals for prototyping, documentation, or handoff.

Step 1 · Load Context

Reads component definitions from the design system (button variants, card patterns, form elements, navigation, feedback components) and all design tokens affecting rendering (color per state, typography, spacing, border, shadow, transition tokens). Also reads page specs to identify which components appear on which pages. Presents a context summary with component type count, token count, states to generate, and referencing pages.

Step 2 · Asset Inventory

Creates a complete inventory table: component type, variant count, state count, and total assets (variants × states). Assigns priority levels — High (used every page: buttons, inputs, navigation), Medium (multiple pages: cards, alerts), Low (specific pages: specialized components). Checks for existing generated assets and presents scope options: all, high priority only, or select specific.

Step 3 · Select Style

Confirms the rendering approach: Vector/CSS (clean and scalable), Realistic (shadows and depth), or Flat (minimal, no shadows). Selects state visualization mode: Grid (all states of one variant in one image, design-system doc style), Individual (each state as a separate asset), or Animated (state transitions as a sequence). Maps design tokens to visual properties for the entire component set.

Step 4 · Generate UI Elements

Generates in priority order: buttons, form inputs, cards, navigation, feedback components. Applies batch strategy per visualization mode — Grid style generates all states in a single prompt; Individual style chains references for consistency. Requires group-level approval before continuing to the next component type. Tracks progress per component group with completion counts.

Step 5 · Review and Iterate

Reviews across three dimensions: design system compliance (exact token values for colors, typography, border radius, shadows, spacing, focus rings), cross-component consistency (visual weight balance, uniform radius and shadows, consistent disabled state treatment), and accessibility (WCAG AA contrast 4.5:1 text, 3:1 large text; visible focus states; distinguishable disabled states). Saves the approved library organized by type to E-Assets/ui-elements/.

workflow-icons Icons 5 steps

Generates consistent icon sets and individual icons matching the project's visual direction and design system. Covers navigation, action, status, feature, social, and decorative icon categories. Batch-generates by group (navigation together, actions together, etc.) to ensure the full set reads as a coherent family. Output is SVG primary with PNG fallbacks at multiple sizes.

Step 1 · Load Context

Reads every icon reference from across all page specs, organized by category. Loads icon-related design tokens (sizes 16/24/32/48px, stroke width, color mode, container type). Checks for existing generated icons and external icon library references in the design system. Presents a context summary with total icon count, categories, existing assets, primary size, and style direction from the design system.

Step 2 · Asset Inventory

Builds a complete deduplicated icon catalog: icon name, category, pages used on, sizes needed. Merges icons that appear across multiple pages, notes all size variants, and identifies similar icons that could share a base (arrow families, CRUD action sets). Calculates total unique icons and total assets (unique × size variants). Presents scope options: all, by category, select individual, or priority first (navigation + actions).

Step 3 · Select Style

Defines the complete icon style — Outline, Filled, Duotone, or Glyph — with detailed parameters per choice (stroke width, line cap, corner radius for outline; fill style for filled; opacity for duotone). Defines grid and alignment rules: canvas size with padding, live area, and pixel grid alignment. Selects color treatment: Monochrome (currentColor), Brand colors by category, or Fixed hex per icon. Confirms output format (SVG primary, PNG fallback).

Step 4 · Generate Icons

Constructs a base prompt from the style configuration and generates icons by group for maximum consistency: navigation set first, then action set, status set, feature set, and social set. Within each group: generates the first icon, gets approval, then uses it as reference for the rest of the group. Optimizes SVG output — clean markup, correct viewBox, currentColor for monochrome, pixel alignment. Offers MCP generation, prompt export, or open-source library match.

Step 5 · Review and Iterate

Presents the full icon set in a grid organized by category, shown at multiple sizes (sm, md, lg) on both dark and light backgrounds. Checks consistency across the set: uniform stroke weight, balanced visual weight, consistent corner radius and detail level, uniform padding, recognizable at smallest size. Also runs a metaphor clarity check: each icon clearly communicates its meaning without ambiguity. Generates size variants (SVG, PNG at 1×/2×/3× for each defined size) after approval. Saves to E-Assets/icons/.

workflow-images Images 6 steps

Generates production-quality images — hero shots, product photos, illustrations, backgrounds, team photos — consistent with the project's visual direction. This workflow has a dedicated References step (step 4) because image generation is where reference chaining matters most: each approved image in a batch becomes the reference for the next, keeping color temperature, lighting direction, and style locked across the set.

Step 1 · Load Context

Loads five context sources: page specifications (image placement, content story, dimensional requirements, alt text), visual direction (brand photography guidelines, color palette, mood, subject preferences, what to avoid), design system (aspect ratios, border radius, overlay treatments, container sizes), existing brand assets and approved images. Presents a context summary with image count, categories, reusable existing assets, and generation needed count.

Step 2 · Asset Inventory

Catalogs all image placements across every page: image name, page, type (hero/product/team/background/illustration/decorative), dimensions, and content description. Groups by type to identify batch opportunities — all hero images form one batch, all team photos share one lighting setup, all product images share one visual treatment. Presents scope options: all, by batch, select specific, or priority (hero and above-fold first).

Step 3 · Select Style

Loads content styles and allows different styles per batch — hero images may be photorealistic while backgrounds are watercolor or flat design. Configures visual parameters per batch: lighting (natural, studio, dramatic, golden hour), color harmony (warm, cool, brand-matched), composition (rule of thirds, centered, dynamic), and mood (professional, energetic, calm, luxurious). Confirms style per batch as a table before proceeding.

Step 4 · Reference Images

Gathers reference images from all available sources: brand photography from the client, mood board imagery, previously approved generated images, and style examples from the content style library. Assigns 1–3 references per batch with stated purpose (mood direction, framing, texture treatment). Defines the reference chaining strategy for batch generation: generate first image, get approval, chain it as reference for image 2, continue through the batch. Documents the fallback strategy for divergent images.

Step 5 · Generate Images

Builds prompts per image: content style, subject, mood, lighting, hex color palette, composition, dimensions, style keywords, reference attachment, and negative prompts. Starts each batch with the hero/anchor image and waits for approval before continuing. Chains approved results as references for subsequent images in the batch. Offers variation counts for key images (1 best, 3 options, or 5 options). Tracks progress per batch with completion counts.

Step 6 · Review and Iterate

Presents the full image gallery grouped by batch/type at intended display size, with page context shown. Checks four dimensions: batch consistency (color temperature, lighting direction, detail level uniform across each batch), brand alignment (color harmonizes with brand, mood matches visual direction, no unintended text or artifacts, cultural sensitivity), technical quality (resolution, no AI artifacts, clean edges, compression-friendly), and overall cohesion across all batches. Saves organized by type to E-Assets/images/.

workflow-videos Videos 5 steps

Generates motion content, animations, and video assets across four complexity levels: Simple (CSS/SVG animations, under 10KB), Medium (Lottie/JSON animations, under 50KB), Complex (produced video with storyboard guidance, under 10MB), and AI Generated (AI video via MCP, under 2MB). Routes each asset to the appropriate tool based on complexity. Accessibility is non-optional — prefers-reduced-motion coverage is verified before final approval.

Step 1 · Load Context

Identifies all motion content requirements from page specs: hero animations, product demonstrations, micro-interactions, background video, and explainer sequences. Loads motion tokens from the design system (duration scale, easing curves, transition types). Checks static assets (images, UI components, illustrations) that motion builds upon. Presents a context summary showing motion asset count, types, duration range, static assets to animate, and full video productions.

Step 2 · Asset Inventory

Catalogs all motion assets with technical requirements: asset name, page, type, duration, and format (MP4/WebM, CSS/Lottie, SVG animation). Categorizes by complexity — Simple, Medium, Complex, and AI Generated — and documents format and file size target for each level. Presents scope options: all, by type, select specific, or priority (hero and above-fold only).

Step 3 · Select Style

Defines the motion personality: Subtle (corporate, medical), Fluid (wellness, lifestyle), Energetic (startup, gaming), or Precise (engineering, SaaS). Configures timing parameters based on the personality choice (base duration, easing curve, stagger delay, loop delay). For produced or AI-generated video, selects a video treatment: Cinematic (shallow DOF, color graded), Documentary (natural, handheld), Motion design (graphics-driven), or Abstract (textures, ambient). Confirms the complete motion style configuration.

Step 4 · Generate Motion Content

Routes each asset to the correct tool by complexity: Simple assets get CSS/SVG keyframe animation code; Medium assets get Lottie JSON descriptions (or MCP generation if supported); Complex assets receive a storyboard, shot list, and production guide; AI Generated assets get video generation prompts with reference frames from approved static images. Previews each asset in page context and checks timing and feel before approving. Tracks progress per complexity group.

Step 5 · Review and Iterate

Previews all motion content in isolation, in page context, and as before/after (static vs. animated). Checks four dimensions: consistency (timing and easing curves uniform, motion direction logical, loops seamless), performance (file sizes within targets, GPU-accelerated CSS, videos compressed, lazy loading for below-fold), accessibility (prefers-reduced-motion respected, no flashing above 3/second, readability unaffected, pause/stop provided for video, static fallback available), and UX quality. Saves organized by type to E-Assets/motion/.

workflow-content Content 7 steps

Generates strategically grounded text content using the Five-Model Framework: Content Purpose, Trigger Map, Customer Awareness Cycle, Action Mapping (Cathy Moore), Badass Users (Kathy Sierra), and Golden Circle (Simon Sinek). Every word must earn its place — this workflow rejects vague goals and builds content decisions from measurable outcomes before writing a single word. Use for landing page copy, CTAs, error messages, feature descriptions, mission statements — any text where psychology and persuasion matter.

Step 0 · Define Content Purpose

Establishes a clear, testable purpose statement before any strategy is applied. Asks what content is being created, what it must accomplish (outcome-focused, not descriptive), who will read it and what state they are in, and how success will be measured. Runs the Model Priority Matrix to select which of the five strategic models matter most for this specific content type. Documents the purpose definition in structured YAML format.

Step 1 · Load Trigger Map Context

Loads the relevant Trigger Map context — the business goal, target persona, driving forces (positive wishes and negative fears), and the customer awareness journey (start level and end level). If no Trigger Map exists from Phase 2, flags the gap and offers to proceed with whatever Product Brief context is available. Confirms the strategic context with the user before documenting it for traceability.

Step 2 · Awareness Strategy

Translates the Trigger Map's awareness positioning into a concrete content strategy across six dimensions: validates the starting awareness level, clarifies the target awareness shift, determines awareness-appropriate language (what words to use and avoid), sets information priorities (essential vs. overwhelming), identifies credibility requirements (personal story vs. expert credentials vs. social proof vs. data), and maps the emotional journey from starting emotion to ending emotion. Documents the complete awareness strategy in structured format.

Step 3 · Action Filter

Applies Action Mapping (Cathy Moore) to identify the specific behavior the user must be able to perform after reading — not vague "understanding" but concrete action ("click the signup button with confidence," "choose the right pricing tier in under 30 seconds"). Traces the action to the Trigger Map business goal and to the persona's driving forces. Works backward from the action to determine what information is essential versus nice-to-know, and identifies content barriers (confusion, fear, effort, distrust, irrelevance) with solutions for each.

Step 4 · Empowerment Frame

Applies Kathy Sierra's Badass Users framework to reframe all content from product-feature-focused to user-capability-focused. Defines the user's current state (struggles, feelings, limitations) and their "badass state" after success (what they can do, how they feel). Identifies the "aha moment" — the perspective shift that unlocks confidence, not just understanding. Rewrites feature-focused language to capability-focused language, makes the transformation path visible and achievable, and identifies cognitive load risks (too many choices, jargon, unclear next steps) with simplifications.

Step 5 · Structural Order

Applies Simon Sinek's Golden Circle to sequence all content elements from Steps 3–4 into a WHY → HOW → WHAT persuasive flow. Maps each content element to its section: WHY (emotional opening that connects to driving forces), HOW (method that bridges emotion to specific solution, contains the aha moment), WHAT (concrete specifics, proof, CTA, risk removal). Sequences within each section in the most persuasive order and validates that the flow feels natural — emotional first, logical second, action last.

Step 6 · Generate and Review Content

Synthesizes all strategic context from Steps 0–5 and generates 2–3 variations that differ in which driving forces they emphasize: Variation A (wish/aspiration-focused), Variation B (fear/pain-avoidance-focused), Variation C (balanced blend). Presents each variation with clear rationale for strategic choices. Guides the user through selection, combination, and refinement. Validates that the final content enables the required action, completes the awareness journey, addresses all barriers, and includes the aha moment. Documents the final content with full strategic traceability.

workflow-figma Figma Export 5 steps

Sends code implementations to Figma for design review, documentation, and visual iteration via the html.to.design MCP server (mcp2_import-html). Use when component states need visual documentation, design system components require Figma library representation, prototype pages need designer review, or design-code parity documentation is needed. Requires Figma account access and the html.to.design plugin installed in Figma.

Step 1 · Connection Check

Verifies that the mcp2_import-html MCP tool from the html.to.design server is accessible in the current session. If not available, guides through setup (Figma account, plugin installation, MCP server configuration in the IDE). Executes a test export and asks the user to confirm the test layer is visible in their Figma file before proceeding. Does not advance until connection is verified.

Step 2 · Identify Export Type

Classifies the export into one of three scenarios using a decision tree: Prototype Page Export (full page or screen, multiple sections, user flow — IDs follow {page}-{section}-{element}), Design System Component (component states, design system docs, reusable component — IDs follow {component}-{variant}-{state}), or Frontend View/Component Block (visual adjustments, spacing iteration, specific UI block — IDs follow {component}-{element}-{descriptor}). Confirms scenario type with the user before proceeding.

Step 3 · Prepare Specifications

Searches for existing specification files containing OBJECT IDs in the Scenarios and Design System folders. If specifications exist, extracts all OBJECT IDs and maps them to components in scope. If no specifications exist, offers to analyze the code and reverse-engineer specifications, generate OBJECT IDs following project conventions, and create a specification document for review. Validates 100% coverage — every component in scope has an OBJECT ID — before proceeding.

Step 4 · Generate and Validate

Generates clean, Figma-compatible HTML with proper OBJECT IDs from the specification mapping. Applies Figma-compatible styling rules: Google Fonts only, all colors as hex values (no CSS variables), all spacing in pixels (no rem/em), simple flexbox/grid layout, inline or style-block CSS. Runs five validation checks: specification coverage, ID naming convention, HTML structure, styling compatibility, and content completeness. Resolves all errors before the export is executed — this step ends with a validated HTML payload, not a Figma layer.

Step 5 · Execute Export

Calls mcp2_import-html with the validated HTML and a descriptive layer name ({Component/Page Name} - {Purpose}). Waits for the MCP response, verifies the node ID was returned without errors, then asks the user to confirm the export is visible in their Figma file with proper layer names matching the OBJECT IDs. Provides troubleshooting guidance if the export is not visible. Documents the complete export summary and updates the design log before returning to the activity menu.

Utility Coordinator workflow-stitch · Google Stitch AI

Stitch is not a menu activity — it is a utility coordinator used within visual workflows (W, P, U) when MCP-powered generation is routed through Google Stitch. The dispatcher workflow routes to workflow-stitch.md when the user selects the Stitch service at the "Select Service" step of any visual activity.

Stitch transforms text prompts, sketches, and reference images into responsive UI designs at stitch.withgoogle.com. The input formula is: Visual Reference + Sketch + Specification = Stitch Generation. Output is exportable to Figma or as HTML/CSS. Stitch works best with detailed specifications, an existing visual reference or screenshot, and a sketch showing layout structure. It does well with single screen generation and clean HTML/CSS export, but works best 2–3 screens at a time.

The Stitch coordinator manages prompt preparation (using templates/stitch-prompt.template.md — never raw specs pasted directly), generation of 2–3 variants, review against spec, and export to Figma or HTML/CSS stored in the spec's Visual-Design/ folder. After Stitch generation, the original specification is updated to reference the Stitch output.
