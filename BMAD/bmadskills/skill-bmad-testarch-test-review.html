§ WHAT Test Quality Review — What does this Skill do?

Skill code
bmad-testarch-test-review  ·  Menu code: TR
Goal
Audit existing test files against TEA quality best practices. Dispatch four parallel quality workers, aggregate their scores into a weighted overall rating, and produce a prioritised list of violations with exact fix suggestions.
Role
Master Test Architect. Acts as a read-only code reviewer — it never modifies or generates test files. It finds quality issues and tells you how to fix them; the actual fixing is for the developer or the bmad-testarch-automate skill.
Distinct from
test-review asks: "Are the tests we have written well?" — are they reliable, independent, readable, and fast? It does not check whether you have tests for all features; that is a separate skill (trace). Coverage is intentionally out of scope here.
Stack detection
Automatically detects whether your project is Frontend, Backend, or Fullstack and loads the right quality rules for your stack. No configuration needed.
Output
test-review.md — overall score, dimension breakdown, violations with file:line references and fixes, top-10 recommendations, recommended next workflow

Review Scope

Set by the review_scope variable. Default is single. If unclear, the skill asks the user before proceeding.

single

One File

Provide a specific file path. Best for reviewing a single test file before merging a PR or after a teammate flags it for quality.
directory

Folder

Review all tests under a chosen folder (e.g. tests/e2e/). Useful for auditing a feature area or a specific test level.
suite

All Tests

Glob all tests across the entire repo. Best for a periodic quality health-check or pre-release audit. May take longer for large suites.

4 Quality Workers — Architecture Overview

Step 3 dispatches four specialised workers in parallel. Each evaluates exactly one quality dimension and writes a structured JSON result to a temp file. Step 3F aggregates all four results after all workers complete.

  • Date.now() / new Date() without mocking

Parallel Safety & Independence

Knowledge Base — What the Skill Knows

Before reviewing your tests, the skill automatically loads the right quality rules for your project. It detects your stack type (Frontend / Backend / Fullstack) and selects the relevant rules — so a Playwright frontend project gets different checks than a Java backend project.

Your project typeWhat gets loaded
All projects Core quality rules: test reliability, data isolation, selector stability, timing best practices
Frontend / Fullstack (Playwright) Additional Playwright-specific rules: network interception, auth session handling, fixture composition
Contract tests (Pact) Consumer-driven contract rules: Pact configuration checks, determinism rules for the Rust FFI layer
Browser evidence mode When enabled: the skill opens your live app to capture screenshots and network logs as review evidence

§ WHY Why Test Quality Review?

A passing test suite is not the same as a trustworthy test suite. Tests can be green today and silently flaky tomorrow — returning false positives in CI, blocking releases for random failures, or becoming unmaintainable as features evolve. The test-review skill turns subjective "code smell" judgements into a measurable, reproducible quality score.

Dimension Without Test Review With Test Review (TR)
Flakiness Tests pass locally but fail intermittently in CI — root cause unknown Determinism worker flags every Math.random(), unmocked API, and hard wait with an exact fix
Parallel safety Tests conflict when run in parallel — forced to run serially, slowing CI Isolation worker surfaces shared state, order dependencies, and missing cleanup
Maintainability Tests rot as features change — nobody understands what a 300-line test file does Maintainability worker enforces naming, structure limits, and no-duplication rules
CI cost Serialised test suites waste CI minutes on unnecessary .serial blocks Performance worker identifies unnecessary serialisation and expensive per-test setup
Quality signal No objective baseline — "tests look fine" is subjective 0–100 score with A–F grade per dimension and overall — reproducible across sprints
Fix guidance Developer must know best practices from memory to spot violations Every violation includes file:line reference, code pattern, and exact fix recommendation
Contract tests Pact FFI non-determinism bugs silently drop interactions on Linux CI Dedicated CDC checks surface missing fileParallelism: false and singleFork config before CI breaks

Workflow Steps — Create Mode (4 Steps)

STEP 01 Load Context & Knowledge Base step-01-load-context.md
  • Determine review scope (single / directory / suite); ask user if unclear
  • Detect stack: scan playwright.config.*, pom.xml, go.mod etc. → frontend / backend / fullstack
  • Load tiered knowledge: Core always (test-quality, data-factories, test-levels-framework, selective-testing, test-healing-patterns, selector-resilience, timing-debugging); Extended on demand; Playwright Utils profile based on stack and whether tests use page.goto / page.locator
  • Load Pact.js Utils fragments if tea_use_pactjs_utils enabled and contract tests are in review scope
  • Gather optional context artifacts: story file (acceptance criteria), test-design doc (priorities), framework config
STEP 02 Discover & Parse Tests step-02-discover-tests.md
  • Collect test files matching the chosen scope; halt if no tests found
  • Parse metadata per file: size/line count, framework detected, describe/test block counts, test IDs, priority markers, imports, fixtures, factories, network interception, waits/timeouts, control flow (if/try-catch)
  • Optional evidence collection (when tea_browser_automation is cli or auto): open browser, start tracing, capture screenshot, record network log, then close session with playwright-cli -s=tea-review close
  • Trace analysis: npx playwright trace open <trace.zip> for failing assertion root-cause
STEP 03 Orchestrate Quality Evaluation (4 Workers) step-03-quality-evaluation.md
  • Resolve execution mode: agent-team → subagent → sequential (capability probe)
  • Dispatch all 4 quality workers in parallel (or sequentially if subagents unavailable); each writes a JSON result file
  • Wait for all 4 workers to complete; verify all 4 output files exist before proceeding
  • Sub-step 3F: Read all 4 outputs, calculate weighted overall score, aggregate violations, prioritise top-10 recommendations
  • Parallel execution gain: approximately 60–70% faster than sequential when agent-team or subagent mode is available
STEP 04 Generate Report & Validate step-04-generate-report.md
  • Produce test-review.md using the review template: overall score + grade, dimension breakdown, critical findings with fixes, warnings, context references
  • Include coverage boundary note: test-review does not score coverage; direct coverage findings to trace
  • Polish output: remove duplicates, verify consistency, fill any gaps marked N/A
  • Validate against checklist.md; fix any gaps before finalising
  • Summarise scope reviewed, overall score, critical blockers, and recommended next workflow

Determinism — Weight 30%

The highest-weighted dimension because non-deterministic tests are the root cause of flakiness — tests that pass locally but fail in CI, or fail intermittently with no reproducible reason. The worker scans for patterns that make test outcomes depend on anything outside the test's explicit inputs.

CategorySeverityPatternFix
Random generation HIGH Math.random() Use faker.seed(12345) for deterministic random data
Time dependency HIGH Date.now(), new Date() Mock system time with test.useFakeTimers() or use fixed timestamps
External API HIGH Real HTTP calls in tests Intercept with page.route() or mock the HTTP client
Hard wait MEDIUM waitForTimeout(N) Replace with expect(locator).toBeVisible() or network-first intercept
Flaky selector MEDIUM CSS class selectors Use getByRole, getByLabel, data-testid

Pact FFI Checks — Contract Test Projects Only

If your project uses Pact (consumer-driven contract testing), the skill runs an extra set of checks on top of the standard determinism rules. These checks target a specific class of bug: contract interactions being silently dropped when tests run in parallel — failures that only appear on Linux CI, never on a developer's machine, making them very difficult to track down.

When the skill reports one of these violations, escalate to a developer — the fix is always a configuration change, not a change to the test logic itself.

CDC Pact FFI Determinism Rules
  • HIGHMultiple API contracts defined inside one test block. Each test should cover exactly one API interaction. When multiple are stacked in a single test, the Pact engine can silently drop some of them — the test passes but the contract is never actually verified.
  • HIGHAll contract tests must run one at a time (serially). Running them in parallel causes them to write to the same contract file at the same time, resulting in a corrupted or incomplete contract. → Open vitest.config.pact.ts and make sure fileParallelism: false is set.
  • HIGHContract tests are missing the "single process" mode. When the project has more than one contract test file for the same consumer/provider pair, they must all run inside a single process to avoid race conditions. → singleFork: true in the Pact test config.
  • HIGHProvider-side contract tests have the same "single process" problem. Same issue as above but on the provider (API server) side of the contract. → Ask a developer to enable singleFork: true in the provider test config.
  • HIGHGlobal test config has concurrency enabled, overriding contract test settings. A project-wide setting that allows tests to run simultaneously is cancelling out the serial-execution requirement for contract tests. → Ask a developer to check for conflicting concurrency settings in the shared config.
  • HIGHThe contract test run script is missing a determinism safety check. There is no automated guard that verifies the contract test configuration is safe before the tests run. → Ask a developer to add the determinism check script to the CI pipeline.
  • LOWContract test config is split across multiple files. The skill cannot fully trace merged settings when config is spread across several files. Advisory only — ask a developer to consolidate the contract test config into one place.

Isolation — Weight 30%

Tests must be fully independent — able to run in any order, in parallel, on any machine, without needing another test to have run first. The isolation worker catches shared state, cleanup gaps, and order dependencies.

What "Isolated" Means
  • Each test creates its own data in beforeEach using factories — never relies on data a prior test created
  • Each test cleans up its data in the fixture teardown (afterEach or await use(data) pattern)
  • No global variables modified within a test unless restored in teardown
  • All tests can run with test.describe.configure({ mode: 'parallel' }) without conflicts
ViolationSeverityFix
Global state mutations between testsHIGHMove state into beforeEach / afterEach scope
Test order dependencies (B depends on A)HIGHEach test must set up its own preconditions
Shared database records without cleanupHIGHUse factories with teardown in fixture await use()
beforeAll leaking side effectsHIGHMove to beforeEach or add matching afterAll cleanup
Missing cleanup for created dataMEDIUMAdd afterEach delete or use factory auto-cleanup
Environment variables modified without restoreMEDIUMSave original value in beforeEach, restore in afterEach

Maintainability — Weight 25%

Well-maintained tests are easy to read, understand, and modify when the underlying feature changes. The maintainability worker checks structure, naming, and code hygiene.

Key Standards Enforced
  • Test names must follow Given-When-Then structure: "[P0] should reject login when password is incorrect"
  • All tests grouped in test.describe blocks — no bare top-level test() calls
  • File length under 100 lines — split by feature area if longer
  • No hardcoded test data — use factory functions with faker
  • No page object classes — tests should be direct and simple
  • No duplicate logic — extract into helpers but keep the test body readable

Performance — Weight 15%

Slow test suites reduce developer feedback speed and increase CI costs. The performance worker identifies tests that are unnecessarily serialised, run expensive setup on every test, or block parallel execution.

Common Performance Wins
  • Remove .serial from test.describe.serial unless tests genuinely share stateful resources
  • Move expensive setup (DB seed, user creation) into beforeAll + teardown when tests only read data
  • Use fixture reuse — create the auth state once per worker, not per test
  • Replace page.$$ with Playwright locators (lazy, auto-waiting, faster)

Score Calculation

Each worker produces a dimension score (0–100) by penalising violations according to their severity. The overall score is a weighted average across the four dimensions.

Violation Penalty Points

HIGH −10 pts
Per violation. Flakiness root causes, test order deps, broken Pact configs, missing cleanup for shared state.
MEDIUM −5 pts
Per violation. Hard waits, flaky selectors, missing cleanup for non-critical data, inconsistent naming.
LOW −2 pts
Per violation. Style issues, advisory improvements, future-proofing notes (Pact pool advisory for single-file suites).

Score formula per dimension: score = max(0, 100 − Σ(penalty per violation))

Grade System

Each dimension and the overall score receive a letter grade. The grade reflects quality health at a glance without reading the full violation list.

A
90 – 100
B
80 – 89
C
70 – 79
D
60 – 69
F
0 – 59

The report also surfaces the top-10 prioritised recommendations — sorted HIGH impact first — across all four dimensions. Dimension scores below 70 automatically elevate their findings to HIGH impact in the recommendation list.

What to do with the grade
  • A / B — Test suite is in good shape. Minor clean-up optional.
  • C — Actionable improvements exist; address HIGH violations before next sprint.
  • D / F — Significant quality debt. HIGH violations should be treated as blockers; run bmad-testarch-automate to regenerate tests with correct patterns, or fix manually guided by the report.
