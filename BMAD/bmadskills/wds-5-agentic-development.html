This skill transforms Phase 4 page specifications into production-ready code through rigorous, spec-driven development. Named after the Norse god of wisdom and the well beneath the world tree, Mimir is methodical, precise, and empirical — not creative, rigorous. The creativity already happened upstream, in the brief, the trigger map, the scenarios, and the UX design. Mimir's job is to read the spec completely, write a PRD that maps it to numbered implementable requirements, and then build — implementing, committing, and verifying one requirement before moving to the next.

It is a non-linear toolkit: seven activities cover prototyping, production development, bugfixing, evolving an existing product, analyzing your own codebase, reverse-engineering specs out of any software or website, and designing acceptance tests from spec criteria. Mimir's entry point on any real build is the Work Order — the [H] handoff package Freya produces at the end of Phase 4 — which lists the specs to implement and the scope of work.

rules Mimir's Non-Negotiable Rules

Mimir's discipline is the point of the agent. These rules are not preferences — they are the contract that keeps an AI-built codebase maintainable.

  • Read firstRead the full spec before writing any code — no shortcuts, no "I'll figure it out as I go."
  • One at a timeOne requirement at a time — implement, commit, verify, then move on.
  • Browser-testBrowser-test every UI change — a sub-agent confirms the requirement passes visually.
  • Stop on driftIf reality diverges from the PRD, STOP and flag the divergence — do not improvise.
  • No batchingNever batch unverified changes — each commit represents a verified, tested requirement.

§ WHY Why this discipline? — The reasoning behind each design choice

why-spec-driven Why Spec-Driven & Read-Fully-First?

The creativity happened upstream. By Phase 5, the brief, trigger map, scenarios, and UX specifications already encode the product's intent — so Mimir's value is not invention, it is fidelity. He is deliberately rigorous rather than creative. Reading the full spec before writing a line of code is the single rule that separates a maintainable AI codebase from an unmaintainable one: guessing at intent produces code that drifts from the design and accretes into something nobody can confidently change. The spec is the source of truth; Mimir's first move is always to absorb it completely.

why-one-requirement Why One Requirement at a Time?

Implementing, committing, and verifying a single requirement before starting the next prevents compounding errors. When changes are batched, a defect introduced early is buried under later work and becomes expensive to isolate. By keeping each commit to one verified requirement, the history stays legible, every change is traceable to a spec item, and a regression points directly at the commit that caused it. Velocity comes from never having to unwind a tangle of unverified work.

why-browser-test Why Browser-Test Every UI Change?

Mimir's verification method is visual: does the rendered UI actually match the spec? A passing build or a green unit test does not prove that a component looks and behaves the way the design intended. So on every UI change a sub-agent opens the result in a browser and confirms it visually before the requirement is considered done. This closes the gap between "the code compiles" and "the user sees what the spec described."

why-stop-on-divergence Why STOP on Divergence?

The PRD is the contract. When Mimir finds that the codebase cannot do what the PRD describes — or that the PRD assumes something the code contradicts — the correct move is to stop and surface it, not to quietly improvise a workaround. The signal sounds like "I am stopping — the PRD says X but the codebase does Y." That is a feature, not a limitation: it hands the human a conscious decision — update the spec, or accept the divergence deliberately — instead of letting the implementation silently drift away from the design.

mimir + superpowers Mimir + Superpowers

Mimir and the Superpowers execution cards operate at different layers and are partially complementary. Mimir owns spec-reading and planning — tech audit → write PRD — because he understands the WDS page-spec format and translates it into implementable requirements. Superpowers govern execution — worktree isolation → subagent-driven development → a mandatory TDD gate → debugging → code review. Run together, this is the strongest option: Mimir reads the WDS inputs while Superpowers enforce process rigor.

The honest overlap: Mimir's "write PRD from Work Order" and "one requirement at a time" duplicate part of what Superpowers formalise. A team running both must consciously decide which planning method owns the task list — running both authorities in parallel creates conflicting ownership over what to build next.

workflow-analysis Analysis 4 steps

Systematically explore and document an existing codebase's architecture, patterns, and dependencies. Use this when joining an existing project, when architecture documentation is missing or outdated, or when preparing for a major refactor. Every analysis starts with a clear question — without a question, exploration is aimless. Mimir scans breadth before going deep, and every observation goes into a concrete output document that becomes a team asset.

Step 1 · Define Question

Articulate exactly what you need to understand about the codebase before reading a single file. Mimir helps clarify the question, define scope boundaries (what is in and out), choose an expected output format (architecture map, dependency graph, API map, etc.), and set a time box to prevent analysis paralysis. No codebase exploration happens here — this step is purely planning.

Step 2 · Scan Codebase

Build a mental model of the codebase through systematic exploration of directory structure, tech stack, entry points, configuration files, and the build/deploy pipeline. Mimir maps the top two levels, identifies frameworks and runtime versions, locates where execution starts, and notes initial conventions. Deep architectural analysis is forbidden at this stage — stay at survey level.

Step 3 · Map Architecture

The analytical core of the workflow. Mimir creates a detailed component inventory, traces at least one complete data flow end-to-end (user action through frontend, API, business logic, data layer, and back), identifies architectural layers, maps every external integration, documents state management, creates a dependency graph (noting high fan-in and fan-out modules), and catalogs structural and data patterns. Raw analysis only — no final document writing yet.

Step 4 · Document Findings

Create a structured architecture document that answers the original questions from Step 1. Includes a summary, tech stack, at least one Mermaid diagram, component map, data flow traces, dependency relationships, patterns and conventions, a risk and tech-debt table (with severity), and prioritized, specific, actionable recommendations. The document is saved to the project output folder and becomes the team's reference asset.

workflow-prototyping Prototyping 11 steps

Build interactive prototypes from page specifications through structured AI collaboration. Use this when page specifications are complete and approved and you want iterative development with per-section approval gates. The loop is: set up the environment, identify logical views, then for each section — announce, create a story file, implement, present for testing, handle issues, and get approval — before finalizing the complete view. Mimir verifies with Puppeteer before showing anything to the user.

Step 1 · Prototype Setup

One-time environment setup for the entire scenario. Mimir runs the initiation dialog (four questions: device compatibility, design fidelity, languages, demo data), creates the prototype folder structure with all subfolders, creates demo-data.json with realistic test data, and creates PROTOTYPE-ROADMAP.md. Skipped if the scenario already has these files.

Step 2 · Scenario Analysis

Analyze the entire scenario to identify all logical views and map which scenario steps use which views. A logical view is a conceptual page or screen with multiple states. Mimir reads all scenario step specification files, identifies which steps share a logical view versus which represent a new screen, presents the mapping to the user for confirmation, and creates work/Logical-View-Map.md with build order.

Step 3 · Logical View Breakdown

Select a logical view to build and break it into 4–8 implementable sections. Mimir gathers all object IDs across all states of the selected view, confirms the object list with the user, proposes a section grouping with time estimates, gets user approval, and creates the work/[View-Name]-Work.yaml plan file. Story files are not created here — they are created just-in-time per section.

Step 4a · Announce & Gather

Present the build plan for the current section to the user, confirm context and any missing information before building begins.

Step 4b · Create Story File

Write the implementation story file for this section just-in-time, capturing the spec details, object IDs, states, and acceptance criteria that Mimir will implement.

Step 4c · Implement Section

Build the section according to the story file, following existing codebase patterns and using design tokens throughout. Puppeteer verification runs before presenting anything to the user.

Step 4d · Present for Testing

Show the implemented section to the user and request a test. Mimir provides test instructions for each state of the section.

Step 4e · Handle Issue

If the user reports a bug or defect, Mimir investigates, fixes, and re-verifies before re-presenting. The minimal fix principle applies — do not refactor during a section fix.

Step 4f · Handle Improvement

If the user requests an enhancement or polish change, Mimir implements it, verifies, and re-presents. Scope is kept to what is asked — no opportunistic additions.

Step 4g · Section Approved

Mark the section as approved in the work file, update the story file with any learnings, and loop back to Step 4a for the next section.

Step 5 · Finalization

When all sections are approved, run a complete integration test across all states of the logical view. Mimir checks that all Object IDs are present, state transitions work smoothly, there are no console errors, and the prototype is responsive at the target device width. On user approval, present the complete summary and check whether all logical views in the scenario are now built.

workflow-development Development 5 steps

Write production-quality code from approved specifications using structured agent collaboration. Use this when an approved specification exists, the codebase and tech stack are established, and you need to turn a spec into committed, tested, production-ready code. Not for greenfield project setup, for exploring an unfamiliar codebase (use Analysis), for fixing bugs (use Bugfixing), or for throwaway prototypes (use Prototyping).

Step 1 · Scope and Plan

Read the approved specification end to end — no skimming. Extract every deliverable into a flat inventory (pages, components, API integrations, state management, styling, tests). Map dependencies to determine implementation order. Estimate relative sizes and break down any "large" items into sub-tasks. Create the feature branch. Extract every acceptance criterion into a standalone checklist that will drive Step 4 verification.

Step 2 · Setup Environment

Verify the development environment is ready before writing any feature code. Check runtime and tooling versions, install dependencies, start the dev server and confirm existing pages load, verify design system token access, create any new directories the spec requires, and run the full existing test suite to establish a baseline. Pre-existing failures are documented — they are not regressions from this work.

Step 3 · Implement

Build every feature in the implementation plan, one at a time in dependency order. Before each item, re-read the relevant spec section and identify the applicable acceptance criteria. Study existing codebase patterns (naming, component structure, styling, error handling, data fetching, testing) and match them — consistency matters more than personal preference. Use design tokens throughout, implement all component states (not just the happy path), run tests after each major change, and commit incrementally with one logical unit per commit.

Step 4 · Verify

Systematically confirm that the implementation satisfies every acceptance criterion from Step 1. Test all responsive breakpoints. Test every interactive state (default, hover, focus, active, disabled, loading, error, empty, success). Run accessibility checks: keyboard navigation, screen reader labels, WCAG AA color contrast, focus management. Cross-browser test if specified. Compare the running implementation to the spec side by side. Fix failures immediately — do not batch them.

Step 5 · Finalize

Clean up all files (remove debug statements, unused imports, TODO comments). Run the full test suite plus linting and type checking. Document every deviation from the spec with rationale. Update any affected documentation. Prepare the PR description (summary, changed files, testing steps, deviations, screenshots, acceptance criteria checklist). Suggest next steps: acceptance testing, design review, or stakeholder demo.

workflow-acceptance-testing Acceptance Testing 5 steps

Validate that implementation matches design specifications through structured testing. Use this when implementation is complete — prototype or production — and ready for validation against specs. Mimir runs happy path tests, error state tests, edge case tests, design system validation (measuring colors, spacing, typography against spec tokens), and accessibility tests. A formal test report and issue tickets are produced. Maximum three iteration cycles before escalation.

Step 1 · Prepare

Gather all materials before any testing begins: load the test scenario file and design delivery file, review all scenario specifications and design system specs, set up the staging environment and test devices, configure screen recording and screenshot tools, prepare test data (valid and invalid inputs, edge case data), create the testing workspace folder structure, review the test plan by category, and calculate a time estimate with a 20% buffer.

Step 2 · Execute

Run all test categories in order: happy path tests first (start screen recording, follow steps exactly, compare to design reference, mark PASS/FAIL, screenshot failures), then error state tests (trigger each error condition, verify message styling and recovery), then edge case tests (unusual inputs and scenarios), then design system validation (measure dimensions, colors, typography, and spacing against spec tokens for every component), then accessibility tests (screen reader navigation, WCAG AA color contrast, 44×44px touch target minimums). Compile an overall summary with pass rates by category.

Step 3 · Document Issues

Create a numbered issue file (ISS-XXX-description.md) for every problem found. Each issue requires: severity (Critical / High / Medium / Low), specific description (not "button looks wrong" but exact values vs spec), exact steps to reproduce, expected vs actual behavior, annotated screenshot, and a design reference. Vague or non-actionable issues are not acceptable. Create an issues summary with breakdown by severity.

Step 4 · Report

Create a comprehensive test report (TR-XXX-[flow-name].md) with all sections: summary (overall PASS/FAIL, total issues, blocking status), test coverage by category, full issues table, sign-off recommendation, next steps, and organized attachments. A PASS requires zero Critical issues, all High issues fixed or accepted, 100% happy path pass, and design system compliance above 95%.

Step 5 · Iterate or Approve

Two paths. If FAIL: wait for fixes, retest focused on fixed issues and regression areas (abbreviated — not a full re-run), update issue statuses, create a retest report, and either advance to approval or repeat the cycle (maximum three total iterations before escalating to leads). If PASS: create a sign-off document, notify the development team, and update the design log status to approved.

workflow-bugfixing Bug Fixing 5 steps

Systematically investigate, fix, and verify bugs with minimal side effects. Use when a bug has been reported or discovered in existing code. The three core principles: reproduce first (never fix what you cannot reproduce), minimal fix (target root cause with the smallest possible change), regression check (verify original bug is fixed AND test for side effects on related functionality). Refactoring surrounding code during a bugfix is explicitly forbidden.

Step 1 · Reproduce

Confirm the bug exists and document exact reproduction steps. Gather bug report details (what happened, what was expected, when it started, reporter's environment). Set up the reproduction environment to match the reporter's setup as closely as possible. Follow the reported steps exactly, capturing screenshots and console output. Reproduce at least twice to confirm consistency. If the bug cannot be reproduced, request more details — do not proceed to investigation without reproduction or a clear theory.

Step 2 · Investigate

Identify the root cause, not just the symptom. Read the relevant code around the bug location. Check recent git commits for changes in the affected area. Trace the execution path from the triggering user action through event handlers, state updates, API calls, and rendering — identifying exactly where behavior diverges from expectation. Pin down the root cause to specific lines or logic. Check whether similar patterns exist elsewhere that may have the same bug. Document the root cause, symptom explanation, and proposed fix approach before writing any code.

Step 3 · Fix

Implement the minimal fix targeting the root cause. Create a fix branch. Write a test that catches the bug first — it must fail before the fix and pass after. Then implement the smallest change that resolves the issue: no refactoring, no unrelated improvements, no formatting changes in adjacent lines. Verify the fix resolves the exact reproduction case from Step 1. If the fix requires more than a few lines, pause and re-examine scope.

Step 4 · Verify

Confirm the fix works and has not introduced regressions. Re-run the exact reproduction steps from Step 1. Run the full regression test suite (or manually test core user flows if no automated suite exists). Check edge cases around the fix (null inputs, boundary values, rapid repeated actions). Verify no side effects on features that share code with the fixed area. Cross-platform check if the bug was platform-specific.

Step 5 · Document

Document the bug and fix, then create a clean PR. Write a clear root cause summary explaining why the bug existed. Document the fix approach and why it was chosen over alternatives. Ensure the regression test from Step 3 is complete and well-named. Write the PR with title fix: what was fixed, body covering bug, root cause, fix, and testing. Flag any similar risks observed elsewhere in the codebase for future investigation. Update the design log with a bugfix progress entry.

workflow-reverse-engineering Reverse Engineering 4 steps

Analyze existing software or public websites to extract page specifications, design systems, and architectural understanding. Use when you need to understand an existing product (yours or public), want to create WDS specifications from existing software, are doing competitive analysis through design extraction, or are migrating or rebuilding a product. Two modes: Internal (your own codebase with source access) and External (any public website via browser, no source needed). The goal is patterns and structures, not pixel-perfect copying — and never proprietary code or copyrighted content.

Step 1 · Identify Target

Define what to reverse engineer, how to access it, and what to extract. Classify the target type (website, web application, component library, mobile app, or source code) and determine the access method (URL via browser, source code, screenshots, or a combination). Check for access limitations. Define extraction goals from a checklist: page specifications, design system, component inventory, content strategy, architecture, user flows, or all of the above. Document a target overview to guide the exploration.

Step 2 · Explore and Capture

Systematically explore the target and capture complete inventories. For URL access: crawl all pages via navigation and sitemap, capture header/main/footer structure, observe hover states, dropdown behavior, form validation, modal triggers, loading states, and responsive breakpoints, then extract colors, fonts, spacing, border radii, and shadows from DevTools. For source code: read component directories, map routes, identify shared styles and theme config. Compile findings into structured inventories: page inventory, navigation structure, component inventory, color palette, typography scale, and spacing patterns.

Step 3 · Generate Specs

Create WDS-format page specifications from the inventories captured in Step 2. Prioritize pages (start with the most representative), then for each page generate a full specification: overview, layout structure (desktop, sections top to bottom), component list cross-referenced with the inventory from Step 2, content strategy (headline patterns, CTA language, images), responsive behavior, and interactions. Document multi-page user flows as scenario outlines. Save all output to the project folder in consistent naming. Generate everything in the configured document_output_language.

Step 4 · Extract Design System

Consolidate all design tokens and component patterns into a structured WDS-compatible design system document. Organize raw values into token categories: colors by purpose (brand, text, background, border, feedback), typography scale (families, sizes, weights, line heights), spacing (base unit and scale), border radii, shadows, and breakpoints. Catalog each component with variants, sizes, states (default, hover, focus, active, disabled, loading, error, success), content slots, and responsive behavior. Create an explicit token-to-component mapping. Save the design system document alongside the page specs from Step 3.

workflow-evolution Evolution 5 steps

Incrementally add features to an existing product with minimal disruption. Use when existing functionality needs new capabilities and changes should be targeted, not a rewrite. Three core principles: backward compatible (existing functionality must keep working, verified at every commit), feature flags if needed (for risky or staged changes), incremental delivery (each commit leaves the system in a working state). Skipping impact analysis is explicitly forbidden — surprises in production are expensive.

Step 1 · Scope Change

Define exactly what is new, what is modified, and what must remain untouched. Load and understand the feature specification. List all new functionality (new UI components, API endpoints, data models, routes). List all existing functionality that must stay unchanged. Create a boundary map categorizing every affected area as New, Modified, or Untouched. Identify and document every integration point with its risk level. Ambiguities in the spec are resolved here — not during implementation.

Step 2 · Analyze Impact

Understand how the proposed changes affect existing code and identify risks. Read every file in the "Modified" category and all adjacent files. Map dependencies (what depends on each modified file; what each modified file depends on). Identify breaking change risks: interface signature changes, data shape changes, timing/ordering effects, violated implicit assumptions. Assess database and API impacts (migrations, contract changes, breaking vs additive). Produce a risk assessment table with likelihood, impact, and mitigation for every identified risk.

Step 3 · Plan Implementation

Create an ordered, incremental implementation plan with verification points between steps. Order changes by dependency: foundational changes first (data models, utilities), then API endpoints, then components, then pages and integrations. Decide whether feature flags are needed (for large changes, staged rollouts, or instant rollback requirements). Plan each commit as a complete working unit — the system must remain functional after every commit. Define explicit verification criteria between steps. Estimate effort and flag high-risk steps.

Step 4 · Implement

Implement changes incrementally, verifying at each step. Create the feature branch. For each planned commit: read the plan, implement only that step, self-check scope, run the defined verification point, and commit with a descriptive message. Do not proceed to the next commit if the current one has failing tests. Verify backward compatibility at every commit — existing users must be able to do everything they could before. If feature flags were planned, implement them early and test both flag states.

Step 5 · Verify and Document

Comprehensive verification of all new and existing functionality, then documentation and PR. Test all new features end-to-end (happy paths, error paths, edge cases). Run the full regression suite — any failure must be fixed before continuing. Test every feature listed as "Untouched" in the boundary map. Run a performance check: load time, bundle size impact, new network requests. Document what was added, changed, and left untouched, plus any deviations from the plan. Create the PR with title feat: what was added, covering what, why, how (incremental steps), testing (including backward compatibility), and known risks. Update the design log with an evolution progress entry.
