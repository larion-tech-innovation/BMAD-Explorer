Code Review · Adversarial Review Engine

Three independent subagents review the same diff with different context — each blind to the others' findings. Findings are triaged into four buckets, written to the story file, and resolved interactively. An AI cannot objectively review its own code; adversarial mandate is the structural solution.

Code Review is the adversarial gate of Phase 4. It does not ask the dev agent to review its own output — that produces rubber-stamp reviews. Instead it spawns three independent subagents with deliberately different contexts, then forces deduplication, triage, and interactive resolution of every ambiguous finding.

step-architecture Step-File Architecture

Code Review uses a step-file design — each step is a self-contained file loaded just-in-time. This enforces sequential execution and prevents the agent from conflating step logic across the workflow.

Execution Rules — No Exceptions
Micro-file design
Each step file is self-contained and complete. Read the entire file before taking any action.
Just-in-time loading
Load only the current step file. Never load the next step file until directed by the current one.
Sequential enforcement
Complete steps in order. No skipping, no optimization of the sequence.
HALT at checkpoints
When a step says HALT, stop and wait for human input before proceeding.

§ WHY Why Adversarial? · The Case Against Self-Review

Adversarial Mandate
"AI tự review code của mình sẽ rubber-stamp — adversarial mandate buộc Amelia phải genuinely hunt for issues."
An LLM that wrote the code carries the original intent into review. It anchors on "what I meant to write" and confirms that the code satisfies that intent — even when it doesn't. Context isolation is not a preference; it is the mechanism that breaks anchoring bias at the architectural level.

Three reviewers with different contexts find different bugs. A reviewer who can't see the spec can't be anchored by it. A reviewer who traverses every branching path finds boundary errors human reviewers skip. A reviewer who cross-checks the original ACs catches spec drift no one noticed during implementation.

self-vs-adversarial Self-Review vs Adversarial Review

Dimension Self-review (same agent) Adversarial review (3 isolated)
Anchoring bias Reviewer knows what was intended — interprets bugs as "working as designed" Blind Hunter has no spec context — cannot assume intent, must read what the code actually does
Branch coverage Happy-path focus — the implementation path is the one that gets reviewed Edge Case Hunter systematically traces every if/else, try/catch, and early return
Spec drift AC drift goes undetected — reviewer remembers the intent, not the original AC text Acceptance Auditor cross-references diff against original AC text, line by line
Finding cross-validation No second opinion — one reviewer, one pass Cross-reviewer deduplication: same bug found by two sources gets merged, gets higher confidence
Clean code verdict "Looks good to me" — inherently unreliable from the author Zero findings is valid — three reviewers with different contexts all passed
v6.x improvement — Removed minimum findings requirement — prior versions fabricated issues to hit a ≥3 threshold on clean code

review-modes Review Modes

  • full modeA story spec file is available. All 3 reviewers run. Acceptance Auditor cross-checks diff against every Acceptance Criterion. decision_needed bucket is active — ambiguous findings that need user input to resolve correctly.
  • no-spec modeNo story file provided. Blind Hunter and Edge Case Hunter run. Acceptance Auditor is skipped with notice to user. In this mode, any finding that would otherwise be decision_needed is reclassified as patch (if fix is unambiguous) or defer (if not).

§ HOW How to Execute · 4-Step Workflow

01 Gather Context — 5-Tier Target Discovery

The skill auto-detects what to review through a cascade. It stops as soon as a tier identifies the target — no extra questions asked beyond what the cascade prescribes.

  1. Tier 1
    Explicit argument. Did the user pass a PR, commit SHA, branch, spec file, or diff source? Also scan for diff-mode keywords: "staged" / "branch diff vs main" / "last N commits" / "this diff". When multiple match, prefer the most specific.
  2. Tier 2
    Recent conversation. Do the last few messages reveal what the user wants reviewed? Look for spec paths, commit refs, branches, PRs, or described changes. Apply the same keyword scan as Tier 1.
  3. Tier 3
    Sprint tracking. Scan for sprint-status.yaml. If exactly one story has status review — suggest it and confirm. If multiple: present numbered options. If none: fall through.
  4. Tier 4
    Current git state. If current branch is not main, confirm: "I see HEAD on <branch> — do you want to review this branch's changes?" If declined: fall through.
  5. Tier 5
    Ask. HALT. Present 5 options: uncommitted changes · staged changes only · branch diff vs base · commit range · provided diff or file list. Wait for user choice.
After Target Is Found
  • Spec contextIf spec file already identified: set review_mode = full. Otherwise, ask: "Is there a spec or story file?" If yes: load it, set full. If no: set no-spec.
  • Diff size checkIf diff exceeds ~3000 lines: warn user, offer chunking by file group. If user opts in: agree on the first group and list remaining groups for follow-up runs.
  • CheckpointHALT. Present: diff stats (files changed, lines +/−), review mode, loaded spec/context docs. Wait for user confirmation to proceed.

02 Parallel Review Chamber — 3 Isolated Subagents

Three subagents launch simultaneously. Each receives a different context package — deliberately. The isolation prevents one reviewer from anchoring on another's interpretation, and ensures each finds bugs the others are structurally unable to find.

Parallel Review Chamber All 3 run concurrently · Context isolation enforced · No cross-reviewer context
BH
Blind Hunter
bmad-review-adversarial-general
Receives
Diff only. No spec file. No context docs. No project read access. Cannot see what the code was supposed to do — only what it does.
Hunts for
  • Logic errors and incorrect assumptions
  • Security vulnerabilities (injection, exposure, validation gaps)
  • Runtime exceptions — nil deref, type mismatch, unhandled errors
  • Off-by-one errors and index boundary issues
  • Each finding: file:line location required
Why isolated: breaks anchoring bias — can't assume "code reflects intent"
EH
Edge Case Hunter
bmad-review-edge-case-hunter
Receives
Diff + project read access. Can traverse the codebase to understand patterns, but does not receive the spec — judges correctness by code structure, not intent.
Hunts for
  • All branching paths: if/else, try/catch, early returns
  • Boundary conditions — empty, null, zero, overflow, max
  • Race conditions and concurrency hazards
  • Deletion-contract audit — removed code's callers/references are checked, absorbed here instead of a standalone layer
  • Outputs: JSON with location, trigger_condition, guard_snippet, potential_consequence
  • Systematic path traversal — not sampling
Why isolated: human reviewers skip path combinations — systematic traversal finds what intuition misses
AA
Acceptance Auditor
full mode only
Receives
Diff + story spec + context docs. The only reviewer with the original Acceptance Criteria. Its job is to compare implementation reality against spec intent — nothing else.
Hunts for
  • AC drift — spec says X, code does Y
  • Missing implementation of specified behavior
  • Missing error paths required by spec
  • Contradictions between spec constraints and code
  • Each finding: AC-id + file:line evidence required
Why isolated: cross-checks the original AC, not the developer's memory of it

If subagents are unavailable, the skill generates prompt files — one per reviewer — in implementation_artifacts and HALTs. The user runs each in a separate session (preferably a different LLM) and pastes findings back. The workflow resumes at Step 3.

03 Triage — Normalize, Deduplicate, Classify

Each reviewer returns a different output format. Step 3 normalizes all findings into a unified schema, deduplicates cross-reviewer overlaps, and classifies each finding into exactly one bucket.

decision_needed
Ambiguous — Human Required
The correct fix depends on intent that only the user knows. The code cannot be correctly patched without their input. Only possible in full mode.
→ HALT, ask user · resolves to patch / defer / dismiss
patch
Fix Now
A real code issue with an unambiguous correct fix. No user input needed to determine the right solution — just apply it.
→ Apply all / walk through / leave as items
defer
Track for Later
A valid concern, but pre-existing and not caused by this change. Does not block merge. Written to deferred-work.md for future work.
→ Append to deferred-work.md · checked off in story
dismiss
Noise
False positive, already handled elsewhere, or not actionable. Dropped from the output. Count is reported in the summary.
→ Drop · count recorded in summary
Deduplication Logic
  • Same issueIf two or more findings describe the same issue, merge them. Use the most specific as the base (edge-case JSON with location beats adversarial prose).
  • Merged detailAppend unique reasoning or location references from the other finding(s) into the surviving finding's detail field.
  • Combined sourceSet source to the merged labels (e.g., blind+edge, edge+auditor). A multi-source finding has higher confidence.
  • Severity calibrationRate severity only after reading the surrounding source (call sites, guards) — never from the diff hunk alone. The "prefer conservative when uncertain" tie-breaker that used to inflate severity has been removed.
  • Failed layersIf any reviewer failed or timed out: report which before announcing results. Zero findings with failed layers = warn user (may be incomplete), not a clean pass.

04 Present, Resolve & Update Status

  • Write firstIf spec file exists: append all findings to story file under ### Review Findings before offering resolution options. Findings are written regardless of how they will be resolved.
  • Finding format[ ] [Review][Decision] · [ ] [Review][Patch] · [x] [Review][Defer] — each marked with file:line location. Deferred items also go to deferred-work.md.
  • Decision firstAll decision_needed items are resolved before patch items. HALT on each — wait for user to say: patch / defer / dismiss. If deferred: ask for a one-line reason to record in the story and in deferred-work.md.
  • Patch handlingThree options (when spec file exists): Apply every patch now · Leave as action items · Walk through each. When no spec file: only apply-all and walk-through available.
  • Clean shortcutIf zero findings remain after triage: state "✅ Clean review — all layers passed" and skip directly to sprint status update.

status-outcomes Status Outcomes

The story's final status from a code review run is determined by whether all actionable findings were resolved. Both the story file and sprint-status.yaml are updated.

All Resolved — Clean
done
All decision_needed and patch findings were resolved (fixed or dismissed). No unresolved HIGH/MEDIUM issues remain. Story moves to done. Next story in the sprint can begin.
Action Items Remain
in-progress
Patch findings were left as action items in the story file, or unresolved issues remain. Story moves back to in-progress. Dev Story re-runs to address review findings via review continuation mode.
Handoff Chain
  • All resolved → done: bmad-sprint-planning or bmad-create-story picks up next ready-for-dev story in the sprint.
  • Action items → in-progress: bmad-dev-story re-runs. Step 3 of dev-story detects the "Senior Developer Review" section and enters review continuation mode.
  • Last story of epic → done: Sprint retrospective (bmad-retrospective) runs after the final story is marked done.
