§ WHAT UX Scenarios — What does this Skill do?

Role
Journey-mapping facilitator — turns the Trigger Map's psychology into linear sunshine-path outlines that name every page Phase 4 must design.
Agent
Freya 🎨, the WDS Designer; she owns Phases 3, 4, 6, 7, and 8, and replaces BMM Sally when WDS is installed.
Phase
WDS Phase 3 — embeds in BMM Planning. Reads Phase 1 and Phase 2 artifacts; hands off a brief to Phase 4 UX Design.

This skill takes the strategy captured in Phase 2 and renders it as concrete user journeys. A scenario is a linear "sunshine path" — the happy path through a single user goal, walked step by step — and its job is to expose every page, screen, and decision point so the next phase has a complete map of what to design. Scenarios are not page specs; they are the connective tissue between the Trigger Map's user psychology and Phase 4's detailed visual specifications.

Use it after the Trigger Map is complete and before any UX Design work. Multiple scenarios are normal and expected — typically one per persona driving force — and together they produce a coverage matrix that tells Freya exactly which pages need spec work in Phase 4.

§ STEPS Workflow Steps — How the skill executes

The skill uses step-file architecture: each step is a self-contained instruction file that is fully read and executed before the next is loaded. Two modes are available — the default creation workflow (9 steps) and a validate mode (5 steps) for reviewing existing scenarios.

main Scenario Creation Workflow 9 steps

01 · Load Context

Read all prerequisite artifacts and detect project state. Freya loads config.yaml, the Product Brief, the Trigger Map, and all persona files, then checks for existing scenario work. If prior work is found, she surfaces a resume / adjust / restart choice before proceeding. Ends with a project context summary for the user to verify before continuing.

02 · Analyze Scope USER CHECKPOINT

Classify the site type (Presentation, Dynamic, or Mixed), build a complete page inventory, assess scale (Small / Medium / Large), and recommend a scenario format and approach mode. Presents the analysis in full and waits for explicit user approval before continuing. This is a gate — the scope set here bounds every scenario that follows.

03 · Build Strategic Context

Extract strategic context from the Trigger Map by tracing complete chains from Business Goal → Persona → Driving Force → Transaction → Scenario. Answers all 7 Decision Matrix questions for each chain, assigns pages from the approved inventory to exactly one chain each, and verifies full coverage. No scenario outlining happens here — only structured planning.

04 · Suggest Scenarios USER CHECKPOINT

Present the complete scenario plan — scenario names (always persona-first, e.g., "Hasse's Emergency Search"), page assignments, persona connections, user and business value, and a coverage check. Waits for explicit user approval. User may combine, add, or remove scenarios before approval. No outlines are written until this gate is passed.

05 · Outline Scenario LOOPS PER SCENARIO

Work through 8 strategic questions — transaction, business goal connection, persona and situation, mental state (hope + worry), device, entry point, mutual success goals, and shortest linear path — one at a time in collaborative dialogue. Alternatively, suggest mode lets Freya answer all 8 from the Trigger Map for user review. Quality gates verify the outline before the scenario file is written and step-by-step page folders are created. Loops once per scenario in priority order.

06 · Generate Overview

Create C-UX-Scenarios/00-ux-scenarios.md — the master index file. Contains a scenario summary table, individual scenario entries with links, and the full page coverage matrix mapping every page to the scenario it serves. All links are verified before the file is considered complete.

07 · Quality Review

Self-review all scenarios across four dimensions: Completeness (7 components), Quality Criteria (7 checks), Mistakes Avoided (6 checks), and Best Practices (4 checks). Minimum thresholds must be met before proceeding. Failing items are identified and fixed; a review summary with per-scenario scores is presented to the user.

08 · Update Design Log

Append a Phase 3 completion entry to _progress/00-design-log.md, listing every artifact file created individually (no "etc." summaries), recording key decisions, and noting quality scores. Existing log entries are never overwritten — only appended.

09 · Handover

Present the Phase 3 completion summary, guide the user through design intent selection for each scenario (Sketch / Discuss / Suggest / Dream Up / Later), save the chosen approach to each scenario's frontmatter, and explain what Phase 4 UX Design does with each approach. Phase 3 ends gracefully here.

validate Validate Mode 5 steps

Triggered by passing validate or -v when invoking the skill. Runs a structured audit of existing Phase 3 output — no scenario files are modified during validation, only read and assessed.

01 · Scenario Coverage

Cross-reference every strategic context chain from the Trigger Map (Business Goal → Persona → Driving Force) against existing scenarios. Verifies that every chain is addressed by at least one scenario, that Priority 1 chains have dedicated scenarios, and that the Trigger Map Connections section in each scenario explicitly references its chain components. Reports with Critical / Warning / Pass severity levels.

02 · Navigation Patterns

Verify that all scenario shortest paths follow WDS navigation conventions. Checks: page naming consistency across scenarios (same page = same name everywhere), linear flow rules (zero "if" statements, valid entry/success points, no impossible jumps or dead ends), and builds a cross-scenario page registry tracking shared pages. Navigation conflicts are identified and reported.

03 · Outline Completeness

Validate all 7 required components of every scenario outline: Scenario Name & ID, Core Feature, Entry Point (device + context + discovery method), Mental State (trigger + hope + worry — all visceral), Success Goals (user and business, both measurable), Shortest Path (linear, minimum viable steps, success marker), and Trigger Map Connections. Each component is scored pass / warning / fail with specific gap descriptions.

04 · Cross-Scenario Consistency

Compare scenarios against each other to find contradictions, imbalances, and duplicates. Verifies: shared pages serve consistent purposes, persona coverage matches Trigger Map priorities, every business goal is addressed, no two scenarios are essentially the same path under different names, and the 00-ux-scenarios.md index accurately reflects all scenarios and their page counts.

05 · SEO Keyword Alignment

Map scenario pages against the SEO keyword strategy from Phase 1. Checks that each page has at least one primary keyword, that keywords match user intent (not forced), and that no two pages compete for the same primary keyword. If no keyword map exists, the gap is noted and the step proceeds to the final report. Compiles results from all 5 validation steps into a validation-report.md saved to C-UX-Scenarios/.

output What a Scenario Outline Contains per scenario

Each scenario file (step 5) follows the same structure, written to C-UX-Scenarios/NN-[scenario-name]/NN-[scenario-name].md.

  • Name & IDThe scenario name and its ID — the handle the coverage matrix and Phase 4 reference.
  • Target PersonaThe persona this journey serves, pulled directly from the Trigger Map.
  • Entry / SuccessThe entry point the user arrives through, and the success state that ends the path.
  • Linear JourneyThe step-by-step user journey — a single linear path, no branches.
  • Per-Step DetailFor each step: what the user does, what they see, and what they feel.
  • Pages RequiredThe pages required at each step — the raw material for the coverage matrix.
  • SEOSEO considerations for the journey, inherited from the brief's keywords.

§ WHY Why UX Scenarios? — The reasoning behind each design choice

why-sunshine-paths Why Sunshine Paths Only?

A scenario captures the happy path through a single user goal and nothing else. Edge cases, error states, and alternative branches are deliberately excluded here — they are handled inside Phase 4 page specs, where the page being designed is the right place to decide what happens when something goes wrong. Keeping scenarios linear keeps them readable as journeys and prevents Phase 3 from collapsing into premature interaction design. The sunshine path answers "what is this experience when it works?" — Phase 4 answers "and what about when it doesn't?"

why-coverage-matrix Why the Coverage Matrix Matters

The coverage matrix in 00-ux-scenarios.md maps every scenario against every page it touches. This is how Freya knows which pages need spec work in Phase 4: a page that appears across several scenarios is high-priority and must satisfy multiple journeys, while a page touched once is scoped narrowly. Without the matrix, Phase 4 would have to re-derive the page inventory from scattered scenario files. The matrix turns a pile of journeys into a single planning surface.

why-strict-ordering Why the Strict Ordering?

The nine steps run in a fixed order, and the skill never loads multiple step files simultaneously and never skips or reorders them. This is not ceremony — the pipeline breaks otherwise. Each step's output is the next step's input: scope (step 2) bounds the scenario plan (step 4), which bounds the outlines (step 5), which feed the overview (step 6). Loading steps out of order, or several at once, lets later reasoning run on inputs that do not yet exist, producing outlines that look complete but are not anchored to an approved scope.

why-one-per-force Why One Scenario per Driving Force?

Scenarios are scoped to a single persona driving force — the specific motivation that brings a user to the product. Multiple scenarios are normal and expected: a product that serves several personas, or one persona with several distinct goals, needs several sunshine paths. Mapping one force per scenario keeps each journey focused and makes the coverage matrix meaningful, because every scenario traces back to a real motivation captured in the Trigger Map rather than an invented use case.
