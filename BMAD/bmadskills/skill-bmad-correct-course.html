Correct Course is the structured response to mid-sprint reality. Things change — a stakeholder adds a requirement, a technical assumption proves wrong, an architectural discovery forces a pivot. Without a process, these changes get silently absorbed, artifacts go out of sync, and no one knows who decided what or why. Correct Course creates the record.

trigger-events When to Trigger This Skill

Trigger
Use for significant changes that affect project artifacts. Not for small implementation decisions that stay within story scope. If the change affects the PRD, epics, architecture, or UX specs — Correct Course is the right tool.
  • New stakeholder requirement — a requirement emerges mid-sprint that was not in the PRD
  • Architectural discovery — implementation reveals a constraint that changes the technical approach
  • Scope shift — story implementation reveals the original scope was wrong or too broad
  • Technical blocker forcing pivot — a chosen technology or pattern doesn't work as planned
  • Strategic pivot — market or business direction changes while sprint is underway
  • Failed approach — a completed story's approach needs to be reconsidered or rolled back

execution-modes Incremental vs Batch Mode

Chosen at Step 1. The mode affects how change proposals are reviewed — not the analysis depth, which is always the same.

Recommended
Incremental Mode
Each change proposal is presented one at a time. User reviews and decides before the next is shown. Allows fine-grained control and course corrections during drafting — edits to one proposal can inform the next.
Per proposal: [A]pprove → [E]dit → [S]kip
Alternative
Batch Mode
All change proposals are drafted first, then presented together for a single review pass. Faster overall but less control over individual proposals during drafting. Better when the change is well understood up front.
All proposals → review together → approve/edit/skip each

§ WHY Why Formal Change Management? · The Silent Absorption Problem

Mid-sprint changes happen constantly. The question is whether they get absorbed silently — creating invisible artifact drift and undocumented decisions — or processed formally with a clear record of what changed, why, and who approved it.

When a change is absorbed silently without process, the PRD still says one thing, the epic says another, and the code does something else. A developer working on the next story reads outdated specs. A reviewer flags AC violations that aren't actually violations — the AC just wasn't updated. No one knows when the decision was made or who made it.

The Sprint Change Proposal is the decision record. It makes the change visible, traceable, and routed to the right person for approval. The audit trail matters — not just for the current sprint, but for the next retrospective, the next epic, and anyone who joins the project later.

ad-hoc-vs-process Ad-Hoc Change vs Correct Course

Dimension Silent / ad-hoc absorption Correct Course workflow
Artifact sync PRD, epics, architecture drift independently — inconsistencies accumulate silently across documents Full artifact load, cross-referenced checklist — all conflicts surfaced and resolved in one session
Decision record No formal record — "we changed direction somewhere in sprint 3" with no way to know what decision was made or why Sprint Change Proposal document with explicit before/after, rationale, and approver name — permanent audit trail
Impact scope Only the obvious changes get made — downstream ripple effects to future stories and epics go undetected 6-section checklist walks through every artifact layer — ripple effects are forced into the open
Approval Change happens informally — often no explicit sign-off, scope disputes emerge later Scope classified as Minor / Moderate / Major — routed to the appropriate approver with explicit sign-off before implementation
Future stories Next story is created from outdated specs — developer builds against wrong assumptions, creates review failures sprint-status.yaml updated immediately — next story creation picks up the corrected state

§ HOW How to Execute · 6-Step Workflow

01 Initialize Change Navigation

  • Gather triggerAmelia confirms the change trigger: "What specific issue or change has been identified?" HALT if the trigger is unclear — cannot analyze impact without a concrete problem statement.
  • Verify artifactsConfirm PRD and Epics are accessible. HALT if either is missing. Load Architecture and UX specs if available.
  • Mode selectionUser chooses: Incremental (recommended — review each proposal one at a time) or Batch (all proposals presented at once for review).

02 Execute Change Analysis Checklist

  • Systematic walkWork through all 6 checklist sections interactively. Each item marked: [x] Done · [N/A] Not applicable · [!] Action needed.
  • Progress reportsAfter each major section, present findings and impacts discovered so far. Running notes maintained throughout.
  • Blocking issuesIf any critical section cannot be completed: HALT and work with user to resolve before continuing to proposal drafting.

03 Draft Specific Change Proposals

  • Old → New formatEach proposal shows explicit before/after with: artifact reference, story ID or section name, old text, new text, and rationale. See format example below.
  • Incremental modePresent each proposal individually. User chooses: Approve [a] · Edit [e] · Skip [s]. Iterate on each before moving to the next.
  • Batch modeDraft all proposals, collect them, then present the complete set for review in one pass.

04 Generate Sprint Change Proposal

  • 5-section documentCompile: Issue Summary · Impact Analysis · Recommended Approach · Detailed Change Proposals · Implementation Handoff.
  • SaveWrite to sprint-change-proposal-{date}.md in planning_artifacts.
  • Review gatePresent complete proposal to user: Continue [c] or Edit [e]?

05 Finalize and Route

  • Explicit approvalGet clear yes/no from user: "Do you approve this Sprint Change Proposal for implementation?" If revise: return to Step 3 (proposal changes) or Step 4 (document structure changes).
  • Scope classificationClassify as Minor / Moderate / Major and route to appropriate agent(s). See Scope Routing below.
  • Sprint syncUpdate sprint-status.yaml to reflect approved epic changes — additions, removals, renumbering, story changes.

06 Workflow Completion

  • SummaryReport: change trigger addressed · scope classification · artifacts modified · routed to (named agents).
  • Deliverables confirmedSprint Change Proposal doc · specific before/after proposals · implementation handoff plan.
  • HandoffConfirm next steps — what the receiving agent(s) will do and when.

proposal-format Change Proposal Format — Old → New

Every approved change is expressed in this explicit before/after format. No vague "update the requirement" — the exact old text and the exact new text, with the rationale that connects them.

Sprint Change Proposal — Individual Change Entry
Artifact: PRD — §3.2 Authentication Story: STORY-123 · User Login Section: Acceptance Criteria
Old
- User can log in with email/password - Session expires after 24 hours
New
- User can log in with email/password - User can authenticate via OAuth 2.0 (Google/GitHub) - Session expires after 24 hours - 2FA optional, enabled by user preference
Rationale Security requirement surfaced in sprint: stakeholder mandated OAuth support and 2FA capability for enterprise tier. Discovered during Story 2.3 implementation when the enterprise API contract was reviewed.

three-paths 3 Change Paths

Checklist §4 evaluates all three options before selecting one. Effort and risk ratings are explicit. The selected path and its rationale go into the Sprint Change Proposal.

Path A
Direct Adjustment
Modify/add stories within the current plan. Timeline and scope largely maintained.
Effort Low
Risk Low
Timeline Minimal impact
Best when Change fits within existing epic structure without major restructuring
Path B
Potential Rollback
Revert recently completed stories to simplify resolving the issue. Justified when rollback cost < total rework cost.
Effort Medium
Risk Medium
Timeline Sprint delay
Best when A completed story's approach is fundamentally wrong and would compound rework if left in place
Path C
MVP Review
Re-evaluate or reduce MVP scope. Core goals may need modification based on new constraints discovered during execution.
Effort High
Risk Medium–High
Timeline Significant
Best when The change affects MVP scope or fundamental project goals — cannot be contained to the current epic

A Hybrid path is also available when the best resolution draws from multiple options — e.g., roll back one story (Path B) and then make direct adjustments (Path A) rather than triggering a full MVP review.

scope-routing Scope Classification & Routing

After the Sprint Change Proposal is approved, scope is classified and the change is routed to the appropriate agent(s) for implementation. This determines who executes — not who decides (that's always the user).

Minor
Direct Implementation
→ Developer agent
Change is contained, unambiguous, and implementable without coordination. Developer executes the approved edit proposals directly. Deliverable: implemented changes.
Moderate
Backlog Reorganization
→ Product Owner + Developer
Change requires story/epic restructuring that needs PO sign-off before Developer executes. Deliverable: Sprint Change Proposal + backlog reorganization plan.
Major
Fundamental Replan
→ PM + Architect
Change affects project direction, architecture, or MVP definition. Escalated to PM and Architect for strategic review. May require full replanning session before sprint resumes. Deliverable: Sprint Change Proposal + escalation notice.

output-document Sprint Change Proposal — 5-Section Structure

Saved as sprint-change-proposal-{date}.md in planning_artifacts. This document is the permanent record of the change decision — referenced in future retrospectives and sprint planning sessions.

  • §1 Issue SummaryProblem statement · when/how it was discovered · concrete evidence (error messages, feedback, constraints)
  • §2 Impact AnalysisEpic impact (which epics, how affected) · Story impact (current and future) · Artifact conflicts (PRD, Architecture, UX) · Technical impact (code, infra, deployment)
  • §3 Recommended ApproachSelected path (Direct / Rollback / MVP Review / Hybrid) · rationale · effort estimate · risk assessment · timeline impact
  • §4 Detailed Change ProposalsAll approved old → new proposals, grouped by artifact type. Each with before/after text and justification.
  • §5 Implementation HandoffScope classification (Minor / Moderate / Major) · named handoff recipients · their specific responsibilities · success criteria for implementation
