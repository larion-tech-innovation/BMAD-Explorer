§ WHAT Implementation Readiness Check — What does this Skill do?

This skill is a structured planning-layer gate that executes before Sprint 1 begins. It reads the PRD, Epics & Stories, Architecture, and optionally the UX document — cross-validates them against each other — and produces a single authoritative verdict on whether the team's planning artifacts are complete and consistent enough to begin development without significant rework risk.

The check is deliberately scoped: it validates the planning layer only, not the technical design. Whether an architecture decision is sound, whether a tech stack will scale, or whether a UX wireframe is aesthetically appropriate — these are outside its scope. What it validates rigorously is whether every functional requirement has a story, whether epics are structured for incremental delivery, whether acceptance criteria are testable, and whether the planning documents are internally consistent.

verdict-system Verdict System

Every run produces exactly one of three verdicts. The verdict drives the team's next action — there is no partial-proceed state.

✅
READY
Artifacts aligned and complete
All functional requirements are covered by at least one epic story. UX (if present) is consistent with the PRD and Architecture. Epics are structured for incremental delivery with no forward dependencies. Acceptance criteria are testable. Database tables are created by the story that first uses them. No Critical issues found anywhere in the validation. Minor issues may exist but do not block development.
Proceed to development · Sprint 1 may begin immediately
⚠️
NEEDS WORK
Critical issues found that will cause wrong features
One or more Critical issues were identified that will cause wrong features to be built, FRs to be skipped entirely, or stories to fail independently in a sprint. Typical triggers: at least one uncovered FR, a forward dependency that breaks epic independence, a technical-only epic with no user value, or a starter template story missing when the Architecture requires one. These must be resolved before Sprint 1 begins — not during it.
Fix all Critical issues · Re-run check · Only then begin Sprint 1
❌
NOT READY
Fundamental gaps across multiple areas
Critical issues are widespread or systemic — not isolated gaps but missing structural elements across multiple planning artifacts. Typical triggers: PRD has no numbered FRs at all, epics do not cover user flows end-to-end, Architecture document is missing entirely, or stories throughout the epics have no acceptance criteria whatsoever. The planning documents must be substantially reworked before any development begins.
Fix Critical issues before ANY development · Full re-check required after

severity-classification Severity Classification

Every issue logged to the report is assigned one of three severity levels. Severity determines the fix timeline — not priority within a category. All Critical issues must be resolved before the overall verdict can improve.

Severity Levels — Fix Timing Matrix check-implementation-readiness · v6.10.0
Severity When to fix Typical examples
🔴 Critical Before development begins — blocks ALL Sprint 1 work FR uncovered in any story; Epic 2 story referencing Epic 3 component (forward dependency); technical-only epic with no user value (e.g., "Setup Database"); missing starter template setup story when Architecture specifies a starter; story with no acceptance criteria whatsoever
🟠 Major Before the affected story begins — blocks that story only Vague or untestable acceptance criteria; database table creation timing violation (all tables created in Story 1 upfront); story referencing a feature from a future story in the same epic; UX requirement implied by PRD but no UX document; UX component not supported by Architecture; brownfield project with no migration or integration stories
🟡 Minor Can be fixed incrementally — does not block sprints Formatting deviations in the epics document; minor structure deviations from BMAD templates; documentation gaps that do not affect story execution; story titles that are slightly technical in phrasing but still deliver recognizable user value

The overall verdict is determined by Critical issues. One or more isolated Critical issues = NEEDS WORK. Widespread, systemic Critical issues across multiple artifacts = NOT READY. Major issues do not change the overall verdict but are logged explicitly so sprint planning can account for them. Minor issues are listed in the report but do not affect sprint sequencing.

scope-boundaries Scope Boundaries — What This Check Does NOT Cover

This skill is a planning-layer validator. Understanding its boundaries prevents mis-use and sets correct expectations for what a READY verdict means.

  • Architecture ↔ Story technical feasibility. Whether a proposed technical approach in a story is sound, whether the chosen database can handle the load described, or whether the API design is scalable — these are architecture review concerns, not readiness check concerns. The check validates that the planning documents are complete and internally consistent, not that the technical design is correct.
  • Story content vs. actual codebase. For brownfield projects, the check does not compare story descriptions against the existing codebase to verify achievability. It only validates planning artifacts against each other.
  • Qualitative UX judgment. Whether wireframes are well-designed, whether user flows are intuitive, or whether the visual hierarchy is appropriate — these are UX quality judgments. The check validates only that UX requirements stated in the PRD are reflected in the UX document and that the Architecture supports the components the UX requires.
  • NFR coverage in epics. Non-Functional Requirements (performance, security, reliability) are validated for presence in the PRD — but the check does not verify that each epic explicitly handles each NFR. NFRs must be embedded in story acceptance criteria per the BMAD method; this is a QA and story-authoring concern, not a planning-gate concern.
  • Test coverage or test plan quality. Implementation readiness is not a test readiness check. Use bmad-testarch-test-design for test planning and test coverage validation.

§ WHY Design Rationale — Why this approach for each decision

why-formal-gate Why a Formal Gate?

Without a named gate, planning artifacts drift into development with unresolved gaps. Teams make implicit decisions about missing FRs — often wrong ones — and discover the gaps mid-sprint when the cost of change is highest. A formal, structured gate forces resolution of those gaps while changes cost almost nothing.

The cost-of-change ratio across the software lifecycle is well-established: fixing an issue costs approximately 1× in planning, 10× in active development (code, tests, and integration work must be redone), and 100× in production (deployment, rollback, customer impact, and rework compound). Catching a missing FR before Sprint 1 means updating a Markdown file. Catching it in Sprint 4 means reworking three stories, re-testing two epics, and potentially delaying the release by weeks.

The gate does not add bureaucracy — it adds structure to a decision point teams were already making informally. If a team was proceeding to development, they were implicitly deciding the planning was good enough. This skill makes that decision explicit, documented, and defensible.

why-step1-only Why Only Step 1 Is Interactive?

After Step 1 completes, John (PM) has a complete inventory of all planning documents — their paths, their versions, and their completeness status. The agent has everything it needs to execute Steps 2–6 autonomously. Requiring [C] Continue after every step would force the user to babysit a process that has no decision points requiring human judgment until the final verdict.

The only reason Step 1 is interactive is the duplicate detection block: if both a whole-file version and a sharded version of the same document type exist, the agent cannot choose which version to use — that decision carries information only the user has (which version is current, which was deprecated). Once that uncertainty is resolved, the remaining five steps execute on deterministic rules: extract FRs, map to stories, check alignment, evaluate quality, produce verdict.

Requiring [C] after every step is theater, not quality control. It creates the illusion of human oversight without providing it — because the user has nothing meaningful to review between steps except intermediate extraction results they did not create and cannot validate without re-reading the source documents themselves.

why-coverage-and-quality Why Validate BOTH Coverage AND Quality?

Coverage alone — "every FR has a story" — is insufficient. A story that says "Users can log in" covers a login FR but is not testable, because it does not specify what credentials are acceptable, what happens on failure, or what the success state looks like. A team following that story will build something, but it may not be the right something.

Quality alone — "every story has BDD-format AC" — is also insufficient. A perfectly written acceptance criterion for a feature that was never mapped to any PRD FR means the team built a feature that was not required. The product ships missing specified functionality while containing un-specified functionality — a worst-of-both-worlds outcome.

The combination of coverage validation (Step 3) and quality review (Step 5) closes both failure modes: nothing important is missing, and nothing present is too vague to execute on. This is the minimum viable validation needed before a team can start development with confidence.

why-duplicate-detection Why the Duplicate Detection Hard Block?

Ambiguous document versions are the single most common cause of validation errors in planning. When both a whole-file PRD and a sharded PRD directory exist in planning_artifacts/, the agent cannot determine which one was last updated or which the team is actually working from. Validating against the wrong version produces a report that is not just incomplete — it is wrong in a way that appears correct, which is more dangerous than a visible error.

Consider the failure mode: the sharded PRD was updated with three new FRs but the whole-file PRD was not regenerated. A validation against the whole-file version reports 100% coverage for the old FR set — missing the three new requirements entirely. The team proceeds to Sprint 1 confident that everything is covered, when three FRs have no stories.

The hard block forces the user to make an explicit, documented choice before the validation runs. This is not a recoverable warning — it is a prerequisite for the report to have any validity at all. The cost of the hard stop (two minutes of user action) is negligible compared to the cost of the failure mode it prevents.

why-not-arch-feasibility Why NOT Validate Architecture Feasibility?

Technical feasibility — whether the chosen architecture can actually implement the stories — is a design question, not a planning question. Answering it requires deep technical analysis: database query planning, API throughput modeling, infrastructure cost analysis, security threat modeling. These are the domain of Winston (Architect) and the architecture review process, not the planning gate.

The implementation readiness check asks a different question: "Is the planning layer complete and consistent?" This is a document-to-document validation, not a document-to-reality validation. It checks that the PRD FR list and the epics coverage map match — not that the architecture can implement any of them at scale.

Conflating the two produces a check that is either too narrow (reduced to a superficial scan because full feasibility cannot be automated) or too broad (days of analysis per project, delaying the gate indefinitely). The readiness check is designed to be fast, automatable, and definitive within its narrow, well-defined scope.

per-step 4 · Pre-Check Preparation

The check runs automatically — no user input between steps. All preparation must happen before invocation. These are the moments where you act, not during the run.

Pre-Check Readiness bmad-implementation-readiness · 6-step check
Preparation ItemWhat to do before invoking
Before Step 1 · Document Discovery Confirm all planning artifact file paths are correct and the files are at their final pre-dev state. If Architecture is missing, decide in advance: Continue (proceed without Architecture) or Pause (create Architecture first). For most projects, Continue is the right call.
Before Step 2 · PRD Analysis Know your PRD's FR and NFR count. When the check reports "FR-03 has no story coverage," you need to know whether FR-03 is in the current phase or a future phase — the fix is different.
Before Step 3 · Epic Coverage Validation Know the expected epic structure — how many epics, what each covers, and which FRs each epic is responsible for. Coverage gaps that look like errors may just be planned-future-phase work.
Before Step 6 · Final Assessment Prepare a fix plan template: for each Critical issue, know who owns the fix and what the target resolution is. The full check + fix cycle typically runs in 30–60 minutes — having owners identified in advance cuts the fix time in half.
What Happens in Each Step
  • Step 1 · Document DiscoveryAgent scans planning_artifacts/ for all docs. Detects whole-file vs. sharded versions. Flags duplicates (HARD BLOCK if found). Warns on missing required documents. Interactive — the only step requiring user input. Awaits [C] Continue before Step 2 proceeds.
  • Step 2 · PRD AnalysisReads the entire PRD — no summarizing allowed. Extracts all Functional Requirements (FR-1, FR-2…) with full text, all Non-Functional Requirements, and any unlabeled additional requirements. Full-text extraction with no compression.
  • Step 3 · Epic Coverage ValidationMaps every PRD FR to an epic or story. Flags FRs with no coverage as MISSING. Calculates coverage percentage. Documents impact of each uncovered FR. Missing FRs = Critical issues that block a READY verdict.
  • Step 4 · UX AlignmentValidates UX Spec ↔ PRD (user journeys vs. use cases) and UX Spec ↔ Architecture (components, performance targets). If UX Spec is absent, checks for implied UI requirements in the PRD. Logs all discrepancies to the report.
  • Step 5 · Epic Quality (9 checks)Autonomous sweep: (5a) user value focus per epic; (5b) independence and no forward dependencies; (5c) story sizing within one dev agent context; (5d) AC completeness and testability; (5e) within-epic dependency map; (5f) DB table creation timing (JIT); (5g) starter template story present when Architecture requires one; (5h) project type alignment; (5i) per-epic compliance checklist.
  • Step 6 · Final AssessmentAggregates all findings from Steps 1–5. Assigns one verdict: READY / NEEDS WORK / NOT READY. Lists all Critical issues with specific recommended actions. Produces ordered next steps by severity. Auto-invokes bmad-help for continuation guidance.

Technique Descriptions
6-Step Automated Check Process
StepWhat It Checks
Step 1 · Document Discovery Inventory all planning artifacts. Flag missing documents. If Architecture is missing, offer Continue or Pause.
Step 2 · PRD Analysis Extract the complete FR and NFR list. Identify key constraints and compliance requirements.
Step 3 · Epic Coverage Validation Map each FR to its corresponding stories. Produce a coverage matrix. Identify FRs with no story coverage — the most common Critical issue.
Step 4 · UX Alignment Compare UX Spec against PRD and Epics. Flag UX decisions not captured in any FR. Flag FRs with no UX coverage that imply a visual component.
Step 5 · Epic Quality Review Evaluate stories: user story format, AC completeness and testability, story sizing, dependency mapping. Flag "As a System" actor anti-pattern.
Step 6 · Final Assessment Synthesize all findings into a verdict (READY / NEEDS WORK / NOT READY) and prioritized action plan sorted by severity.
