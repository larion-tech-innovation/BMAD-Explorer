The core design phase — turns scenario outlines into development-ready written specifications a developer can build from without further design calls.
Replaces
This is the WDS-native equivalent of bmad-ux; Freya replaces Sally (BMM UX Designer) when WDS is installed.

Writes page specs, storyboards, and an updated design log so any later session resumes from the recorded state.

This skill produces the documents development is built from. The output is deliberately written spec documents — not Figma files, not code: structured text describing exactly what every page contains, how each section behaves, and what every state looks like. Freya transforms the scenario outlines from Phase 3 into per-page specifications stored within C-UX-Scenarios/NN-[scenario]/, alongside storyboard specifications and an updated design log.

Use it after Phase 3 Scenarios, once user flows are outlined and the Trigger Map and Project Brief are in place. Because the activity menu is non-linear, you can drop into conceptualization, sketch interpretation, specification, validation, or handover in whatever order the page in front of you needs — Freya reads the design log on activation to suggest where to go next.

output What a Page Specification Contains canonical

Every page spec produced by this skill follows the same structure, written within C-UX-Scenarios/NN-[scenario]/. A complete spec covers metadata, overview, ordered sections, and a full per-section breakdown.

  • Page MetadataName, URL, scenario context, and SEO title / description / keywords.
  • Page OverviewPurpose of the page, where the user is arriving from, and where they leave to.
  • SectionsEvery section on the page, listed in order.
  • ContentPer section — exact copy or a content brief.
  • InteractionPer section — what happens on hover, click, form submit, and error.
  • SpacingRelative guidance ("generous", "tight") or design token references.
  • TypographyHierarchy, weight, and size — described relatively.
  • State VariationsEmpty, loading, error, and success states for each section.
  • Object RegistryNamed components referenced in the spec — the seed for Phase 7.

§ WHY Why UX Design This Way? — The reasoning behind each design choice

why-written-specs Why Written Specs Instead of Pixels?

The WDS principle is that the spec is the new code. AI generates code readily but struggles to change it — once a feature is implemented, re-prompting toward a moving target is brittle and expensive. So the written specification, not the generated artifact, becomes the source of truth. A page spec describes content, interaction, spacing, typography, and every state in structured text a developer (or Mimir) can implement directly. Because it is text, it is reviewable, diffable, and durable: when something needs to change, you change the spec and regenerate, rather than wrestling the code. Figma files and pixels are downstream of the spec, not a substitute for it.

why-validate Why Validate ([SA]) Before Handoff?

The Validate activity is Freya's self-audit against a quality rubric, run before anything reaches Mimir. It catches the failures that otherwise surface as implementation ambiguity: missing content, undefined interactions, empty-state gaps, navigation errors, and SEO incompleteness. Each of these, left in a spec, forces the developer to guess — and a guess made during implementation is far more costly to unwind than a gap closed during specification. Validating before handoff is how Phase 4 keeps its promise that Mimir can build without coming back with design questions.

why-object-registry Why the Object Registry Matters

Named components registered in a page spec are not a formality — they are the mechanism by which Phase 7 (Design System) knows what to extract. The design system grows organically from actual usage rather than upfront planning, which means it can only harvest components that were named and registered as they appeared in real specs. Skip the registry and the Phase 7 extraction has nothing to work from; maintain it diligently and the design system assembles itself from the work already done.

why-trigger-map Why Always Load the Trigger Map First?

Freya's first principle is strategy before design — design without strategic context produces generic results. The Trigger Map carries the business goals and user psychology that should drive every design choice; loading it first means each section, each piece of copy, and each interaction is anchored to a reason rather than decoration. Freya loads it automatically as her first action, but it is worth verifying it exists — if the Trigger Map is missing, the design has nothing strategic to connect to.

The Design Log Suggests Your Next Move

Because Freya reads the design log on every activation, she does not need you to remember where you left off. The log's Backlog / Current / Design Loop Status / Log structure lets her open a session by proposing the next logical action — the page to specify next, the spec ready to validate, or the flow ready for handover. Trust that suggestion as your starting point.

workflow-conceptualize Conceptualize / Discuss 1 step

The default design starting point. Lead a focused creative dialog for each page — what does it need, can we simplify it, then visualize and specify. Use this whenever a page needs design thinking before anything is committed to spec. Freya asks two anchor questions (D1: what is the main action; D2: can we simplify or eliminate this step entirely), then continues a natural discussion grounded in the Trigger Map before offering wireframe, sketch, or direct-to-spec visualization options.

Step 1 · Page Design Dialog

Sets the scene from the scenario context (device, arrival state, driving forces), runs the D1/D2 dialog, facilitates a natural design discussion, then presents wireframe / sketch / spec options. After each completed stage a two-option transition offers either the next logical step on this page or the next scenario page. The design log is updated at every transition — status values progress through discussed → wireframed → specified.

workflow-dream Dream Up Design autonomous

Uses the same step sequence as Suggest but with autonomous execution — Freya creates all pages in the scenario flow without waiting for confirmation at each step, then presents the complete result for user review. Best when the user trusts Freya to make reasonable decisions and prefers to react to a finished proposal rather than approve each move. The user can type stop or pause at any time to interrupt and switch to Suggest mode for finer control.

Requires an existing Phase 3 scenario — Freya will not invent a scenario here. If none exists, she routes back to Phase 3 to outline it first. After completion Freya presents a summary table of all pages created and key decisions made, then offers to start building or explore responsive/storyboard detail.

workflow-sketch Analyse Sketches 1 step

Interprets user-provided sketches — photos, screenshots, hand-drawn wireframes, or verbal descriptions — and translates them into structured page specifications. Use this when the user has already sketched something and wants Freya to read and formalize it rather than propose a fresh design. Freya reads holistically first, calibrates scale against previously analysed pages, identifies sections, and cross-references established patterns from the project before going section-by-section.

Step 1 · Sketch Analysis

Reads the sketch holistically to establish scale and identify sections, presents the interpreted structure for user confirmation, then goes section-by-section to identify objects top-left to bottom-right. For each section Freya presents her interpretation, the user confirms or refines, then content and translations are gathered before a batch specification is generated for that section. After all sections are complete the workflow returns to the main design loop.

workflow-suggest Suggest Design 8 steps

Collaborative step-by-step page creation — Freya proposes each step and waits for user confirmation before moving on. Use this when you want collaborative control over structure and direction before writing the full specification. Requires an existing Phase 3 scenario. After each page, Freya checks for complexity (responsive states, storyboard items, forms) and presents a two-option transition: explore that complexity or move to the next scenario page.

Step 8 · Page Context

Routes the user to the appropriate page-creation workflow based on what they have available — a sketch or wireframe triggers the sketch analysis path; a verbal concept triggers the discussion path. Establishes the "page first, visualization second" principle and confirms what materials exist before proceeding.

Step 9 · Page Name

Agrees on the page's canonical name — the handle used in the scenario folder, design log, and all cross-references. Names are kept short and descriptive so the log table stays readable.

Step 10 · Page Purpose

Defines the page's single primary purpose in one sentence. Forces clarity on what the page is for before any structure is proposed — a page without a clear purpose is a candidate for elimination or merger.

Step 11 · Entry Point

Establishes where users arrive from — which page, link, notification, or external source brings the user here. Entry point context drives the assumed knowledge and mental state the page must accommodate.

Step 12 · Mental State

Captures what the user is thinking, feeling, and hoping for when they land on this page — drawn from the scenario's Q4 answers (hopes and worries). All design decisions made from here are anchored to this mental state.

Step 13 · Desired Outcome

Defines what success looks like for both the user and the business when the user leaves this page. The desired outcome is the design's north star — every section, piece of copy, and call-to-action should serve it.

Step 14 · Variants

Identifies page variants — authenticated vs. unauthenticated states, empty vs. populated data, different user types, or A/B test candidates. Variants that require significantly different layouts become separate page specs; minor differences become state variations within a single spec.

Step 15 · Create Page Structure

Creates the initial page folder and boilerplate specification file within the scenario's folder, seeded with the metadata gathered in steps 8–14. This is the blank canvas the Specify workflow (steps-p/) will fill in during the next activity.

workflow-specify Write Specifications 9 steps

Creates a complete, implementation-ready page specification — the full written document Mimir (or a developer) builds from. Use this when a page structure already exists (from Suggest, Dream, or Sketch) and needs every detail filled in: content, interactions, component definitions, states, validation, spacing, and typography. The nine steps are sequential and must not be skipped. After completion the design log status advances to specified.

Step 1 · Page Basics

Captures fundamental page information — title, URL/route, user goal in one sentence, entry and exit points, and SEO data (primary keyword, secondary keywords, URL slug) for public pages. SEO fields are mandatory for any page that will be publicly indexed.

Step 2 · Layout Sections

Defines the page's section structure in order — what sections exist, their purpose, and their sequence on the page. Section order is a design decision: it reflects the information hierarchy and the path the user's eye should travel.

Step 3 · Components and Objects

Defines the components and objects within each section — type, ID, hierarchy (parent/child), and whether a component is page-specific or a candidate for the design system. Objects are named here and registered for Phase 7 extraction.

Step 4 · Content and Languages

Writes the actual content for every text element — headings, body copy, button labels, placeholder text, error messages — across all supported languages. This step produces the copy that will appear in the built product, not placeholder text.

Step 5 · Interactions

Defines user interactions for every interactive element — hover states, click behaviors, form submissions, navigation triggers, and dynamic content loading. The developer should be able to implement every interaction from this step without making design decisions.

Step 6 · States

Specifies all non-default page and section states — loading, empty, error, success, and any conditional variants (authenticated vs. guest, populated vs. empty list). Every state that can occur must be designed, not left to developer judgment.

Step 7 · Validation

Defines form validation rules and constraints — required fields, format rules, character limits, and the error messages shown for each validation failure. Validation is specified per-field, not globally, so the developer knows exactly what to enforce.

Step 8 · Spacing and Typography

Documents spacing relationships between sections and elements (using design tokens or relative descriptors like "generous" / "tight") and typography hierarchy — which text styles apply to headings, body copy, labels, and captions. Spacing tokens that appear for the first time are extracted to the design system immediately.

Step 9 · Generate Spec

Assembles all gathered information into the final formatted page specification document, saved to the page's folder within C-UX-Scenarios/NN-[scenario]/pages/. The generated spec is the authoritative output — the source of truth Mimir reads and implements from.

workflow-validate Validate Specs 10 steps

Systematically audits page specifications for completeness, consistency, and quality before anything is handed to development. Runs through ten validation checks in sequence and produces a written diagnostic report saved to _progress/validation-report.md. Run this activity after completing specifications and before triggering the Handover workflow. Issues found here are far cheaper to fix than gaps discovered during implementation.

Step 1 · Page Metadata

Verifies that the page spec declares platform, page type, primary viewport, interaction model, and navigation context — all inherited from the scenario's platform strategy. Missing or mismatched metadata is flagged as a critical issue because it breaks the developer's assumed technical context.

Step 2 · Navigation

Checks that entry points, exit points, breadcrumbs, and navigation items are fully defined and internally consistent. Navigation gaps are a common source of implementation questions — this step closes them before handover.

Step 3 · Page Overview

Validates the page's overall structure description — purpose statement, flow position, and the relationship to adjacent pages. An incomplete overview forces the developer to infer context from code rather than reading it.

Step 4 · Page Sections

Audits each section for completeness — content, interactions, states, and object definitions must all be present. Any section with placeholder text or undefined interactions is flagged for resolution.

Step 5 · Section Order

Verifies the section sequence follows a logical progression that matches the user's mental model and the page's desired outcome. Section order that contradicts the user flow is caught and flagged here.

Step 6 · Object Registry

Confirms all components used in the spec are registered in the Object Registry. Unregistered components cannot be extracted by Phase 7 — this step ensures nothing is left unnamed.

Step 7 · Design System Separation

Checks that page-specific components are correctly distinguished from design-system candidates. Components that appear more than once across specs should be extracted; page-specific implementations should not be incorrectly promoted.

Step 8 · SEO Compliance

Validates heading hierarchy, meta title and description, keyword alignment, and URL slug correctness for public pages. SEO gaps found in spec are trivial to fix; SEO gaps found after implementation require code changes and re-deployment.

Step 9 · Design System Consistency

Cross-checks component usage across all page specs in the scenario — the same component should use the same name, states, and behavior everywhere. Inconsistencies signal a naming or extraction problem that would surface as duplicated code.

Step 10 · Final Validation

Produces the overall quality assessment and final validation report. If all checks pass the spec is cleared for handover. If issues remain, Freya suggests which to fix first based on severity — critical issues block handover; warnings are documented for resolution.

workflow-visual Visual Design 3 steps

Creates visual representations of page designs using external tools and integrates the results back into the written specification. Use this when the user wants mockups, AI-generated assets, or a prototype to validate layout and visual direction before full development. Supported tools include Excalidraw (editable wireframes), Nano Banana (AI image generation), Google Stitch (HTML/CSS mockups), Figma (professional design), and HTML prototypes.

Step 1 · Visual Approach

Reviews the page specification and presents the available visual tools — Excalidraw for layout iteration, Nano Banana for AI-generated assets, Stitch or HTML for production-quality mockups, Figma for precise design work. The user chooses the tool; if Nano Banana is selected for the first time, a setup verification step runs before proceeding.

Step 2 · Generate Visual

Executes the visual generation using the chosen tool — drawing the Excalidraw layout, composing and running the Nano Banana prompt, generating the Stitch component, or scaffolding the HTML prototype. The generated artifact is saved to the page's Sketches/ folder and referenced in the spec.

Step 3 · Review and Integrate

Presents the generated visual for review, iterates based on feedback, and integrates the agreed result back into the page specification. The spec is updated to reference the visual artifact as the approved layout reference — implementation follows the spec, not the visual directly.

workflow-design-system Design System 3 steps

Defines, updates, and reviews design system components used across page specifications. Use this to extract shared components after multiple pages are specified, update an existing component definition, or review the design system for consistency and gaps. Extraction follows the "second use" rule — a component stays inline in its first page spec and is extracted to the design system only when it appears a second time. Spacing tokens extract on first use.

Step 1 · Review Current Design System

Loads and inventories all components in D-Design-System/ — name, category, usage count across page specs, and last updated date. Cross-references the inventory against all page specifications to identify components used but not defined, components defined but unused, and inconsistencies in how the same component is referenced across pages.

Step 2 · Define Component

Walks through defining or updating a specific component — type, variants, states, behavior, and usage guidelines. Component definitions follow the object-type templates in data/object-types/ and are saved to the appropriate category folder in D-Design-System/.

Step 3 · Validate Usage

Checks that the newly defined or updated component is referenced consistently across all page specifications — same name, same states, same behavior. Usage inconsistencies are surfaced as a list of pages that need updating, so the design system stays the single source of truth.

workflow-handover Design Delivery / Handover 6 steps

Packages a complete, testable scenario flow into a Design Delivery (DD-XXX) and hands it off to BMad (Mimir) for implementation. Use this when a scenario is fully designed — all page specifications exist, validation has passed, and the flow is end-to-end testable. This workflow does not start until the completion checklist confirms no gaps or blockers remain.

Step 1 · Detect Completion

Runs a checklist across Phase 4 outputs (all scenarios specified, interactions defined, error states designed) and Phase 5 design system outputs (all components defined, tokens documented, states and variants complete). Also verifies the flow is testable end-to-end, delivers measurable value, and has no unresolved design decisions. Does not proceed until the flow is confirmed complete.

Step 2 · Create Delivery

Packages the scenario specifications, design system components, and supporting artifacts into a Design Delivery document (DD-XXX) using the delivery template. The DD is the formal handover artifact — it contains everything Mimir needs to implement the flow without returning with design questions.

Step 3 · Create Test Scenario

Defines the validation tests that confirm the implemented flow matches the design intent — user-facing acceptance criteria, edge cases to verify, and the definition of "done" for this delivery. The test scenario travels with the DD so Mimir knows what to verify after building.

Step 4 · Handoff Dialog

Runs a structured handoff conversation — Freya summarizes the key design decisions, highlights anything non-obvious in the specs, and flags any areas where the developer should exercise judgment. Uses the handoff dialog scripts from data/handoff-dialog-scripts.md to ensure nothing important is left unsaid.

Step 5 · Hand Off

Executes the official handoff — saves the DD to the project output folder, updates the design log to reflect the handover status, and signals to BMad that the delivery is ready for development. The design log's Backlog and Current tables are updated so the next session starts with an accurate picture of what remains.

Step 6 · Continue

Returns to the design log to determine what comes next — another scenario to design, a spec to validate, or a completed project. Presents the adaptive dashboard summary so the next session or the next phase can start from a clear state.
