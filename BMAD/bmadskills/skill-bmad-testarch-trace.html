§ WHAT Coverage Traceability & Quality Gate — What does this Skill do?

Skill code
bmad-testarch-trace  ·  Menu code: TC
Goal
Answer the question: "Do we have enough test coverage to release?" — map every requirement to the tests that cover it, identify gaps by priority, and produce a deterministic release verdict.
Role
Master Test Architect. Operates as a two-phase data pipeline: Phase 1 collects and maps coverage; Phase 2 applies deterministic gate rules and emits the verdict.
Knowledge base
test-priorities-matrix.md · risk-governance.md · probability-impact.md · test-quality.md · selective-testing.md
Output
traceability-matrix.md (human-readable) · e2e-trace-summary.json (CI/CD) · gate-decision.json (when gate-eligible)

Two-Phase Architecture

The workflow is structured as two distinct phases separated by a temp-file handoff. Phase 1 is data collection; Phase 2 is decision-making. They cannot be interleaved.

Phase 1 · Steps 1–4

Coverage Matrix

  • Resolve coverage oracle (requirements, spec, or inferred journeys)
  • Discover & categorize all tests by level
  • Map oracle items → tests (build matrix)
  • Analyze gaps by priority + heuristics
  • Write complete matrix to temp JSON file
Phase 2 · Step 5

Gate Decision

  • Read Phase 1 matrix from temp file
  • Apply deterministic gate rules (P0/P1/Overall thresholds)
  • Apply oracle confidence overlay
  • Write gate-decision.json + e2e-trace-summary.json
  • Display final verdict with rationale
Phase handoff

Step 4 records the exact temp file path in the traceability-matrix.md frontmatter as tempCoverageMatrixPath. Step 5 reads this key — if missing, it halts rather than reconstructing the path, because the timestamp cannot be reliably replicated in a different execution context.

§ WHY Why Coverage Traceability?

Without a traceability matrix, release decisions rely on developer intuition and informal coverage estimates. Gaps in P0/P1 requirements slip through undetected. The trace skill replaces guesswork with a deterministic, auditable gate signal that CI/CD pipelines can consume directly.

Dimension Without Trace With Trace (TC)
Release decision Based on developer confidence — no structured evidence Deterministic gate verdict (PASS / CONCERNS / FAIL / WAIVED) with documented rationale
Coverage visibility Unknown which requirements are actually tested Every requirement mapped to specific tests with FULL / PARTIAL / NONE status
Priority gaps P0 and P1 gaps only discovered post-release P0–P3 gap analysis before release — P0 must be 100%, P1 ≥ 90%
Blind spots Auth negative paths, API endpoint gaps, error handling silently untested 5 structural heuristics (H1–H5) surface common coverage blind spots
CI/CD integration No machine-readable release signal gate-decision.json consumed by pipelines for automated enforcement
Coverage source Assumed from test counts alone Traced against formal requirements, API contracts, or inferred journeys (4 oracle types)
Audit trail None e2e-trace-summary.json — always emitted, versioned, links requirements to evidence

Workflow Steps — Create Mode (5 Steps)

STEP 01 Phase 1 Resolve Coverage Oracle & Load Knowledge Base step-01-load-context.md

Step 1 does 3 things in sequence before moving to the next step:

① Find the Coverage Oracle

  • Automatically searches in priority order — stops as soon as the best available type is found
  • Step 1: Is there a User Story / Acceptance Criteria / PRD / Test Design? → Use it (best option)
  • Step 2: Is there an OpenAPI / Swagger / GraphQL Schema? → Use it
  • Step 3: Is there a link to Jira / Confluence / Linear? → Follow it if an adapter is available
  • Step 4: Nothing available → Read source code and infer user journeys (confidence: medium)
  • If none of the 4 types can be resolved → skill halts and asks you what is missing

② Load evaluation rules

  • Automatically loads internal rules: priority matrix (P0–P3), risk governance, test quality — no action needed from you

③ Read project artifacts & report

  • Reads available files: story file, test design doc, tech spec, API contract, route maps
  • Reports clearly: which Oracle was selected, confidence level (high / medium / low), and why that Oracle was chosen
  • Saves results to traceability-matrix.md for subsequent steps to use
STEP 02 Phase 1 Discover & Catalog Tests step-02-discover-tests.md

Step 2 does 3 things in sequence — scan, classify, then flag blind spots:

① Find all relevant tests

  • Scans the test folder for test files matching the oracle — by test ID, feature name, or spec file pattern (*.spec.*, *.test.*)
  • If the oracle is synthetic (inferred from source code): also matches by route names, UI labels, form actions (create, edit, save, delete, checkout), and auth/logout flows
  • Halts and reports if no tests are found for the given scope

② Classify each test by level

  • Tags every test as one of: E2E · API · Component · Unit
  • Records key info per test: ID, title, file location, test level, and whether it is skipped/pending/blocked

③ Flag common coverage blind spots

  • API gaps: endpoints referenced in requirements but with no direct API test
  • Auth gaps: login/permission flows that only test the happy path — missing "access denied" or invalid token scenarios
  • Error-path gaps: requirements covered only by happy-path tests — no validation error, timeout, or server-error scenarios
  • UI journey gaps: screens or user flows in the oracle with no E2E or component test covering them
  • These signals are passed to Step 3 and Step 4 to produce accurate gap findings in the final report
STEP 03 Phase 1 Map Oracle Items → Tests (Build Matrix) step-03-map-criteria.md

Step 3 does 2 things — connect requirements to tests, then verify the connections are genuinely complete:

① Build the traceability matrix

  • Takes each requirement from the oracle (user story AC, API endpoint, or inferred journey) and finds which tests cover it
  • Assigns a coverage status to each requirement:
  • FULL — critical path and key failure scenarios are both tested
  • PARTIAL — happy path only, or missing an important negative/error scenario
  • NONE — no tests found for this requirement
  • UNIT-ONLY / INTEGRATION-ONLY — covered at one level but missing higher-level validation

② Validate that "covered" actually means covered

  • P0/P1 requirements must have coverage — if any P0 or P1 item maps to zero tests, it is immediately flagged as a critical gap
  • Happy-path-only = PARTIAL, not FULL — a requirement is not fully covered if it only has success-scenario tests and no error/invalid-input tests
  • API requirement without endpoint test = not FULL — must have a test that directly calls the endpoint, not just a UI test that happens to trigger it
  • Auth requirement without a "denied" test = not FULL — must include at least one test verifying that unauthorized users are blocked
  • UI journey without an E2E test = not FULL — must have a test that walks through the complete flow, not just unit tests of individual components
STEP 04 Phase 1 — Final Analyze Gaps & Generate Coverage Matrix step-04-analyze-gaps.md

Step 4 is the final step of Phase 1. It takes everything built in Steps 1–3 and produces a complete picture of what is covered, what is missing, and what to do about it — then hands that picture to Step 5 for the gate decision.

① Identify and prioritize gaps

  • Collects all requirements with status NONE (no tests) and PARTIAL (incomplete tests) from the matrix built in Step 3
  • Sorts gaps by priority — P0 gaps are URGENT, P1 gaps are HIGH, P2 are MEDIUM, P3 are LOW
  • Also pulls in the blind spots flagged in Step 2: API endpoint gaps, auth "denied" gaps, error-path gaps, UI journey gaps

② Calculate coverage numbers

  • Counts how many requirements are FULL, PARTIAL, and NONE
  • Calculates coverage percentage per priority tier: P0 %, P1 %, P2 %, P3 %, and overall %
  • Only FULL status counts toward the coverage percentage — PARTIAL does not

③ Generate actionable recommendations

  • P0 gaps (URGENT): create tests immediately via bmad-testarch-atdd
  • P1 gaps (HIGH): expand coverage via bmad-testarch-automate
  • PARTIAL requirements (MEDIUM): complete missing failure/error scenarios
  • Auth / API / UI gaps (HIGH): add denied-path, endpoint, and E2E tests
  • Synthetic oracle used (MEDIUM): promote inferred journeys into formal acceptance criteria

④ Display Phase 1 summary & pass to Step 5

  • Shows a summary: total requirements, coverage %, gap count by priority tier, and blind spot counts
  • Saves the complete coverage matrix for Step 5 to use when making the gate decision (PASS / CONCERNS / FAIL)
  • Step 4 does NOT make the gate decision — that is Step 5's job
STEP 05 Phase 2 — Final Gate Decision step-05-gate-decision.md

Step 5 is the final step — it reads the coverage data built in Steps 1–4 and makes the gate decision: can this story / epic / release move forward, or does it need more tests first?

Note

Step 5 produces one of four gate statuses. The gate is only evaluated when allow_gate: true and collection_status: COLLECTED. Otherwise the result is NOT_EVALUATED and only the summary JSON is emitted.

① Apply gate decision rules

  • Reads the complete coverage matrix produced by Step 4
  • Applies 4 rules in order — the first rule that fails determines the verdict:
  • Rule 1 — P0 must be 100%: Any P0 requirement with NONE coverage → FAIL
  • Rule 2 — Overall must be ≥ 80%: Too many gaps across all priorities → FAIL
  • Rule 3 — P1 must be ≥ 80%: Too many core features uncovered → FAIL
  • Rule 4 — P1 ≥ 90% and all above pass: → PASS
  • Rule 5 — P1 is 80–89% and all above pass: → CONCERNS (can proceed but gaps must be tracked)

② Apply confidence adjustment

  • If the oracle used was synthetic (inferred from source code, not from formal AC), a PASS is automatically downgraded to CONCERNS — coverage numbers look good but the reference itself is uncertain
  • If no active tests were found at all, confidence is also downgraded regardless of oracle type

③ Write outputs & final report

  • Appends the gate verdict, rationale, coverage summary, and top recommendations to traceability-matrix.md
  • Writes gate-decision.json — machine-readable result consumed by CI/CD to block or allow the merge/deploy
  • Writes e2e-trace-summary.json — portable summary for dashboards and reporting pipelines

Configuration Reference

This section explains two config settings that control how the skill behaves. Both are set in (.claude/skills/bmad-testarch-trace/workflow.yaml). In most cases the defaults work fine — only change them when your situation requires it.

Decision Mode — who makes the gate call?

Controls whether the gate verdict is produced automatically by rule, or left to a human to decide.

ModeWhat happensWhen to use
deterministic Skill applies gate rules R1–R5 automatically and outputs PASS / CONCERNS / FAIL. Default. Use this for all standard runs.
manual Skill presents coverage stats only. A team member reads the report and makes the final call. Gate rules are shown as advisory — not enforced. When the team wants to override the rules for a specific release (e.g. known legacy gap, stakeholder decision).
How to change
Open workflow.yaml and set: decision_mode: "manual" (or back to "deterministic"). Restart the skill after saving.

Collection Mode — does the gate run at all?

Controls whether the skill performs a real trace and runs the gate, or skips because tests are tracked elsewhere or inaccessible.

ModeGate runs?When to use
contract_staticYesDefault — tests are in the repo; skill reads and traces them normally.
inventory_onlyYesOnly need a count and catalog of tests, no deep tracing required.
deferred_sharedNoCoverage is tracked in an external tool (e.g. TestRail). Skill imports the summary only; gate is skipped.
waivedNoGate deliberately skipped for this run. Requires stakeholder approval and a documented justification.
restrictedNoTest sources cannot be accessed due to policy or permission restrictions.
inaccessibleNoTest directory not found or unreadable — skill cannot proceed with tracing.
How to change
Open workflow.yaml and set: collection_mode: "deferred_shared" (or whichever mode fits your situation). When collection mode is anything other than contract_static or inventory_only, the gate does not run and only the summary JSON is emitted.

Output Files

FileDescription
{test_artifacts}/traceability-matrix.md Primary human-readable report — oracle summary, full requirements-to-tests matrix, gap analysis, gate verdict with rationale, recommendations
{test_artifacts}/e2e-trace-summary.json Machine-readable summary (schema v0.1.0) — coverage stats, priority breakdown by level, heuristics, blockers, recommendations, links. Always emitted (even when gate is NOT_EVALUATED). Consumed by CI/CD and reporting dashboards.
{test_artifacts}/gate-decision.json Slim gate signal — only emitted when gate-eligible (PASS/CONCERNS/FAIL/WAIVED). Contains verdict, rationale, P0/P1/Overall status flags, and links for downstream CI enforcement.
/tmp/tea-trace-coverage-matrix-{ts}.json Phase 1 temp file — complete coverage matrix JSON. Read by Step 5 via path stored in frontmatter. Not a permanent artifact.
After the Trace Report — Next Steps
  • PASS → proceed to release; optionally run bmad-testarch-nfr for NFR gate
  • CONCERNS → fix P1 gaps identified in recommendations; re-run trace or get waiver
  • FAIL → run bmad-testarch-atdd for P0 gaps (URGENT), bmad-testarch-automate for P1 gaps (HIGH); re-run trace
  • WAIVED → release proceeds; blockers logged for next sprint
