§ WHAT What does this Skill do?

The bmad-testarch-automate skill generates a full suite of automated tests for a project. Unlike bmad-testarch-atdd — which generates scaffolding-first tests before implementation — automate runs after code exists, either expanding coverage beyond what ATDD produced or generating tests from scratch for an existing codebase.

It detects the project stack (Frontend / Backend / Fullstack), determines which tests are needed at each level (E2E, API, Unit, Integration, Contract), then dispatches parallel worker subagents to generate each type concurrently. A final aggregation step combines outputs, creates shared fixture infrastructure, and validates quality.

Only Hard Requirement

A test framework config must exist (playwright.config.ts or cypress.config.ts). If missing, the skill halts and instructs the user to run bmad-testarch-framework first. All other inputs — story files, PRDs, ATDD outputs — are optional.

Execution Modes

The skill selects its execution mode automatically based on what artifacts are available in the project. The mode shapes how targets are identified in Step 2.

Integrated

BMad-Integrated Mode

  • Triggered when a story file, PRD, or test-design artifact is available
  • Maps acceptance criteria directly to test scenarios
  • Checks for existing ATDD outputs — avoids generating duplicate coverage
  • Expands beyond ATDD: edge cases, negative paths, variations
  • References risk scores and priority classifications from test-design
Standalone

Standalone Mode

  • Triggered when only source code is available (no story, PRD, or test-design)
  • Auto-discovers features by scanning source directory
  • Prioritizes by: untested paths → complex business logic → external integrations → critical user journeys
  • Performs browser exploration (Frontend/Fullstack) to verify actual UI selectors
  • Scans route handlers, OpenAPI/Swagger, DB models for Backend targets
Auto-discover Sub-mode

When neither target_feature nor target_files is specified, the skill enters auto-discover mode — it scans the entire source_dir and identifies features lacking test coverage. If source code cannot be found, the skill asks the user to specify explicit targets.

Worker Architecture

Step 3 operates as an orchestrator that dispatches specialized worker subagents. Each worker generates one type of test independently, writing its results to a timestamped temp file. Step 3C (Aggregation) then reads all temp files and writes everything to disk.

Worker A · Always

API Test Generation

step-03a-subagent-api.md
  • Generates tests/api/[feature].spec.ts
  • Uses apiRequest() helper if Playwright Utils enabled
  • Uses data factories for test data (no hardcoded values)
  • Tests happy path + error scenarios at each priority level
  • Performs Provider Source Scrutiny before any Pact interaction (if CDC enabled)
  • Output: /tmp/tea-automate-api-tests-{ts}.json
Output schema: success, tests[], fixture_needs, test_count
Worker B-E2E · Frontend / Fullstack

E2E Test Generation

step-03b-subagent-e2e.md
  • Generates tests/e2e/[feature].spec.ts
  • Uses browser automation for selector verification (cli / mcp / auto / none)
  • Applies network-first pattern: page.route() before page.goto()
  • Uses resilient selectors: getByRole, getByLabel, getByText
  • Closes CLI sessions with close, never close-all
  • Output: /tmp/tea-automate-e2e-tests-{ts}.json
Output schema: success, tests[], fixture_needs, test_count
Worker B-backend · Backend / Fullstack

Backend Test Generation

step-03b-subagent-backend.md
  • Generates unit tests (tests/unit/) and integration tests (tests/integration/)
  • Detects framework language: pytest / JUnit / go test / xUnit / RSpec
  • Uses idiomatic patterns per framework (table-driven Go tests, @Nested JUnit, etc.)
  • Mocks all external dependencies (no real DB/queue calls in unit tests)
  • Generates contract tests when microservices patterns detected
  • Output: /tmp/tea-automate-backend-tests-{ts}.json
Output schema: success, subagentType, testsGenerated[], coverageSummary

Step 3C: Aggregation

After all workers complete, the aggregation sub-step reads all temp files, writes test files to disk, deduplicates fixture needs across workers, generates shared fixture infrastructure, and calculates combined statistics for the summary report.

Fixture Aggregation
  • Collects all fixture_needs from every launched worker
  • Deduplicates: one auth fixture, one data factory per entity type
  • Categorizes: Authentication, Data Factories, Network Mocks, Test Helpers
  • Generates files: tests/fixtures/auth.ts, tests/fixtures/data-factories.ts, tests/fixtures/network-mocks.ts, tests/fixtures/helpers.ts

§ WHY Why use this Skill?

ATDD scaffolds happy-path tests. Implementation produces code. But coverage after those two steps is still shallow — edge cases, error states, negative paths, contract boundaries, and cross-layer integration are left uncovered. Automate fills that gap systematically.

Dimension Without Automate With Automate (this skill)
Post-ATDD coverage Tests only cover acceptance criteria happy paths; edge cases, negative paths, and P2/P3 scenarios are missing Expands ATDD output to full coverage: error states, boundary conditions, variations — deduplicating what ATDD already covered
Existing codebase No systematic method to add tests — developers guess what to cover, write tests inconsistently Auto-discovers untested paths by scanning source, routes, OpenAPI specs, and DB models; prioritizes by criticality
Multi-stack complexity API, E2E, and unit tests written separately with different patterns, tools, and test data strategies 3 workers run in parallel (API + E2E + Backend), each generating idiomatic tests for its level from one invocation
Selector fragility E2E tests use CSS class selectors that break on UI refactors; flaky tests committed to main Browser exploration verifies real selectors; resilient getByRole/getByLabel patterns enforced; optional auto-healing loop
Framework consistency Tests follow different patterns across features — different auth helpers, different factory approaches, different fixture strategies All tests use the same shared fixture infrastructure, selector conventions, and factory functions generated once and referenced everywhere

When to Use

The skill is activated by user phrases or by the BMad framework routing to it after implementation is complete.

ContextTrigger phrase / conditionMode selected
Post-ATDD expansion "expand test coverage" · "add more tests" · "increase coverage" BMad-Integrated — reads ATDD outputs, avoids duplication
New feature complete "automate our tests" · "generate test suite for [feature]" · "test this feature" BMad-Integrated if story/PRD found; otherwise Standalone
Existing codebase "add tests to this codebase" · "we have no tests, start automating" Standalone — auto-discovers from source code
Targeted coverage gap "add tests for src/payment/" · "cover the checkout flow" Standalone with selective coverage target, explicit target_files
Prerequisite missing Framework config absent (playwright.config.ts / cypress.config.ts) HALT — "Run bmad-testarch-framework first"

§ HOW How to Execute

Execution Mode Resolution

Before dispatching workers, Step 3 resolves the best available execution mode. An explicit user instruction in the active run always takes priority; fallback logic applies when probing is enabled.

Priority 1 agent-team Runtime-native parallel squad; highest throughput
Priority 2 subagent Parallel subagents; ~40–70% faster than sequential
Priority 3 sequential One worker at a time; always supported; baseline speed
Config auto Probe capability, select best supported mode automatically
Resolution Precedence
  • Explicit user instruction during the active run (e.g., "use sequential")
  • tea_execution_mode value in workflow config (agent-team / subagent / sequential / auto)
  • Runtime capability probe result (when tea_capability_probe: true)
Probe Disabled Behavior

When tea_capability_probe: false, the requested mode is honored strictly. If that mode cannot be executed at runtime, the skill fails with an explicit error — no silent fallback.

Steps (C Mode)

The create mode runs four main steps. Step 3 is an orchestration step that dispatches parallel worker subagents; those workers are not counted as independent steps but as nested subagents within Step 3.

STEP 01 Preflight & Context Loading step-01-preflight-and-context.md
  • Detect stack: scan package.json, pom.xml, go.mod, requirements.txt etc. → classify as frontend / backend / fullstack
  • Determine execution mode: BMad-Integrated if story/PRD/test-design found; otherwise Standalone
  • Load tiered knowledge: Core always (test-levels, test-priorities, data-factories, selective-testing, ci-burn-in, test-quality); Extended on demand; Specialized when relevant
  • Read config flags: tea_use_playwright_utils, tea_use_pactjs_utils, tea_pact_mcp, tea_browser_automation, tea_execution_mode
  • Halt if framework scaffolding missing; instruct user to run bmad-testarch-framework first
STEP 02 Identify Automation Targets step-02-identify-targets.md
  • BMad-Integrated: map acceptance criteria → test scenarios; check existing ATDD outputs to avoid duplicate coverage; plan expansion (edge cases, negative paths)
  • Standalone: auto-discover features in source_dir, prioritize by coverage gaps and criticality
  • Frontend/Fullstack: optional browser exploration via playwright-cli to verify real UI selectors and user flows
  • Backend: scan route handlers, OpenAPI/Swagger specs, DB models, service integrations for test targets
  • Build Provider Endpoint Map (when tea_use_pactjs_utils enabled): table mapping consumer endpoints to provider route handlers
  • Produce coverage plan: targets by test level (E2E/API/Unit/Integration), P0–P3 priorities, scope justification
STEP 03 Generate Tests (Orchestrator) step-03-generate-tests.md
  • Resolve execution mode: agent-team → subagent → sequential (capability probe)
  • Generate unique timestamp for temp file naming (/tmp/tea-automate-*-{timestamp}.json)
  • Dispatch Worker A (API) — always launched regardless of stack
  • Dispatch Worker B-E2E (E2E) — launched for Frontend and Fullstack only
  • Dispatch Worker B-backend (Backend) — launched for Backend and Fullstack only
  • Wait for all launched workers; verify all expected output files exist
  • Report performance metrics (execution mode, elapsed time, parallel gain)
  • Sub-step 3C: Aggregate outputs, write test files to disk, generate shared fixtures, calculate summary stats
STEP 04 Validate & Summarize step-04-validate-and-summarize.md
  • Run checklist validation: framework readiness, coverage mapping, test quality, fixtures/factories, no orphaned CLI sessions
  • Optional test healing loop (when auto_heal_failures: true): run tests → identify failures → apply healing strategy → re-run; max 3 iterations
  • Mark unfixable tests with test.fixme() and detailed comments (what failed, what was tried, why healing failed)
  • Polish output: remove duplication, verify consistency, fill any gaps
  • Write final automation-summary.md with coverage plan, file list, assumptions, and recommended next workflow
Resume Support

Each step appends to automation-summary.md with YAML frontmatter tracking stepsCompleted, lastStep, and lastSaved. If a run is interrupted, the skill can resume from the last completed step without regenerating earlier work.

Test Types by Stack

The skill generates tests at the appropriate levels for each stack type, using the test-levels-framework knowledge fragment to avoid duplicate coverage (same behavior tested at multiple levels).

Test Level Worker Stack Output Path Purpose
API Worker A FE BE FS tests/api/*.spec.ts Business logic, service contracts, HTTP status codes, error handling
E2E Worker B-E2E FE FS tests/e2e/*.spec.ts Critical user journeys — happy path only, full system integration
Unit Worker B-backend BE FS tests/unit/*.[test/spec].* Pure business logic, edge cases, error handling (no external deps)
Integration Worker B-backend BE FS tests/integration/*.[test/spec].* Database, message queues, cache, external service clients
Contract (CDC) Worker A BE FS pact/http/consumer/, pact/http/provider/ Consumer-driven contracts via Pact.js (when use_pactjs_utils: true)

Contract Testing (CDC / Pact)

When tea_use_pactjs_utils: true, Worker A also generates Consumer-Driven Contract (CDC) tests using Pact.js. This is a distinct, opt-in capability requiring extra scrutiny steps.

CDC Provider Source Scrutiny Protocol

Before generating ANY Pact interaction, the subagent must read the provider's actual source code (or OpenAPI spec) to verify the real response shape. Generating contracts from consumer-side types alone is the #1 cause of verification failures.
