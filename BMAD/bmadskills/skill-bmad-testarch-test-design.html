§ WHY Why Test Design & Risk Assessment?

Without a test design, teams test what developers happen to think of — missing high-risk paths, duplicating effort across levels, and discovering architecture testability gaps only after implementation has started.

Dimension Without Test Design With Test Design (TD)
Risk visibility Risks discovered when they cause production failures 6-category risk register with P×I scores before implementation begins
Test scope Whatever developers and QA think of — ad-hoc, incomplete P0–P3 priority matrix tied to risk scores; every requirement decomposed into atomic scenarios
Architecture gaps Testability problems (no observability, shared state, tight coupling) found during testing Testability review (Controllability / Observability / Reliability) before the implementation gate
Test level selection E2E for everything — slow, brittle, expensive CI Right level per scenario: E2E critical paths, API business logic, Unit edge cases. No duplicate coverage.
Resource planning No estimates, or single-point exact numbers that erode trust when wrong Interval-based effort estimates per priority tier (~25–40 hours); timeline as week ranges
Cross-team alignment Developers and QA have different understanding of what "tested" means Shared Architecture doc (for Dev/Arch teams) and QA doc (for QA team), both referencing the same risk IDs
Epic handoff Requirements go straight to development; testing is an afterthought BMAD handoff doc maps P0/P1 risks to epic/story concerns before decomposition starts

step-03 · Testability & Risk Assessment

  • Filesteps-c/step-03-risk-and-testability.md
  • Testability reviewSystem-Level only. Evaluates 3 dimensions: Controllability (state seeding, mockability, fault injection), Observability (logs, metrics, traces), Reliability (isolation, reproducibility, parallel safety). Output: 🚨 ACTIONABLE CONCERNS first, then ✅ Assessment Summary.
  • Risk assessmentAll modes. Identify real risks (not features). Classify by 6 categories. Score P × I. Flag high risks (≥ 6). Define mitigation, owner, timeline.
  • ASRsArchitecturally Significant Requirements — each marked ACTIONABLE (must decide) or FYI (no decision needed).
How to Assess Testability — 3 Steps adr-quality-readiness-checklist.md · System-Level only
  • ① Read artifactsRead tech-spec.md, PRD.md, and architecture/ADR docs to understand what the system does and how it is designed. You need enough context to answer "does this exist?"
  • ② Score all 29 criteriaGo through all 8 ADR categories (29 criteria). For each criterion, mark one of three statuses based on what the artifacts say:
    • ✅ Covered — artifact confirms it exists
    • ⚠️ Gap — artifact does not mention it, or explicitly shows it is missing
    • ⬜ Not Assessed — not enough information to conclude
  • ③ Classify each ⚠️ GapFor every gap, ask one question: "If this is missing, will QA be blocked from writing or running tests?"
    • Yes → 🚨 ACTIONABLE — must be fixed before testing begins. Record: Concern · Impact · What architecture must provide · Owner · Timeline.
    • No → ✅ FYI — note it, no immediate action required.
All 8 Categories and Their Criteria
Category 1
Testability
4 criteria
  • – Components can be tested in isolation
  • – Test environment is separated from production
  • – External dependencies can be mocked or stubbed
  • – Test data can be provisioned programmatically
Category 2
Data Strategy
3 criteria
  • – Test data is isolated from production data
  • – Data factories exist for all major entities
  • – No shared mutable state between parallel test workers
Category 3
Scalability
4 criteria
  • – NFR thresholds are defined (latency, throughput, error rate)
  • – Load testing has been run at projected peak volume
  • – Auto-scaling rules are defined and tested
  • – Database connection pooling is configured correctly
Category 4
Disaster Recovery
3 criteria
  • – RTO and RPO targets are defined and documented
  • – Backup restore procedure has been tested
  • – Rollback procedure exists and has been dry-run
Category 5
Security
4 criteria
  • – Authentication mechanism is implemented and tested
  • – Authorization (RBAC/ABAC) is enforced at all endpoints
  • – Sensitive data is never logged or returned in error responses
  • – OWASP Top 10 risks have been addressed
Category 6
Observability
4 criteria
  • – Structured logging is in place on all services
  • – Metrics (latency, error rate, throughput) are being collected
  • – Distributed tracing is implemented (request IDs propagated)
  • – Alerts are configured for P0 error conditions
Category 7
Quality of Service
4 criteria
  • – Latency budgets are defined per endpoint
  • – Rate limiting is implemented and tested
  • – Circuit breaker or retry logic handles downstream failures
  • – SLA/SLO targets are documented
Category 8
Deployability
3 criteria
  • – Zero-downtime deployment is supported
  • – Feature flags exist for risky changes (can be turned off post-deploy)
  • – Deployment pipeline includes automated smoke tests

P × I Scoring Matrix

Each risk is scored on two dimensions (1–3 each). The product is the Risk Score.

Scoring Scale probability-impact.md
Dimension123
Probability Low — unlikely to occur Medium — possible High — likely or already seen
Impact Low — minor degradation Medium — significant feature impact High — blocks core flow, data loss, revenue impact
Score Thresholds P × I = 1–9
ScoreLevelRequired action
6 – 9 High Mandatory mitigation plan: numbered steps, Owner, Timeline, Status, Verification. Architecture doc includes these; QA doc references them.
3 – 5 Medium Document in risk register with mitigation approach. Monitor during implementation.
1 – 2 Low Note in risk register. Reassess if scope changes. No immediate action required.
  • All risks get a unique ID: R-001, R-002, …
  • Both Architecture doc and QA doc reference risks by the same IDs for cross-document consistency
  • Mitigation plans in Architecture doc cover Backend/DevOps/Arch/Security actions only — QA mitigations go in QA doc

Priority = risk and business importance, NOT execution timing. P0/P1/P2/P3 sections describe criteria and purpose — the Execution Strategy section separately handles when tests run.

Priority Table test-priorities-matrix.md · step-04
PriorityCriteriaPurposePass rate gate
P0 Blocks core functionality AND risk score ≥ 6 AND no workaround exists Must pass before any release. Failing P0 = ship nothing. 100%
P1 Critical paths + medium/high risk + common user workflows Important but not absolute blockers. 1 failure in many = acceptable with a plan. ≥ 95%
P2 Secondary features + low/medium risk + edge cases Valuable but acceptable to defer until after initial release. Best effort
P3 Nice-to-have + exploratory + benchmarks + performance baselines Track and improve over time. Never blocks release. No gate

step-04 · Coverage Plan & Execution Strategy

  • Filesteps-c/step-04-coverage-plan.md
  • Coverage matrixDecompose requirements into atomic test scenarios. Select test level (E2E / API / Component / Unit). Assign P0–P3. No duplicate coverage across levels.
  • ExecutionSimple model: PR / Nightly / Weekly. PR = all functional tests if <15 min. Nightly/Weekly = long-running or expensive suites. Do not re-list all tests.
  • EstimatesRanges only (e.g., "~25–40 hours" — never "36 hours"). P0/P1/P2/P3 + total + timeline as week ranges.
  • Quality gatesP0 pass rate = 100%, P1 ≥ 95%, coverage target ≥ 80%.

Test Level Selection

Test Levels test-levels-framework.md
LevelUse forAvoid when
E2E Critical user-facing flows; end-to-end happy paths; high-risk integrations Business logic testable at API level; edge cases better served by unit tests
API Complex business logic; contract validation; backend integrations; all non-P0 flows Purely UI interactions; pure computations without side-effects
Component UI component behavior; form validation; state management; accessibility Full user workflows that span multiple pages
Unit Algorithms; edge cases; pure functions; error handling branches Integration points; anything requiring external services

Execution Strategy — Keep Simple

Use the PR / Nightly / Weekly model. Do not create complex tier structures. Do not re-list all tests in this section — refer to the coverage plan.

  • Every PRAll functional tests (P0 + P1 + P2) — Playwright parallelises 100s of tests in 10–15 min. Run everything if under 15 min total.
  • NightlyPerformance tests (k6, Locust, JMeter) · Long-running suites (30–60 min) · Tests requiring large datasets
  • WeeklyChaos / resilience tests · Full regression pass · Exploratory / P3 scenarios
  • Principle"Run everything in PRs unless expensive/long-running." Avoid smoke/P0/P1/P2 tier hierarchy — complexity with no benefit.

Resource Estimates — Intervals Only

Always use ranges. Never give exact numbers like "36 hours" — use "~25–40 hours" instead. Exact calculations create false precision and erode trust when estimates miss.

Architecture Document — test-design-architecture.md

§ WHAT — Dedicated to the Architecture & Dev team. Covers concerns, testability gaps, and NFR requirements. Contains no test code or test scenarios — only what the architecture team must decide or provide before QA can begin writing tests. Target ~150–200 lines.

§ WHY — Architecture teams need a completely different view from QA: they need to know blockers (what prevents test development), not specific test scenarios. Separating this document prevents architects from reading through hundreds of test cases to find the decisions that require their attention. Actionable-first: 🚨 blockers always at the top, FYI at the bottom.

QA Document — test-design-qa.md

§ WHAT — Dedicated to the QA team. A test execution recipe: what to test, how to test it, with which tools, and what QA needs from other teams before testing can begin. Contains concrete test scenarios, P0–P3 coverage plans, execution strategy, and effort estimates. Complements — does not duplicate — the Architecture doc.

§ WHY — QA needs a completely different view from architects: they need test scenarios, execution timing, and tooling — not architectural blockers or NFR design gaps. Keeping this document separate lets QA work from a focused, actionable recipe while referencing the Architecture doc for risk context. Duplication between the two documents is explicitly avoided.

BMAD Handoff Document — test-design/{project}-handoff.md

§ WHAT — A bridge document generated at the end of System-Level mode only. It carries TEA's test design outputs — risk assessments, P0/P1 scenarios, quality gates — into BMAD's create-epics-and-stories workflow so that quality requirements are embedded in epic and story decomposition from the start, not added as an afterthought.

§ WHY — Without this handoff, test constraints stay siloed in QA documents and never reach the epic/story authors. Epics get written without quality gates; stories get acceptance criteria that ignore P0 risks. This document makes quality a first-class input to implementation planning by providing a structured mapping from risks and test scenarios to the exact epics and stories that must address them.

Epic-Level Document — test-design-epic-{epic_num}.md

§ WHAT — The primary output of Epic-Level mode. A single, self-contained document covering the complete test plan for one epic: risk assessment across all 6 categories, NFR planning, a prioritized test coverage plan (P0–P3), execution order, resource estimates, quality gate criteria, and mitigation plans for high-priority risks. One file per epic; consumed directly by QA engineers, dev leads, and story authors.

§ WHY — Unlike the system-level QA document (which spans the full product architecture), this document is scoped to a single epic — making it immediately actionable for the team doing the work. It gives story authors concrete P0/P1 test scenarios to embed as acceptance criteria, gives dev leads clear quality gate thresholds, and gives QA a prioritized execution plan sized to the epic's risk profile rather than the full system.

quality-gates Quality Gates & Key Checklist Items

Risk Assessment Quality
  • All risks have unique IDs (R-001, R-002, …)
  • Probability and Impact values are 1, 2, or 3 — not descriptive labels
  • Scores calculated correctly (P × I)
  • High-priority risks (≥ 6) clearly flagged and have mitigation plans
  • Mitigation strategies are specific and actionable (not "add monitoring")
  • Owner and timeline assigned to each mitigation
Coverage Matrix Quality
  • All requirements mapped to test levels
  • No duplicate coverage (same behavior tested at multiple levels)
  • P0 scenarios represent <10% of total scenario count
  • Risk linkage documented in coverage table
Execution Strategy Quality
  • Simple structure: PR / Nightly / Weekly — not complex tier hierarchy
  • PR execution: all functional tests unless infrastructure overhead is significant
  • No redundancy: coverage plan not re-listed in execution strategy
  • Philosophy stated: "Run everything in PRs if <15 min"
Resource Estimates Quality
  • All estimates use interval ranges (e.g., "~25–40 hours")
  • No exact calculations cited (no "18 × 2h = 36h")
  • Timeline as week range, not day count
Document Quality (Anti-Bloat)
  • Architecture doc: concerns-focused, NOT implementation-focused. No test code examples.
  • QA doc: implementation-focused, NOT theory-focused. No architecture summaries.
  • No repeated notes across 10+ sections — write once at the top, reference briefly
  • Professional tone — no excessive emojis, no "absolutely/fantastic/excellent"
  • Both docs use same risk IDs, same priority levels, same blocker references
