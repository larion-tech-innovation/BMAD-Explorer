Test Framework Setup

Automatically sets up a complete, production-ready test framework for your project — directories, config, fixtures, data factories, sample tests, and documentation. No manual scaffolding needed.

Skill code: bmad-testarch-framework

Stacks: Frontend · Backend · Fullstack

§ WHAT What does this skill do?

Skill code
bmad-testarch-framework
Menu code
TF
Role
Acts as a Master Test Architect. Detects your project's tech stack, picks the right test framework, and builds the entire test infrastructure for you.
Autonomous
Yes — runs automatically. Detects the stack, selects a framework, and generates all files without asking you for input at every step. Only stops if a prerequisite is missing.

Think of this skill as someone who sets up the testing "house" before anyone starts writing tests. It creates the folder structure, wires up the config, adds reusable helpers and data factories, writes a sample test, and documents everything — so the team can start writing actual tests immediately without worrying about boilerplate.

This skill must run before ATDD, Automate, or CI skills — they all depend on the infrastructure this skill creates.

Supported Stacks

The skill adapts to your project type. It detects whether you have a frontend, backend, or fullstack project and sets up the right framework for each.

Frontend

Browser Testing

  • Playwright (default) or Cypress
  • TypeScript config included
  • Fixture + mergeTests pattern
  • Faker-based data factories
  • Network interception helpers
Backend

API / Unit Testing

  • Python → pytest
  • Java / Kotlin → JUnit 5
  • Go → go test (built-in)
  • C# / .NET → xUnit
  • Ruby → RSpec
  • Rust → cargo test
Fullstack

Both Layers Together

  • Browser framework for frontend
  • Backend framework for the server language
  • Separate directory structures per layer
  • Single shared .env.example
  • One unified README
Outputs

What Gets Created

  • Framework config file
  • tests/ directory tree
  • Fixtures + data factories
  • Sample test (passes on first run)
  • .env.example
  • tests/README.md

§ WHY Why use this skill?

Every project eventually needs tests. The difference is whether the infrastructure is set up intentionally — with consistent patterns, clean isolation, and CI-ready config — or assembled piece by piece as needs arise, leading to inconsistency and technical debt.

Without vs With Framework Setup

Area Without Framework Setup With Framework Setup (TF)
Setup time Hours of boilerplate. Every developer decides their own folder layout and timeout settings. Minutes — the entire scaffold is generated in one pass with consistent structure across the team.
Test data Hard-coded values or manual database seeds that get stale. No cleanup after test runs. Faker-based data factories with entity tracking and cleanup(). Each test run starts clean.
Fixtures State leaks between tests. Fixtures copy-pasted across files, teardown missing or incomplete. Pure function → fixture → mergeTests pattern enforced. Auto-cleanup built in from day one.
First test run Config may fail silently — wrong base URL, browser not installed, reporters not wired. Validated config with a sample test written to pass on the very first npm run test:e2e.
CI integration Manual env config per environment. JUnit XML output not set up — CI can't parse test results. JUnit + HTML reporters pre-configured. Env vars templated in .env.example. CI-tuned retries and workers.
Onboarding "Ask the person who set this up" — undocumented steps, unclear how to run in headed mode. tests/README.md covers setup, running, debugging, architecture, best practices, and CI notes.

When to invoke this skill

"setup test framework" Starting a new project and need the full test infrastructure from scratch.
"initialize testing framework" Project exists but has no tests/ directory or framework config yet.
"setup Playwright" / "setup pytest" You want a specific framework — overrides the auto-selection logic.
"initialize test infrastructure" Existing project that needs proper fixtures, factories, and helpers added.
Before running ATDD ATDD requires a fixture index and mergeTests pattern to already exist.
Before running Automate Automate requires tests/support/ directories and data factories to be present.
Dependency Rule

Framework → ATDD → Automate → CI. Always run this skill first. The others depend on the files it creates.

§ HOW How to run it

4 Operating Modes

When you invoke the skill, it asks which mode you want. Pick the one that matches your situation.

C

Create

First-time setup. Runs all 5 steps and builds the full infrastructure from nothing.

R

Resume

Picks up an interrupted Create run from where it left off. Reads the saved progress file and skips completed steps.

V

Validate

Audits an existing framework setup against the full checklist. Reports PASS / WARN / FAIL per item — no files are changed.

E

Edit

Revises one specific output file (config, fixtures, README) after the initial setup. Only touches what you ask it to change.

Create Mode — 5 Steps

This is the main flow. Steps run sequentially and progress is saved after each one, so the skill can resume if interrupted.

STEP 01 Preflight Checks step-01-preflight.md
  • ① Detect the stack — reads config.test_stack_type; if set to auto, scans the project root for manifest files to determine frontend / backend / fullstack. See detection table below.
  • ② Validate prerequisites — checks that a project manifest exists, no conflicting framework is already installed, and write permissions are available. Halts immediately if any check fails.
  • ③ Save context — writes detected stack type and project details to {test_artifacts}/framework-setup-progress.md for use in later steps.
StackDetected byWhat gets set up
frontend package.json with React / Vue / Angular / Next deps; playwright.config.*; vite.config.* Browser-only testing. Playwright or Cypress. TypeScript config.
backend pyproject.toml · pom.xml · go.mod · *.csproj · Gemfile · Cargo.toml API / unit testing. Framework matched to language. No browser setup.
fullstack Both frontend and backend indicator files present Both browser and backend frameworks. Separate directory structures. Unified .env.example.
Override Rule

An explicit test_stack_type in config always wins over auto-detection. If you set it, the scan is skipped entirely.

STEP 02 Framework Selection step-02-select-framework.md
  • ① Check for explicit config — if config.test_framework is set to a specific value (not auto), that value is used directly. No further logic needed.
  • ② Apply selection logic — for frontend: default is Playwright, Cypress only when specific criteria apply (see table below). For backend: maps the detected language to its idiomatic framework.
  • ③ Announce decision — states which framework was selected and why before proceeding.

Frontend — Playwright vs Cypress

STEP 03 Scaffold the Framework step-03-scaffold-framework.md
  • ① Create directory structure — builds all folders: tests/e2e/, tests/support/fixtures/, tests/support/helpers/, etc. (exact layout depends on stack).
  • ② Generate framework config — creates playwright.config.ts (or equivalent) with timeouts, reporters, parallelism, base URL, and CI-tuned settings. Also creates .env.example and .nvmrc.
  • ③ Create fixtures and factories — implements the fixture index with the mergeTests pattern and at least one Faker-based data factory with entity tracking and cleanup(). Loads extra knowledge fragments (Playwright Utils, Pact.js) if configured.
  • ④ Generate sample tests and helpers — writes a working example test and shared helpers (API helper, auth helper) that demonstrate the fixture pattern.
STEP 04 Documentation & Scripts step-04-docs-and-scripts.md
  • ① Write tests/README.md — covers: how to set up, how to run tests (headless and headed), folder architecture, best practices, CI integration notes, and links to knowledge base references.
  • ② Add test scripts to project config — adds the minimal test command to package.json (frontend: "test:e2e": "npx playwright test"), Makefile, pyproject.toml, or build.gradle depending on the stack.
STEP 05 Validate & Summarize step-05-validate-and-summary.md
  • ① Run the checklist — goes through all items: preflight passed, directory structure correct, config valid, fixtures and factories present, docs and scripts in place. Fixes any gaps found.
  • ② Report results — summarizes: framework selected, all artifacts created, knowledge fragments applied, and recommended next steps (run ATDD, Automate, or CI skill).

Validation Checklist — Key Items

Step 5 (and Validate mode) checks these items. All must pass for the setup to be considered complete.

Prerequisites
  • Project manifest exists (package.json, pyproject.toml, pom.xml, go.mod, *.csproj, Gemfile, or Cargo.toml)
  • No conflicting framework already installed
  • Stack type correctly identified (frontend / backend / fullstack)
Directory & Config
  • tests/ root directory created
  • tests/support/ directory created — this is the critical pattern for frontend projects
  • Framework config file created and syntactically valid
  • Timeouts, reporters, parallelism, and artifact capture correctly configured
  • .env.example contains TEST_ENV, BASE_URL, API_URL
Fixtures & Factories
  • Fixture index with mergeTests pattern implemented
  • At least one data factory using Faker with entity tracking and cleanup()
  • Auto-cleanup logic present in fixture teardown
Sample Tests
  • Example test file created demonstrating fixture usage
  • Test uses semantic locators (roles, labels, visible text) — no CSS selectors or XPath
  • Test follows Given-When-Then structure
  • No hard-coded waits or sleep() calls
Documentation & Code Quality
  • tests/README.md created with setup, running, architecture, best practices, CI, and troubleshooting
  • Test script added to project config file
  • No credentials or hardcoded secrets in any config file
  • No placeholder text left behind (TODO, FIXME, replace-me)

After the Setup — Next Steps
  • Copy .env.example to .env and fill in your environment values (BASE_URL, API_URL, credentials)
  • Run npm install (or language equivalent) to install dependencies
  • For Playwright: run npx playwright install to download browser binaries
  • Run the sample test: npm run test:e2e — it should pass on the first try
  • Then run bmad-testarch-atdd or bmad-testarch-automate to start generating real tests
