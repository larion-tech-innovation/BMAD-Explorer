The bmad-prd skill is the single entry point for all PRD work. It handles creation of a PRD from scratch through a structured discovery process, validates an existing PRD against a 7-dimension rubric, and applies targeted updates with blast-radius analysis. All three intents share the same agent (John), the same output folder, and the same three output files.

Create

  • Brain dump → artifact scan → calibration → draft
  • Parallel subagents scan all sources
  • Concern scan drives Adapt-In cluster selection
  • Fast path or coaching mode
  • Produces prd.md + addendum.md + .memlog.md
Validate
Validate

  • 2 reviewers by default (rubric walker + adversarial)
  • 7 rubric dimensions, graded Excellent to Poor
  • Verdict → proceed / fix / must update

  • Loops back into Update if issues found

prd-structure PRD Structure — Essential Spine + Adapt-In

Every PRD starts with the Essential Spine — nine sections that are always present. Additional sections are pulled in from the Adapt-In menu only when the product's concern profile demands them.

Essential Spine — Always Present

The 9 Essential Sections prd.md
SectionPurposeKey Detail
§0 Document Purpose Who the PRD is for, how it is structured, references to existing inputs Sets reader expectations; links to brief, PRFAQ, brainstorming artifacts
§1 Vision 2–3 paragraphs, 3–5 year direction Blue Ocean Canvas lens optional — Eliminate / Reduce / Raise / Create
§2 Target User §2.1 Jobs To Be Done · §2.2 Non-Users (v1) · §2.3 Key User Journeys Named-persona narratives; UJ-1 … UJ-N; JTBD framing (Christensen)
§3 Glossary Every domain noun defined once; no synonyms New v6.8: downstream workflows source-extract from PRD using Glossary terms as anchors. Auto-built by agent as drafting proceeds.
§4 Features FR-1 through FR-N with testable Consequences Behavioral narrative per feature group; each FR realizes one or more user journeys
§5 Non-Goals (Explicit) What the product will NOT do in v1 Out-of-scope is as load-bearing as in-scope; prevents scope creep
§6 MVP Scope In Scope (bulleted) + Out of Scope for MVP Each out-of-scope item carries reason + defer target: v2 / v3 / research
§7 Success Metrics Primary · Secondary · Counter-metrics New v6.8: Counter-metrics — what NOT to optimize; prevents gaming one metric at expense of another
§8 Open Questions Numbered questions that become future tickets Triaged during Finalize step 4; phase-blockers addressed first
§9 Assumptions Index Every [ASSUMPTION] tag surfaced and indexed All [ASSUMPTION] inline tags are collected here for stakeholder review

FR Format — Functional Requirements

Every functional requirement follows this exact structure. FR numbers are the backbone of the entire workflow: FR-03 in PRD → component in Architecture → Story references FR-03 → QA verifies FR-03.

#### FR-N: {Short capability name}
[Actor] can [capability] [under conditions]. Realizes UJ-X.

**Consequences (testable):**
- {Specific testable condition}
- {Another testable condition}

**Out of Scope:** (optional)
- {Explicit bound}
  • Actor is always a named user or system — never "the user" without persona context
  • Capability is a verb phrase: "assigns a confidence score", not "confidence scoring"
  • Consequences must be independently testable — each one is a potential QA scenario
  • Example: "we need a confidence score" → FR-11: System assigns Confidence Score (3 levels: High / Medium / Low)

§ WHY Why This Approach?

A PRD without numbered FRs, explicit non-goals, and a decision log is not a product requirements document — it is a wish list. Every structural choice in the Essential Spine exists to prevent a specific failure mode in downstream workflows.

Without Failure Mode With bmad-prd (v6.10.0)
Numbered FRs "Confidence score" in PRD, "trust rating" in architecture, "accuracy signal" in stories — same feature, three names, three implementations FR-11 in PRD → FR-11 in architecture component → FR-11 in story AC → QA verifies FR-11. One anchor, no ambiguity.
Explicit Non-Goals Scope creep — everything that was not explicitly excluded gets built eventually §5 Non-Goals + §6 MVP Out-of-Scope each with reason + defer target. A PRD that only lists in-scope enables unlimited creep.
Decision Log After 6 months, no one remembers why option B was rejected — debate restarts from zero .memlog.md captures every decision: # | Date | Decision | Rationale | Status. Audited during Finalize.
Counter-metrics (v6.8) Optimizing throughput crushes accuracy; optimizing activation crushes retention §7 Counter-metrics tell the architect what NOT to optimize — prevents gaming one metric at expense of another.
Glossary (v6.8) "Is a 'client' the same as an 'account'?" Dispute surfaces in UX, architecture, and QA independently §3 Glossary anchors every domain noun once. Downstream workflows source-extract using Glossary terms.
Testable Consequences per FR FRs are too vague to test — QA invents scenarios that do not match PM intent Each FR carries explicit, independently testable Consequences. QA scenarios derive directly from these.
External Validator PRD author will miss their own blind spots — self-validation is an antipattern Validate intent uses a DIFFERENT LLM from the author. 2 reviewers: rubric walker + adversarial reviewer.

skills 3 · Skills That Help This Session

Facilitator Mindset bmad-prd · session quality
SkillWhy it matters
Product thinking (what, not how) The PRD describes behavior, not implementation. "System processes uploaded payroll PDF via OCR" is a requirement. "Use AWS Textract" is an implementation choice — keep it out of FRs.
Decisive prioritization When the agent asks MoSCoW — be decisive. If everything is Must, nothing is Must. Aim for at least 30% of FRs in Should or lower.
Quantitative instinct "Fast" becomes "p95 < 200ms." "Many users" becomes "up to 10,000 concurrent." Every NFR must have a number — if you don't know it, ask engineering before the session.
Comfort saying "out of scope" One of the hardest PRD skills is explicitly excluding features. Unwritten out-of-scope items become future misunderstandings. Know which features are definitely NOT in this release before the session starts.

per-step 4 · Per-Phase Preparation

v6.8 PRD uses a Discovery phase followed by the Essential Spine (§0–§9). The categories: [Stakeholder] = information from clients or other teams; [Domain] = concepts to understand; [Session] = how to act during the AI interaction.

Phase-Level Readiness bmad-prd · create intent · v6.10.0 Discovery + Essential Spine
Phase / SectionWhat to have ready
Brain Dump [Stakeholder] Gather all source artifacts before the session — brainstorming output, PRFAQ distillate, prior briefs, transcripts. The agent auto-scans and loads them. More input → fewer clarifying questions mid-session.
Stakes Calibration [Domain] Know your project type before the session: Hobby/solo · Internal tool · Consumer launch · Regulated (healthcare, finance, government). Stakes determine project level (L0–L4) and depth of each section. When uncertain, bias upward — the agent can narrow scope, not retroactively add skipped content.
Working Mode [Session] Decide in advance — Fast Path if concept is clear and speed matters (agent auto-drafts with [ASSUMPTION] tags, then you review); Coaching Path if you need to work through it section by section collaboratively. You can switch modes at any point.
§1 Vision [Stakeholder] Know the 2–3 year direction before the session. Blue Ocean Canvas frame: what would you Eliminate / Reduce / Raise / Create relative to the market? A vague Vision produces vague FRs downstream.
§2 Target User [Stakeholder] Interview at least 1 real user per persona before this session: "Walk me through what you do on a typical [task]." Grounded personas anchor all FR decisions. Use real names — "Linh, bookkeeper processing payroll for 15 clients every Friday" beats "the accountant".
§4 Features [Stakeholder] Ask: "If we could only ship 5 features, which would you refuse to launch without?" — [Domain] Know atomic FR format: 1 FR = 1 user-visible behavior independently testable. Know user story format: "As a [persona] / I want to [action] / So that [outcome]."
§5 Non-Goals [Stakeholder] Ask: "What features are definitely NOT in this release?" / "What would make you reject this release?" — Explicitly naming out-of-scope items is as load-bearing as naming in-scope items. Do not skip this section.
§6 MVP Scope [Stakeholder] Ask: "If we had to ship in half the time, what would we cut?" / "Which features create dependencies that block others?" — [Domain] MVP = minimum to be usable and shippable. Phase 1 may be larger, but resist putting 80% of features in Phase 1.
§7 Success Metrics [Stakeholder] Ask: "What number would you look at 6 months after launch to decide if this was worth building? What is that number today?" — [Domain] Know primary metric vs counter-metrics (v6.8: what NOT to optimize, prevents gaming one metric at expense of another). Both required in every PRD.
Finalize · Validate [Session] Use a different LLM for Validate than the one that wrote the PRD — the authoring LLM has internalized its own assumptions and will miss its own blind spots. This is an explicit antipattern in v6.8, not optional guidance. Have the destination tool (Confluence, Notion, etc.) available for Finalize handoffs.

create Create Intent tạo mới

Begin with a structured brain dump — everything the user knows goes in one pass. The skill then runs an artifact scan, stakes calibration, and working mode selection before drafting the full PRD (Essential Spine + any triggered Adapt-In sections).

Discovery Phase — 4 Stages

Discovery Phase bmad-prd v6.10.0 · Create Intent
StageWhat happens
Source scan Auto-scan planning_artifacts/ for brainstorming, PRFAQ, brief, and any domain inputs. Surface paths only — user confirms which apply before content is read.
Brain dump User provides all context in one pass — requirements, constraints, transcripts, goals, competing priorities. "Anything else?" is mandatory before narrowing. Subagent web research spawns in parallel during this stage.
Artifact scan Parallel subagents read each confirmed source and extract decision signals relevant to the user's stated focus. Parent receives a relevance-filtered digest — not a raw doc dump. Missing inputs are flagged before drafting starts.
Stakes calibration One probe: Hobby/solo · Internal tool · Consumer launch · Regulated. Sets depth for every section — accessibility floor, compliance flags, NFR thresholds, validation rigor. Also classifies project level (L0–L4); determines which Adapt-In sections are triggered.
Working mode Fast Path — batch remaining gaps into 1–2 consolidated questions → draft full PRD with [ASSUMPTION] tags → user reviews and iterates. Coaching Path — walk through Essential Spine section by section; push back on thin assumptions in real time. Switchable at any point.

Finalize Process — 8 Steps

Runs at the end of every Create session and must be re-run after each Update that returns status to draft.

Finalize — 8 Steps bmad-prd v6.10.0
StepWhat happens
1 · Decision log auditWalk .memlog.md with user; confirm each entry is captured in PRD, addendum, or explicitly set aside. Missing entries flagged.
2 · Input reconciliationSubagent per source input checks PRD + addendum for gaps — especially qualitative ideas (tone, voice, feel) that the FR structure silently drops.
3 · Reviewer passRubric walker + adversarial reviewer run in parallel and render an HTML validation report. Findings tiered by severity (Critical/High surfaced first). Resolve before polish.
4 · Triage open itemsAll Open Questions, [ASSUMPTION] tags, [NOTE FOR PM] callouts. Phase-blockers resolved first; non-blockers deferred to .memlog.md.
5 · Polishbmad-editorial-review-structure then bmad-editorial-review-prose on prd.md + addendum.md. Structural passes run before prose — do not polish text that will soon be cut.
6 · External handoffsExecute configured handoffs (Confluence, Notion, Jira…); surface returned URLs/IDs. Skip and flag unavailable tools.
7 · CloseSet prd.md frontmatter status: final. Log finalization to .memlog.md. ⚠ If context was exhausted before Close, check stepsCompleted in frontmatter — start new session with explicit "Finalize this PRD — start from the Close step."
8 · on_complete hookRun if configured in customize.toml. Suggest common next steps: bmad-ux, bmad-architecture, bmad-create-epics-and-stories.

validate Validate Intent separate llm

Validation is only meaningful when performed by a different LLM than the one that wrote the PRD. Self-validation is an explicit antipattern in bmad-prd v6.8.

Antipattern Warning: Always use a DIFFERENT LLM than the one that authored the PRD. The authoring LLM has internalized its own assumptions and will not catch its own blind spots. This is not optional.

7 Rubric Dimensions

Validation Rubric bmad-validate-prd · rubric-walker
DimensionWhat it testsCommon failure
Decision-readiness Can a team make build/not-build decisions from this PRD alone? Vague FRs; no scope boundary; missing NFRs
Substance over theater Are FRs specific and testable, or are they marketing language? "Seamless experience", "powerful dashboard" — not testable
Strategic coherence Does Vision → User Journeys → Features form a coherent chain? Features that do not realize any user journey; vision misaligned with scope
Done-ness clarity Is it unambiguous when each FR is complete? FRs without testable Consequences; missing acceptance signals
Scope honesty Are non-goals and MVP exclusions explicit and reasoned? Only in-scope items listed; no rationale for deferred items
Downstream usability Can UX, Architecture, and Engineering work from this without a PM in the room? Missing Glossary; FR numbering inconsistent; ambiguous actor references
Shape fit Does the PRD's scope and depth match the product's stakes? Hobby-level depth for launch-level product; over-engineered for a spike task
Reviewer Configuration
  • Rubric walkerSystematic pass through all 7 dimensions; grades each Excellent / Good / Fair / Poor; cites specific sections
  • AdversarialPlays devil's advocate — finds assumptions, contradictions, and missing edge cases the rubric walker misses
  • DefaultBoth run by default. Additional reviewers can be added for regulated domains or high-stakes launches.

Verdict → Action Table

Good / Excellent

Proceed to downstream workflows. Document any minor findings as future improvement notes. No blocking action required.

Fair

Fix all High findings before downstream work begins. Medium findings may proceed with PM sign-off. Re-validate after fixing High items.

Poor

Must Update before any downstream workflow. Do not proceed to UX, Architecture, or Epics. Use bmad-prd → Update then Validate again.

Validation Flows
  • Flow A: Create → Validate → [issues found] → Update → Validate again
  • Flow B: [stakeholder feedback received] → Update → Validate

update Update Intent cập nhật

Every update starts with a blast radius analysis. No change is made without first mapping cascade effects across the document.

Blast Radius Analysis (Pre-edit)

  • Cascade checkIdentify all FRs, NFRs, Vision text, and decision log entries that touch the thing being changed
  • Conflict checkSource-extract the change from inputs; compare against existing text; surface conflicts before applying
  • ApplyMake changes with full awareness of what else must change to keep the document coherent
  • StatusAfter Update, PRD status returns to draft; Finalize must be re-run to set back to final

Deprecation Note

bmad-edit-prd is now a deprecated forwarding shim. All update work goes through bmad-prd → Update. The deprecated skill forwards to the unified skill but direct use is not supported and will not receive updates.

Blue Ocean Canvas
DimensionQuestion
EliminateWhat industry assumptions can be completely removed? What do we offer that customers don't actually value?
ReduceWhat can be below industry standard? What are we over-delivering on relative to customer need?
RaiseWhat should be raised well above industry standard? Where does the customer want more and gets less?
CreateWhat does no one in the industry currently offer? What new sources of value can we invent?
