Sprint Planning is the gateway to Phase 4. It scans every epic file, extracts the full inventory of epics and stories, cross-references that inventory against story files on disk, and produces a single YAML tracking file the entire Phase 4 toolchain depends on. Nothing in Phase 4 moves without this file.

§ WHY Why Sprint Planning?

Without a sprint-status.yaml, every Phase 4 workflow must independently re-parse epic files — with no guarantee their view of the current development state agrees.

Dimension Without Sprint Planning With Sprint Planning
Status tracking Manual notes or memory — each team member's view of what's done may differ Single YAML file; all Phase 4 agents read the same source of truth
Status discovery Developer must scan story folders by hand to know which stories have been created Automatic: filesystem scan detects existing story files and upgrades their status to at least ready-for-dev
Inventory accuracy Stories added to epics may never appear in tracking if someone forgets to update a list Every epic, story, and retrospective from every epic file is guaranteed an entry
Re-run safety Overwriting tracking manually risks resetting work that is already in progress Idempotent: running again only upgrades statuses, never downgrades completed or in-progress work
Downstream integration Each agent re-derives project state from scratch — slow, error-prone, inconsistent All Phase 4 skills (create-story, dev-story, sprint-status) read directly from the YAML without re-parsing epics
Sharded epics Multi-file epic structures are tedious to aggregate and compare manually Automatic index + section file assembly into one complete, ordered inventory

when-to-use When to Use This Skill

  • Phrase 1"run sprint planning" / "generate sprint plan"
  • Phase startFirst entry into Phase 4 — before any stories are created or developed
  • Epic updateAfter epics are modified to add new stories or epics
  • Re-syncAfter story files are created outside the normal create-story workflow
  • AuditWhen sprint-status.yaml is suspected to be out of sync with the filesystem

§ HOW How to Execute · 5-Step Workflow

Step 1 · Parse Epic Files

  • GoalBuild a complete inventory of every epic and story defined in planning_artifacts.
  • DiscoveryLook for a whole document first: epics.md, bmm-epics.md, or any *epic*.md file. If not found, look for the sharded format: epics/index.md + all section files listed in the index.
  • PriorityIf both whole and sharded versions exist, the whole document takes precedence.
  • ExtractionParse epic headers (## Epic 1:), story sub-headers (### Story 1.1: Title), and build a numbered inventory in source order.

Step 2 · Build Status Structure

  • GoalCreate a flat key→status map with all epics, stories, and retrospectives in the correct order.
  • Entry orderFor each epic: epic entry → story entries (in source order) → retrospective entry. Then next epic block.
  • Default statusesEpic: backlog · Story: backlog · Retrospective: optional
  • Key formatSee Story ID Conversion rules — all keys are deterministic, kebab-case derivations of the source header.

Step 3 · Apply Status Detection

  • GoalAutomatically upgrade statuses where the filesystem provides evidence of progress.
  • Story file checkFor each story key, check if {story_location}/{story-key}.md exists on disk.
  • If file foundUpgrade story status to at least ready-for-dev.
  • Preservation ruleIf a prior sprint-status.yaml already records a more advanced status (e.g. in-progress, done), preserve it verbatim. Never downgrade.
Never-Downgrade Rule The filesystem can only prove a story file exists — it cannot prove the story is still backlog if a developer has already started work. Any status more advanced than what file detection can determine is preserved from the prior sprint-status.yaml without modification.

Step 4 · Generate sprint-status.yaml

  • GoalWrite the complete tracking file to {implementation_artifacts}/sprint-status.yaml.
  • Dual metadataMetadata appears twice: once as YAML comment lines (for human readability) and once as key:value fields (for machine parsing by downstream agents).
  • OrderingAll items in order: epic → its stories → its retrospective → next epic block.
  • Self-documentationSTATUS DEFINITIONS and WORKFLOW NOTES embedded as comment blocks — the file is its own reference.

Step 5 · Validate & Report

  • GoalConfirm the generated file is complete, consistent, and valid before reporting to the user.
  • CompletenessEvery epic, story, and retrospective from the epic files appears in sprint-status.yaml. No extra entries that have no source in the epic files.
  • ValidityAll status values match defined state machine values. YAML syntax is valid and parseable.
  • ReportDelivers: file location · total epics · total stories · epics in-progress · stories completed · next steps for the user.

state-machine Status State Machine

Three separate state machines govern tracking. Transitions are strictly one-directional except for Retrospective. The color vocabulary below is used consistently across this document and the generated YAML.

Status Flow Diagrams SKILL.md · Status State Machine
Epic
backlog → in-progress → done
Story
backlog → ready-for-dev → in-progress → review → done
Retrospective
optional ⇄ done
Status Definitions SKILL.md · Additional Documentation
StatusApplies toMeaningSet by
backlog Epic, Story Not yet started. For stories: only exists as a header in the epic file — no story file on disk yet. Sprint Planning (default on first run)
ready-for-dev Story Story file created in {story_location}/. Ready to be picked up by a developer agent. Sprint Planning (auto-detect) or create-story
in-progress Epic, Story Actively being worked on. Epics transition here automatically when the first story file is created. Developer manually or dev-story
review Story Implementation complete. Awaiting code review. Dev runs the code-review workflow at this stage. Developer after finishing implementation
done Epic, Story, Retrospective Completed. For epics: all stories in the epic have reached done. Developer or reviewer
optional Retrospective Retrospective can be conducted but is not required for epic or project completion. Sprint Planning (default for all retrospectives)

story-id Story ID Conversion Rules

Every story header in the epic file is converted to a deterministic kebab-case key for sprint-status.yaml. The same key is used for the corresponding story file on disk.

Source ### Story 1.1: User Authentication
Period → Dash 1-1
Title → Kebab user-authentication
Final key 1-1-user-authentication
  • Epic entries use the key epic-{num} — e.g. epic-1, epic-2
  • Retrospective entries use epic-{num}-retrospective — e.g. epic-1-retrospective
  • Story file on disk must match the derived key: {story_location}/1-1-user-authentication.md
  • The conversion is applied consistently — no exceptions, no overrides

output Output: sprint-status.yaml single source of truth

One flat YAML file. Self-documenting via embedded comment blocks. Consumed directly by all Phase 4 workflows without transformation or re-parsing of epic files.

File Schema sprint-status-template.yaml
FieldTypePurpose
generated datetime Timestamp when the file was first created by Sprint Planning. Not updated on subsequent runs.
last_updated datetime Timestamp of the most recent Sprint Planning run. Updated on every run.
project string Project name from _bmad/bmm/config.yaml.
project_key string Short project identifier. Default: NOKEY when using file-system tracking.
tracking_system string Backend for status tracking. Default: file-system.
story_location path Directory where story files live. Used by downstream workflows to locate and check story files.
development_status map Flat key→status map. Covers every epic, story, and retrospective. Order: epic → stories → retrospective, per epic block.

validation Validation Checklist

Step 5 runs these checks before reporting completion. A failed check must be resolved before the summary is delivered to the user.

Complete Coverage
  • Every epic found in *epic*.md files has an entry in sprint-status.yaml
  • Every story found in *epic*.md files has an entry in sprint-status.yaml
  • Every epic has a corresponding epic-{num}-retrospective entry
  • No entries in sprint-status.yaml that cannot be traced to an epic or story in the epic files
Status Integrity
  • All status values are legal values from the state machine definitions
  • No status was downgraded from a value recorded in a prior run
  • Items appear in the correct order: epic → stories → retrospective per block
File Quality
  • Generated YAML parses without error
  • Metadata appears both as YAML comment lines and as key:value fields
  • last_updated reflects the current run's timestamp
  • File written to correct path: {implementation_artifacts}/sprint-status.yaml
Handoff Readiness
  • Summary report delivered to user: file location · epic count · story count · in-progress count · done count
  • Next steps communicated: review file → use to track progress → agents update statuses as they work → re-run to refresh auto-detected statuses
