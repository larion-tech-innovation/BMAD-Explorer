This skill belongs to Amelia, the dev agent — not the TEA (Test Architect Enterprise) module. Given a feature that is already implemented, it generates working API and E2E tests, runs them immediately, and fixes any failures on the spot before writing the summary.

Single Responsibility — Scope Boundary

This skill generates and verifies tests only. For code review: use bmad-code-review. For risk-based test planning, quality gates, or contract testing: use the TEA module (bmad-testarch-*).

vs. TEA Automate (bmad-testarch-automate)

Both generate test scripts, but serve fundamentally different use cases.

§ WHY Without vs With

Features get shipped. Tests don't get written unless something makes them easy to write and verify. This skill fills that gap — pragmatically, without ceremony.

Dimension Without bmad-qa-generate-e2e-tests With bmad-qa-generate-e2e-tests
Test coverage after feature Zero — tests get scheduled "later" and often never written. API + E2E tests generated in the same session the feature is finished.
Framework detection Dev must know what framework to use and how to set it up from scratch. Auto-detected from package.json and existing test files. No setup required.
Generated test quality Devs default to brittle CSS selectors and hardcoded waits that break on refactor. Semantic locators enforced (getByRole, getByLabel, getByText). No sleeps.
Broken tests after generation Common — generated code often has import errors, wrong paths, or stale assumptions. Run-and-fix loop in Step 4: all tests must pass before the skill finishes.
Test scope discipline Either too many tests (exhaustive, high maintenance) or too few (only happy path). Opinionated default: happy path + 1–2 critical error cases. Readable, maintainable.
Visibility / reporting No record of what was tested or what coverage exists. test-summary.md with file list, pass/fail status, API and UI coverage metrics.
Upgrade path When tests need to grow, dev doesn't know what to reach for next. Explicit "When to Upgrade to TEA" table guides teams to the right next tool.

§ HOW Running the Skill

Steps (0–5)

STEP 0 Detect Test Framework auto-detect
  • Scan package.json for test dependencies (playwright, jest, vitest, cypress, etc.)
  • Check existing test files to understand the project's patterns and conventions
  • Use whatever framework the project already has — never impose a new one without reason
  • If no framework exists: analyze project type (React, Vue, Node API…) → search for recommended framework → suggest and use it (or ask user to confirm)
STEP 1 Identify Features to Test
  • Ask user: specific feature/component name, a directory to scan, or auto-discover
  • Auto-discovery: scan source_dir for components, routes, and API endpoints
STEP 2 Generate API Tests if applicable
  • For each API endpoint or service, generate tests covering: status codes (200, 400, 404, 500), response structure validation
  • Scope: happy path + 1–2 critical error cases — not exhaustive
  • Follow the project's existing test patterns (fixtures, helpers, naming conventions)
STEP 3 Generate E2E Tests if UI exists
  • For UI features: test user workflows end-to-end (clicks, form fills, navigation)
  • Use semantic locators — roles, labels, visible text — never CSS selectors or XPath
  • Assert visible outcomes that a user would observe
  • Keep tests linear and simple — no complex fixture composition
  • Follow the project's existing test patterns
STEP 4 Run Tests & Fix Failures required — not optional
  • Execute all generated tests using the project's test command
  • If any test fails: fix immediately — read the failure output, patch the test, re-run
  • Repeat until all tests pass — the summary is only written after this condition is met
STEP 5 Create Summary
  • Write markdown summary listing all generated test files with pass/fail status
  • Include coverage metrics: API endpoints covered (N/total), UI features covered (N/total)
  • Note suggested next steps (CI integration, additional edge cases)
  • Save to {implementation_artifacts}/tests/test-summary.md
  • Validate against checklist.md before finishing

API Tests

API tests verify endpoint contracts — status codes and response shapes. Target: happy path + 1–2 error scenarios.

// Example — follows project's existing style test('GET /api/users/:id returns user', async ({ request }) => { const response = await request.get('/api/users/1'); expect(response.status()).toBe(200); const body = await response.json(); expect(body).toMatchObject({ id: 1, email: expect.any(String) }); }); test('GET /api/users/:id returns 404 for unknown id', async ({ request }) => { const response = await request.get('/api/users/99999'); expect(response.status()).toBe(404); });

E2E Tests

E2E tests simulate real user interactions. The key rule: use semantic locators that reflect how users perceive the UI — never CSS class selectors.

// Semantic locators — roles, labels, visible text test('user can log in successfully', async ({ page }) => { await page.goto('/login'); await page.getByLabel('Email').fill('user@example.com'); await page.getByLabel('Password').fill('password123'); await page.getByRole('button', { name: 'Sign In' }).click(); // Assert visible outcome the user would see await expect(page.getByText('Welcome back')).toBeVisible(); await expect(page).toHaveURL('/dashboard'); });

Run & Fix — What Makes This Skill Different

Step 4 is what separates this skill from pure generators. Every generated test must pass before the skill completes.

Rule: all tests green before finishing

The skill runs the project's test command, reads the failure output, patches the test, and re-runs. This loop continues until output is clean. The summary is only written after this condition is met.

This is intentionally different from TEA automate, which generates tests but does not run them — TEA expects CI to verify tests through priority-gated pipelines.

Keep It Simple

This skill is explicitly designed to avoid over-engineering. Tests should be easy to read and easy to maintain — not architecturally impressive.

Do
  • Use standard test framework APIs
  • Focus on happy path + critical errors
  • Write readable, maintainable tests
  • Use semantic locators (roles, labels, text)
  • Run tests to verify they pass
  • Follow existing project patterns
Avoid
  • Complex fixture composition
  • Over-engineering and unnecessary abstractions
  • CSS selectors or XPath locators
  • Hardcoded waits or sleeps
  • Order-dependent tests
  • Exhaustive edge-case coverage
