spine-structures Spine Structures — DESIGN.md vs EXPERIENCE.md

Two independent spines serve different audiences. DESIGN.md is anchored to the Google Labs design.md spec. EXPERIENCE.md is product-shaped and freely extensible.

Dimension DESIGN.md — Visual Identity EXPERIENCE.md — IA + Behavior
Spec anchor Google Labs design.md specification BMad EXPERIENCE.md convention
Frontmatter YAML with token fields: colors · typography · rounded · spacing · components Status, version, last-updated fields; no token YAML
Body sections (canonical order) Brand & Style · Colors · Typography · Layout & Spacing · Elevation & Depth · Shapes · Components · Do's and Don'ts Foundation · Information Architecture · Voice and Tone · Component Patterns · State Patterns · Interaction Primitives · Accessibility Floor · Key Flows
Conditional sections Sections are omittable; order is fixed when present Inspiration & Anti-patterns · Responsive & Platform — added when triggered. Invented sections freely added for product-specific concerns.
Token rule Color tokens MUST have hex (or light/dark pair) — missing hex is a critical issue References DESIGN.md tokens as {"{"}path.to.token{"}"} — never raw hex values
Key Flows format n/a Named protagonist (e.g. "Pary on iPad") — NOT "the user". Numbered steps. Climax beat. Failure path.
Conflict rule When spines conflict with mockups or wireframes — SPINES WIN. Write this in DESIGN.md frontmatter once at Finalize.

ui-inheritance UI System Inheritance

If Foundation in EXPERIENCE.md specifies a UI system — shadcn, MUI, native UIKit, Jetpack Compose, or similar — both spines inherit from it automatically:

  • DESIGN.mdOnly references and extends the system's defaults. Does not re-specify values the system already provides.
  • EXPERIENCE.mdOnly specifies behavioral delta from the system's defaults. Patterns the system handles natively are omitted.
  • Conflict handlingIf project overrides a system default, document the override explicitly in DESIGN.md with rationale.

§ WHY Why This Skill Is Structured This Way

Why two separate spines?

Visual identity (DESIGN.md) and behavioral specification (EXPERIENCE.md) serve different audiences and change at different rates. A designer iterates color tokens and type scale independently of IA restructuring. Keeping them separate prevents conflation — no architectural change should force a brand token audit, and no rebrand should require re-specifying state patterns.

Why spines win over mockups?

Mockups are explorations — they're made to show possibilities, not to commit to them. Spines are decisions — they record what was deliberately chosen and why. When they conflict, the decision wins. This prevents a visual prototype from silently overriding a deliberate accessibility or density choice that the team agreed to.

Why named protagonists in Key Flows?

"The user" is a fiction. "Pary on iPad at 10pm, kids asleep" forces specific form-factor, voice, and density decisions. The protagonist's device, context, and emotional state constrain the design space in ways that generic user stories cannot. Mirror source-spec names verbatim — do not rename personas introduced in the PRD or user research.

Why elicit, not impose?

Color choices affect brand identity for years. Font selections affect brand equity and sometimes licensing costs. Sally's job is to help the product owner make informed decisions — not to impose a style preference. If Sally autonomously chooses a palette, she removes the owner's agency and creates a spec the team never fully committed to. Every consequential choice is surfaced for owner confirmation.

per-step 4 · Per-Step Preparation

Discovery runs sequentially with gates between steps. Each step closes before the next opens. Prepare for the decision-heavy steps — automatic steps need no preparation.

Step-Level Readiness bmad-ux · 8-step discovery
StepWhat to have ready
Step 1 · Source Scan Know which planning artifacts apply to this project. Be ready to confirm or reject the file paths the agent surfaces — also be ready to add sources outside the standard folder (Figma links, brand decks, prior briefs).
Step 2 · Brain Dump Prepare answers to 3 questions: (1) existing design language — colors, type scale, spacing system; (2) target users and how they currently interact with the app; (3) key design challenges you already know about.
Step 3 · Stakes Read Decide in advance: hobby · internal tool · consumer product · regulated environment? Stakes calibrate accessibility enforcement, compliance notes, and how hard Sally pushes back on risky design decisions.
Step 4 · Working Mode Fast Path (batch gaps → draft both spines → iterate) or Coaching Path (walk decisions area by area, creative tools woven in). Third option: Design Handoff — assemble Discovery inputs into a prompt for an external design tool (Figma, Vercel v0, etc.) and import the result.
Step 6 · Journeys Prepare a named protagonist per persona — specific person, platform, context. "Pary on iPad, reviewing data before a 9am meeting" anchors form-factor and interaction density decisions. Mirror persona names from the PRD verbatim.
Step 7 · Form-factor Closure Know your primary surface before reaching this step. If journeys don't resolve form-factor naturally, be ready to answer explicitly — ambiguity here cascades into navigation patterns, spacing, and breakpoints.

validate-rubric Validate Intent — 8-Category Rubric

Used by both the Validate intent and the rubric walker in Finalize Step 1. No files are written when running as Validate intent — all findings are surfaced in conversation.

8-Category Rubric Validate Intent · Rubric Walker
#CategoryWhat is checked
1 Flow Coverage Each user journey / requirement has a Key Flow with: named protagonist, numbered steps, climax beat, failure path. No flow references "the user" generically.
2 Token Completeness All DESIGN.md tokens defined. Color tokens have hex (or light/dark pair) — no abstract color names without values. All {"{"}path.to.token{"}"} cross-references in EXPERIENCE.md resolve to a defined token.
3 Component Coverage Every component has: a row in DESIGN.md Components (visual specification) AND a corresponding entry in EXPERIENCE.md Component Patterns (behavioral specification). No half-covered components.
4 State Coverage Every IA surface covered for: empty state · cold load · focus state · error state · offline state · permission-denied state. Missing states flagged with severity.
5 Visual Reference Coverage Every file in mockups/ + wireframes/ + imports/ has an inline link in at least one spine. Orphaned assets flagged — artifacts without spine references will not survive a handoff.
6 Bloat & Overspecification Flags: pixel specs where tokens suffice · source material restatement without synthesis · prose where a table communicates more clearly · sections that repeat spine-level decisions without adding information.
7 Inheritance Discipline User journey names match verbatim from source PRD or research docs. Glossary terms are consistent across both spines. All token references resolve. UI system inheritance is honored — no re-specification of system defaults.
8 Shape Fit DESIGN.md follows canonical section order (Brand & Style → Colors → … → Do's and Don'ts). EXPERIENCE.md contains all required default sections; any dropped defaults are defensibly absent with a logged reason.

Technique Descriptions
Color Themes Tool
ElementDetail
Output Subagent produces .working/color-themes-{n}.html — 4–6 variations rendered side-by-side with a header (name + emotional register), semantic token chips for all roles, and a realistic UI snippet using real product content (no lorem ipsum).
Critical rule Variations MUST differ in emotional register — not just hue swaps. A calm navy palette is distinct from a confident coral palette. Light + dark modes shown side-by-side when both are in scope.
Technical constraint Inline CSS only. Concrete hex values in <style> comments. No external network calls — the file must be self-contained.
Design Directions Tool
ElementDetail
Output Subagent produces 3–6 .working/direction-{slug}.html files — each a complete visual personality applied to the same hero screen. 2–3 sentence rationale + near-1:1 hero screen in a device frame + at least one state variant (error, empty, or loading).
Directions differ on Density · Type weight · Motion implication · Brand register — not palette swaps. Each direction is a distinct personality, not a color rotation of the same layout.
Content rule Real product content in every mockup. No "Lorem ipsum." No placeholder copy. No generic form labels. The content must reflect the actual product being designed.
Excalidraw Wireframes Tool
TypeDetail
IA diagram Boxes-and-arrows showing: auth stack, main surfaces, modal routes, settings hierarchy. Maps navigation structure, not visual design.
Flow wireframe Screen-by-screen rectangles left-to-right at low fidelity; arrows labeled by user action causing the transition; annotations mark the climax beat and edge-case paths.
Critical index rule Every element index field MUST be exactly 2 characters (e.g., a0, aZ, b3). Three characters causes a silent "Error: invalid file" with no diagnostic output. The file fails to open with no error message — verify every element before writing.
Peak-End Rule — UX Design Principle (Kahneman)
PrincipleApplication
Core rule Users remember two moments: the peak emotion (most intense positive or negative) and the end emotion (how the session felt when it closed). The average experience fades — the moments are what remain.
Design implication Deliberately design 1 peak moment (the climax beat — where the product delivers its core value) and 1 strong end moment (clear success state, celebration, or confident next step). Map these explicitly in user journey narration.
Discovery Step 6 The protagonist journey narrative identifies the climax beat and failure path. The climax beat becomes the peak design target. The failure path becomes the error state design target.
