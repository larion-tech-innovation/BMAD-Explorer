§ WHAT Trigger Mapping — What does this Skill do?

Strategic discovery facilitator — connects business goals to user psychology and produces the Trigger Map that drives design prioritization.

Reads A-Product-Brief/project-brief.md for context; writes the Trigger Map that Freya reads in Phases 3 and 4.

This skill connects business goals to user psychology. The Trigger Map answers why users actually take actions — it goes deeper than user stories, mapping the emotional and rational driving forces behind behavior, then surfacing which features will have the most impact. It is the bridge between the strategic foundation set in the Project Brief and the scenario and UX work that follows: a thin map produces design built on guesses about motivation; a rich map gives every later phase a grounded reason for each decision.

Use it after the Project Brief is complete, before any scenario or UX work, on every WDS project. It turns the brief's business goals and target groups into a psychology-aware reference that downstream agents consume directly.

§ WHY Why Trigger Mapping? — The reasoning behind each design choice

why-psychology Why Map Psychology, Not Just Stories?

User stories describe what a user does; they rarely capture why. WDS treats the why as the load-bearing input to design, because a feature aimed at the wrong motivation can be perfectly built and still fail to move behavior. The Trigger Map — grounded in Effect Mapping by Mijo Balic & Ingrid Domingues — works backward from business goals through the target groups to the emotional and rational driving forces that actually trigger action. Mapping psychology, not just tasks, is what lets later phases choose features that change behavior rather than features that merely exist.

why-negative-forces Why Include Negative Driving Forces?

Classic Effect Mapping captures positive driving forces — the aspirations, motivations, and gains that pull a user toward action. WDS enhances the method by mapping negative driving forces as well: the fears, blockers, and frustrations that hold a user back. Behavior is the net of both. A product that only amplifies aspirations while ignoring what users are afraid of, or what frustrates them, designs for an idealized user who does not exist. Surfacing the negatives gives design something concrete to remove or reassure, which is often higher-leverage than adding another aspirational feature.

why-impact-matrix Why the Feature Impact Matrix Matters

The Feature Impact Matrix is critical for Phases 3 and 4. It links each candidate feature to the driving forces it affects, so impact is visible rather than assumed. Freya reads it directly to prioritize which scenarios and pages to design first — high-impact features earn early, detailed attention; low-impact ones are deferred or dropped. Without the matrix, prioritization defaults to opinion; with it, the design sequence is anchored to mapped user motivation.

why-mermaid Why Mermaid Diagrams Serve Double Duty

The map embeds five Mermaid diagram types (goals, platform, groups, forces, connections). They serve double duty: as visual communication they let stakeholders see the relationships between goals, groups, and forces at a glance; as machine-readable reference the same diagrams are parsed by other agents downstream without re-deriving the structure from prose. One artifact satisfies both the human review loop and the agent pipeline. Note that the alliterative persona names — "Harriet the Hairdresser" — are by design: alliteration makes archetypes memorable across long projects and many sessions.

§ STEPS Step-by-Step Walkthrough — What happens inside each step

step-00 From Existing Docs — Extraction Steps path 02 only

These six steps run only when existing documentation is provided. They synthesize artifacts into structured trigger map data before both paths converge on the shared workshop and generation steps. Saga frames all questions as validation ("your material suggests X — is this correct?") rather than cold extraction.

Step 00a · Documentation Synthesis

Saga receives and reads all provided artifacts (PRFAQ, PRD, research notes, brief), building a mental map of what is covered versus what is absent. The step produces a clear summary of documentation strengths and gaps before any extraction begins. Nothing is generated from this step — it is purely analytical.

Step 00b · Business Goals Extract

Extracts and validates the vision statement and SMART strategic objectives from the provided documentation. If a vision is vague or implied, Saga infers it and asks for confirmation; if it is missing, goals are built through collaborative dialogue. Vague metrics are transformed into SMART format before the step closes.

Step 00c · Target Groups Extract

Extracts target user groups and deepens them into behavioral personas. Demographic-only descriptions are insufficient — Saga probes for context, goals, frustrations, and current behavior to build psychological depth. The step narrows groups to a maximum of three or four and asks for confirmation before proceeding.

Step 00d · Driving Forces Extract

Maps both positive and negative driving forces for each confirmed persona. Documentation typically emphasizes positive wants — this step actively probes for negative drivers (fears, blockers, frustrations) because they are often more powerful due to loss aversion. Both sides must be confirmed for every persona before the step completes.

Step 00e · Prioritization Extract

Surfaces priority signals from the documentation (resource allocation, research depth, frequency of mention) and uses them to rank target groups and driving forces with an impact-times-feasibility assessment. Every ranking must have documented rationale. The step concludes with a design focus statement synthesizing the top group and top drivers.

Step 00f · Gap Analysis

Compares the synthesized trigger map data against what downstream phases need and identifies what was strong, weak, or missing in the original documentation. Strategic alignment is validated — vision versus plan versus groups versus forces. Critical gaps are addressed through focused mini-workshops or flagged for future research before proceeding to the shared overview step.

step-01 Overview & Mode Selection both paths

Both paths converge here. Saga presents the Phase 2 scope and offers three engagement modes before proceeding to workshops.

Step 01 · Overview

Saga presents the Phase 2 scope and offers three engagement modes: Workshop (facilitated, 45–60 min), Suggest (AI-generated with per-step review, 20–35 min), or Dream (fully autonomous with final review, 15–25 min). Suggest and Dream modes initialize a layered research approach — reading WDS form guides, extracting project context, running domain web searches, generating, and self-reviewing each step before presenting. The mode must be explicitly chosen before the step closes.

step-02–05 Workshops 1–4 both paths · workshop content

Four workshops build the core trigger map data. The From Existing Docs path enters these with pre-validated data already in place; the From Scratch path builds everything live.

Step 02 · Business Goals Workshop

Workshop 1 captures the vision statement and transforms it into three to five SMART objectives. Saga starts with aspirational questions ("think big — if everything goes perfectly, what position do you want to hold?") then systematically sharpens each metric for specificity, measurability, achievability, relevance, and time-bound targets. Vague goals are not accepted without being made measurable.

Step 03 · Target Groups Workshop

Workshop 2 identifies the user groups whose behavior will drive the business objectives, then narrows to two to four focus groups with clear rationale. Each group is developed into a narrative persona — not a bullet list — with a name, a day-in-their-life description, goals, frustrations, and current workarounds. Personas must feel like real people before the workshop closes.

Step 04 · Driving Forces Workshop

Workshop 3 maps three to five positive drivers and three to five negative drivers for every persona. Saga explains the loss aversion principle — that negative drivers often have more design leverage than positive ones — and systematically explores both "toward" motivation (aspirations, desired outcomes, status goals) and "away-from" motivation (fears, frustrations, risks to avoid) for each persona. Cross-group patterns are identified after all personas are mapped.

Step 05 · Prioritization Workshop

Workshop 4 forces ranked decisions on business goals, objectives, target groups, and driving forces — and challenges every choice with "why is X more important than Y?" to prevent gut-feel prioritization. The workshop closes with a MoSCoW-style design focus statement: primary group, must-address drivers, should-address drivers, and could-address drivers if time permits.

step-06 Feature Impact Assessment optional but recommended

Five steps produce the Feature Impact Analysis document. This group is optional but strongly recommended — it is the artifact Freya reads to decide which scenarios and pages to design first.

Step 06a · Extract Features

Saga reads the project brief silently and extracts all strategically relevant features, skipping basic authentication, standard profiles, and routine CRUD unless they are uniquely strategic for this project. The list is presented as a numbered table for easy review, and the user can add, rename, combine, or remove features before confirmation.

Step 06b · Confirm Assessment Approach

Before running the assessment, Saga explains the scoring methodology: High/Medium/Low impact per persona, with the primary persona weighted more heavily (High = 5 pts, Medium = 3 pts, Low = 1 pt; other personas High = 3 pts, Medium = 1 pt, Low = 0 pts). The user must explicitly confirm readiness before the assessment begins.

Step 06c · Make Assessment

Saga assesses every feature against every persona's positive and negative driving forces, scoring and ranking the complete list. High-impact features — those directly addressing a major want or directly mitigating a major fear — are highlighted with strategic rationale. The user reviews the ranked table and may adjust any score before the assessment is confirmed.

Step 06d · Generate Feature Impact Document

Generates the complete B-Trigger-Map/feature-impact-analysis.md document, including a ranked features table, per-feature rationale, Must Have / Consider / Defer prioritization tiers, strategic implications, and questions for the designer. A summary is presented for final confirmation before the file is saved.

Step 06e · Feature Workshop Wrap-Up

Presents a complete summary of all deliverables produced during the feature impact work and explains how the Feature Impact Analysis guides Phase 4 UX Design — which scenarios to design first, where to invest detailed attention, and which personas matter most for each feature. Confirms readiness to proceed to document generation.

step-07 Hub Document & Persona Files document generation

Seven steps generate the complete trigger map document set. Each document follows a strict template and must be saved before the next step begins.

Step 07a · Generate Hub Document

Creates 00-trigger-map.md — the main entry point — with the Mermaid diagram, on-page summaries for each section (visible without clicking through), navigation links to all sub-documents, the key transformation statement, the flywheel, and a "How to Read" section explaining the diagram's left-to-right flow. All data is extracted character-for-character from workshop outputs; no paraphrasing is permitted. Target length: approximately 220–250 lines.

Step 07b · Generate Business Goals Document

Creates 01-Business-Goals.md with the vision statement, three priority tiers of SMART objectives, the flywheel explanation showing how goals connect causally, and success metrics alignment linking each persona to each objective. Language must be empowering and organic ("create awesome users") — conversion-oriented language is explicitly forbidden. Target length: approximately 150–160 lines.

Step 07c · Generate Primary Persona

Creates the primary persona document (02-[Name]-the-[Role].md) with all thirteen required template sections, including a full before/after emotional transformation journey and a champion creation story. Every one of the six driving forces (three wants, three fears) must include a Product Promise showing specifically how the product addresses that force. Target length: approximately 250–375 lines.

Step 07d · Generate Secondary Persona

Creates the secondary persona document (03-[Name]-the-[Role].md) with the same thirteen-section structure, but with a validation strategy focus — what this persona needs to see to trust the product, and the evaluation journey from discovery through adoption to advocacy. Every driving force includes a Product Answer rather than a Product Promise. Routes to the tertiary step if a third persona exists.

Step 07e · Generate Tertiary Persona

Creates the optional tertiary persona document (04-[Name]-the-[Role].md) with an organic value discovery focus — this persona finds and appreciates the product naturally, without being "targeted." The section unique to this persona shows the recognition path from experience through appreciation to word-of-mouth advocacy. Step is skipped if only two personas were defined.

Step 07f · Generate Key Insights

Creates 05-Key-Insights.md synthesizing all trigger map data into nine sections of actionable design direction: the flywheel, primary development focus areas, critical success factors, design implications per page or experience section, emotional transformation goals written in first person, the design focus statement, and development phase recommendations. All insights must derive from actual workshop data — generic advice is rejected.

Step 07g · Quality Check

Runs a thirteen-dimension quality verification across all generated documents — file structure, Mermaid diagram rendering, content consistency, language quality, cross-references, persona completeness, hub length, business goals length, key insights length, feature impact scoring, priority tier consistency, driving force Product Promises, and markdown formatting. Any issues found are corrected and re-verified before proceeding to Mermaid diagram generation.

step-08 Mermaid Diagram Generation optional · visual layer

Eight steps assemble the Mermaid relationship diagram embedded in the hub document. This group is skippable if the token budget is tight — the textual map is complete without it.

Step 08a · Initialize Diagram Structure

Sets up the Mermaid configuration with the base theme, Inter font at 14px, left-to-right flowchart direction, and section comments. Determines all node IDs from actual data: business goals as BG0/BG1/BG2, the central platform as PLATFORM, target groups as TG0/TG1/TG2, and driving forces as DF0/DF1/DF2.

Step 08b · Format Business Goals Nodes

Creates each business goal node with top padding, an emoji plus ALL CAPS title, three to five key points, and bottom padding. No HTML formatting tags are permitted inside nodes. Priority is indicated by vertical order (BG0 appears topmost) rather than special formatting — all business goal nodes share consistent light gray styling.

Step 08c · Format Platform Node

Creates the central PLATFORM node with the product name in ALL CAPS, a category or tagline, and a multi-line transformation statement describing the before/after emotional change. The statement must span three to five lines for readability. The platform node uses a slightly darker gray fill and a thicker 3px border to distinguish it visually from all other nodes.

Step 08d · Format Target Group Nodes

Creates persona nodes with emoji, ALL CAPS name, priority level (PRIMARY TARGET / SECONDARY TARGET / TERTIARY TARGET), and three to four concise profile traits. The emoji chosen for each TG node is recorded — it must be used identically in the corresponding DF node for visual matching across the diagram.

Step 08e · Format Driving Forces Nodes

Creates driving forces nodes using the same emoji as the corresponding TG node, with WANTS and FEARS headers in plain ALL CAPS (no emoji on headers), exactly three checkmark-prefixed positive drivers, and exactly three X-prefixed negative drivers. Driver text is kept under 40 characters per line. The emoji matching is verified before the step completes.

Step 08f · Create Connections

Draws all arrows using simple --> syntax: all BG nodes to PLATFORM, PLATFORM to all TG nodes, and each TG node to its matching DF node (TG0 to DF0, TG1 to DF1, etc.). The total connection count is verified — for three personas this is nine connections. No fancy arrow styling is used.

Step 08g · Apply Styling

Applies four exact CSS class definitions to all nodes: businessGoal (light gray #f3f4f6), platform (medium gray #e5e7eb with 3px border), targetGroup (near-white #f9fafb), and drivingForces (light gray #f3f4f6). Colors are exact specifications — no variations are permitted. All nodes are assigned to their correct class.

Step 08h · Mermaid Quality Check

Verifies the complete assembled diagram across seven dimensions: configuration (font, fontSize, direction), node formatting (padding, ALL CAPS, no HTML tags), emoji usage (TG/DF matching, WANTS/FEARS headers plain), driving forces (exactly three per category), connections (all paths present, no broken links), styling (exact color codes, correct border widths), and syntax (no errors, all brackets closed). Issues are fixed and re-verified before proceeding to finalization.

step-09 Finalize & Handover both paths · closing sequence

Six steps close the Phase 2 workflow with cross-referencing, a final quality gate, a handover package for Freya, and a design log update.

Step 09a · Finalize Hub

Orchestrates the complete document generation sequence, verifying that all mandatory files exist in B-Trigger-Map/: the hub with a Mermaid code block, the business goals document, one persona file per target group, and the key insights summary. The feature impact document is also verified if the feature workshop was run. Missing files are generated before proceeding.

Step 09b · Add Cross-References

Adds bidirectional navigation links to every document — each sub-document links back to the hub, the hub links forward to every sub-document, and persona documents link to one another. No document is left isolated. Cross-references are verified to be bidirectional before the step closes.

Step 09c · Quality Check

Runs the final end-to-end quality verification: all documents exist and are non-empty, the Mermaid diagram renders correctly, cross-references are working, terminology is consistent across all files, all personas have complete driving forces, and the feature impact document is present if the workshop was completed. Found issues are fixed before marking the check as passed.

Step 09d · Create Handover Package

Presents a structured handover summary for Freya (the UX Designer): the complete file tree, the primary persona's transformation summary, the top three must-address positive drivers, the top three must-avoid negative drivers, feature priorities from the impact matrix if available, and three key design implications. This is a presentation step — no new files are created.

Step 09e · Update Design Log

Reads the existing _progress/00-design-log.md and appends a Phase 2 completion entry listing every artifact file individually (no "etc." shorthand), a specific two-to-three sentence summary of personas identified and key strategic insights, and any significant decisions made during workshops. Existing entries are never overwritten.

Step 09f · Provide Activation

The final step of Phase 2. Provides activation instructions for starting the UX Design phase with Freya in a new conversation, explains the continuous delivery approach (completed scenarios can be handed to development while more are being designed), and surfaces alternative paths if the feature impact workshop was skipped. Only the Return to Menu option is available — this is the end of the Phase 2 workflow.
