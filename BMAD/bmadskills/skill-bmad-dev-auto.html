§ WHAT Dev Auto · One Iteration, No Human in the Loop

Skill code
bmad-dev-auto
Purpose
Runs one full iteration of an unattended development loop — clarify and route, plan, implement, review — turning a task or change intent into a hardened, reviewable artifact with zero human back-and-forth.
Input types
Free-form task/change intent · existing spec file path with recognized status frontmatter (resumes at the matching step) · epic story reference
Step architecture
Just-in-time step loading, same convention as Quick Dev — each step is a self-contained file, sequential, no skipping. State tracked via spec frontmatter.
Output
A spec file in implementation_artifacts/ tracking the run, plus bmad-dev-auto-result-{slug-or-timestamp}.md if no spec file exists to write the result into.
Invocation
By name only. Not registered in bmad-help.csv/module-help.csv — no menu row, no chained routing yet.

Dev Auto is Quick Dev's unattended sibling. Where Quick Dev pauses for a human checkpoint before implementation and can hand off findings for human judgment, Dev Auto never stops to ask — every fork in the road resolves to either a deterministic action or a HALT with an explicit blocked status. This makes it suitable for loop-driven or scheduled runs where no one is watching the terminal.

halt-protocol HALT Protocol

The only sanctioned way to end a turn
Known spec file
If {spec_file} is known and exists, update its status in frontmatter and append the missing result details under ## Auto Run Result.
No spec file yet
Create {implementation_artifacts}/bmad-dev-auto-result-<slug-or-timestamp>.md with frontmatter status: <final status> and a body stating status + blocking condition, if any.
Resolve on_complete
Run resolve_customization.py --skill {skill-root} --key workflow.on_complete. If non-empty, follow it as the final instruction before exiting.
Stop
The workflow ends here. No further steps run in this turn.

Every blocking condition in the workflow — unclear intent, a dirty working tree, an intent gap, a spec that fails Ready for Development, an implementation that doesn't verify, a review loop past 5 iterations — routes to this same protocol with status blocked and a specific blocking condition string. Nothing is left ambiguous for whoever reads the result later.

Subagents are mandatory when instructed, and always synchronous. Every subagent call — including the two parallel reviewers in Step 4 — is invoked, awaited in the same turn, and only then does the workflow continue. Never backgrounded or detached: because this workflow runs unattended, there is no event loop left to resume a yielded turn, so a backgrounded subagent would stall the run permanently. If subagents are unavailable at all, HALT immediately with status blocked and blocking condition no subagents.

§ WHY Why an Unattended Loop? · No One Is Watching the Terminal

Quick Dev assumes a human is present to approve a plan and to make judgment calls on ambiguous findings. Dev Auto assumes the opposite — it may run in a queue, a scheduled job, or a loop where no one will answer a prompt for hours.

Every decision point that Quick Dev would surface to a human — approve the plan, judge an ambiguous review finding, decide whether to split a multi-goal request — becomes, in Dev Auto, either a deterministic rule (patch findings auto-fix, defer findings queue silently, reject findings drop silently) or an explicit HALT with a blocking condition a human can act on later. The workflow never guesses at an open question and never waits indefinitely for input that isn't coming.

This is also why the review layer is deliberately smaller than Quick Dev's: two reviewers (Blind Hunter, Edge Case Hunter) instead of three. Quick Dev's third reviewer, Acceptance Auditor, checks acceptance criteria against the spec — a judgment call that benefits from a human able to weigh in on genuinely ambiguous AC violations. Dev Auto keeps the two reviewers whose findings route to deterministic actions without needing that judgment.

§ HOW The 4 Steps · Clarify → Plan → Implement → Review

Step 01
Clarify & Route
Resolve intent from the invocation prompt. Epic-story detection loads/compiles epic context and prior-story continuity. Version-control sanity check. Derive spec slug, set spec_file.
No checkpoint — HALT on unclear intent or dirty tree
Step 02
Plan
Investigate codebase (subagents, distilled summaries only). Fill spec-template.md. Self-review vs. Ready for Development. Multi-goal / oversized warnings recorded, never blocking.
No checkpoint — HALT on unresolved intent gaps or a failed RFD gate
Step 03
Implement
Capture baseline_revision. Set spec in-progress. Hand spec to a synchronous implementation subagent. Verify every task and acceptance criterion after it returns.
No checkpoint — HALT if verification fails
Step 04
Review
Construct diff since baseline. Launch Blind Hunter + Edge Case Hunter in parallel, no context. Classify and cascade findings. Finalize: Auto Run Result, commit, HALT done.
Loopback gate to Step 3 (bad_spec) — no human checkpoint

finding-cascade Finding Classification Cascade Step 4 · review

Findings from the two reviewers are deduplicated and routed in this order. The first two tiers trigger a loopback; lower tiers only matter when neither fires. Severity assigned by a reviewing subagent is disregarded — this workflow reassigns severity itself, since reviewers operate under deliberate information asymmetry (no spec, no prior context).

spec-status Spec Status Machine

The spec file's frontmatter status determines which step a resumed run enters. A recognized status on the invocation's target spec is what lets Step 1 skip straight past clarify-and-route.
