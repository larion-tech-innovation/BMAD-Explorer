Sprint Status · Visibility, Risks & Next Action

Reads sprint-status.yaml — the single source of truth for Phase 4 — and produces a clear sprint snapshot: story counts by status, epic and retrospective progress, open action_items carried forward from past retrospectives, detected risks, and a deterministic next-action recommendation. Operates in three modes: interactive (human-facing dashboard), data (machine-readable key/value for other skills), and validate (structural file check).

No time estimates
The skill deliberately avoids estimates. Sprint status is about what is happening, not when it will finish. Risks are surfaced so the team can respond, not so they can predict a date.

Sprint Status is the dashboard read. It doesn't change anything — it reads sprint-status.yaml, validates it, detects risks, computes the next recommended action using a deterministic priority ladder, and presents the results. It also functions as a machine interface: other skills call it in data mode to get structured counts and the next workflow recommendation without displaying anything to the user.

§ WHY Why Structured Visibility? · Status Without Guesswork

When sprint state is implicit — in the developer's head, in commit messages, in chat threads — the next right action requires reconstruction every time. Sprint Status makes that reconstruction instant and deterministic.

The most common sprint overhead isn't implementation — it's orientation. "What should I work on now?" "Is there something waiting on me?" "Are we still on track?" Without a structured status view, each of these questions requires the developer to reconstruct sprint state from git history, conversation context, and memory. That reconstruction takes minutes and produces answers of varying reliability.

Sprint Status reads a single file and answers all three questions in under a second, with the same priority logic every time. The recommendation is never ambiguous — it follows a deterministic 6-level priority ladder. The risks are detected mechanically from the file state — no human needs to remember to check them.

single-source sprint-status.yaml as the Single Source of Truth

sprint-status.yaml is the flat tracking file that the entire Phase 4 workflow writes to and reads from. Every status-changing skill — Sprint Planning, Create Story, Dev Story, Code Review, Retrospective, Correct Course — updates this file when stories and epics transition. Sprint Status is the only skill that reads it without writing.

  • Keys — epicsEntries starting with epic- (and not ending with -retrospective). Statuses: backlog / in-progress / done.
  • Keys — storiesEverything else, e.g. 1-2-login-form. Statuses: backlog / ready-for-dev / in-progress / review / done.
  • Keys — retrosEntries ending with -retrospective. Statuses: optional / done.
  • Keys — action_itemsA dedicated action_items section, appended to by bmad-retrospective at the close of each epic. Sprint Status validates the section and surfaces any still-open items; Sprint Planning preserves them on regenerate rather than dropping them.
  • Stale warningIf last_updated is more than 7 days old (or missing, falling back to generated), the skill warns that the file may be stale — a signal to run the relevant skill to sync state.

§ HOW Next Action Priority Ladder · Deterministic Recommendation

The recommendation is computed once, the same way, every time. Six conditions are evaluated in priority order — the first that matches becomes the recommendation. Stories are sorted by epic number then story number (1-1 before 1-2 before 2-1) when selecting "the first" matching story.

Next Action Priority Ladder Evaluated top → bottom. First match wins. Sorting: epic num, then story num.
P 1
1
Condition · Highest priority
Story is in-progress
At least one story has status in-progress. Someone is actively working. Don't interrupt or start something new — recommend continuing the first active story.
Recommend bmad-dev-story first in-progress story
P 2
2
Condition
Story is in review
At least one story has status review. Implementation is done but code review hasn't run. This is a blocking state — unreviewed stories prevent the epic from progressing.
Recommend bmad-code-review first review story
P 3
3
Condition
Story is ready for dev
At least one story has status ready-for-dev. A spec has been written and approved. Time to start implementation — pick up the first ready story.
Recommend bmad-dev-story first ready-for-dev story
P 4
4
Condition
Story is in backlog
At least one story has status backlog. No stories are ready to implement yet. The next step is to draft a story — write the spec and get user approval before development begins.
Recommend bmad-create-story next backlog story
P 5
5
Condition
Retrospective is pending
All implementation items are done, but at least one retrospective has status optional. Epic is complete — run the retrospective to capture learnings before moving to the next epic.
Recommend bmad-retrospective pending retro
P 6
★
Condition · Terminal
All implementation items done
Every story is done, every epic is done, every retrospective is done or not applicable. The sprint is complete. Nothing left to recommend — congratulate the team.
Outcome Sprint complete! Congratulate the user

risk-detection Risk Detection

Six risk conditions are checked mechanically while parsing the file. Detected risks are appended to the dashboard output. They don't block the recommendation — they inform the user of issues that need attention alongside the next action.

Severity Trigger condition Risk message / recommended action
warn Story has status review Suggests running bmad-code-review — a review-blocked story is unresolved work that prevents epic progress
warn Story is in-progress and no stories are ready-for-dev Recommend staying focused on the active story — no queue to pull from, multi-tasking would create context overhead
warn All epics are backlog and no stories are ready-for-dev Prompt bmad-create-story — nothing is ready to implement, planning is the blocking item
warn last_updated is more than 7 days old (or missing) "sprint-status.yaml may be stale" — the file hasn't been updated recently; status counts may not reflect current reality
warn One or more entries in action_items are still open from a prior epic's retrospective Lists the open items with owner — a signal to close them out before the next epic starts, not just log new ones
error Story key's epic number doesn't match any existing epic (e.g., story 5-1-… but no epic-5) "Orphaned story detected" — a story without a parent epic suggests a file corruption or a manual edit gone wrong
error Epic is in-progress but has no associated stories in the file "In-progress epic has no stories" — an active epic with no stories suggests missing story entries or a planning gap

interactive-steps Interactive Mode — Step by Step

  • Step 0 · ModeDetermine execution mode from caller. Default: interactive. Jump to Step 20 for data mode, Step 30 for validate mode.
  • Step 1 · Locate fileTry sprint-status.yaml. If missing: exit with prompt to run sprint-planning. If found: continue.
  • Step 2 · Parse & validateRead full file. Classify entries as epics / retrospectives / stories. Count statuses. Map legacy values. Validate all statuses — prompt for corrections if any are unrecognized. Detect all 6 risks.
  • Step 3 · Select recommendationEvaluate priority ladder (P1–P6). Store next_story_id, next_workflow_id, next_agent.
  • Step 4 · DisplayOutput: project + tracking info, story counts by status, epic counts, open action_items carried forward from prior retrospectives, next recommendation command, and any detected risks.
  • Step 5 · MenuPresent 4 choices and wait for user input. See menu options below.

menu-choices Interactive Menu — 4 Options

[1]
Run Recommendation
Outputs the exact command to run the recommended workflow, with the story_key pre-filled if the workflow targets a specific story.
[2]
Stories by Status
Shows all story keys grouped: In Progress, Review, Ready for Dev, Backlog, Done. Useful for reviewing the full picture before deciding on next steps.
[3]
Raw YAML
Displays the full contents of sprint-status.yaml. For when the parsed view isn't enough and the user needs to see the exact file state.
[4]
Exit
Exits cleanly. Runs workflow.on_complete if configured (e.g., posting to Slack, syncing to a tracker). Then terminates.

validate-output Validate Mode — Structured Output

Returns machine-readable key/value results. Called by other skills before relying on sprint-status.yaml data. Three outcomes:

# Outcome A — file is valid
is_valid = true
message = "sprint-status.yaml valid: metadata complete, all statuses recognized"

# Outcome B — file missing
is_valid = false
error = "sprint-status.yaml missing"
suggestion = "Run sprint-planning to create it"

# Outcome C — invalid status values
is_valid = false
error = "Invalid status values: [key: 'value', ...]"
suggestion = "Fix invalid statuses in sprint-status.yaml"

Required metadata fields checked: generated · project · project_key · tracking_system · story_location. last_updated is optional for backward compatibility.
