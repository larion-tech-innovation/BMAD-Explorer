This skill improves existing products through targeted, incremental changes. It is the full WDS pipeline — Analyze, Scope, Design, Implement, Test, Deploy — compressed into a tight loop and applied to brownfield products: one view, one scenario, one improvement at a time. Each cycle reuses the discipline of the larger pipeline at a deliberately smaller scope, and every change is isolated on its own feature branch before it reaches main.

Use it when you have an existing product and want to add or improve a specific feature; when you do not want to redo the full Phase 1–7 pipeline; and when you follow a kaizen (continuous improvement) development model where small, data-driven improvements compound over time.

§ WHY Why Product Evolution? — The reasoning behind each design choice

why-miniature-pipeline Why a Miniature Pipeline Instead of Phases 1–7?

Brownfield products do not need a from-scratch restart. The strategy, users, and visual direction already exist in a shipped product — re-walking Phases 1 through 7 would re-derive context that is already settled. Phase 8 keeps the discipline of the full pipeline — Analyze, Scope, Design, Implement, Test, Deploy — but compresses it into a tight loop sized to a single improvement. You get the rigor without the overhead of a full greenfield pass.

why-focused-scope Why Focused Scope — One Improvement Per Cycle?

Limiting each cycle to one view, one scenario, and one improvement keeps changes reviewable and low-risk. A small, well-bounded change is easy to reason about, easy to test, and easy to roll back if it misbehaves. Bundling several improvements into one cycle blurs cause and effect and makes the review — and any later debugging — far harder than it needs to be.

why-branch-based Why Branch-Based?

Every change is isolated on its own feature/[improvement-name] branch, reviewed through a PR, and only then merged into main. Isolation means an in-progress improvement never destabilizes the shipped product, the review has a clean, self-contained diff to evaluate, and the merge into main is a deliberate, reviewed step rather than an accident of working directly on the trunk.

why-kaizen Why Kaizen?

Kaizen is the practice of small, data-driven, incremental improvements that compound over time. Rather than betting on one large redesign, Phase 8 makes steady, measured changes — each one grounded in what the existing product reveals. Note that Phase 8 can feed back into Phases 2, 3, and 4: if an evolution surfaces a new user type or driving force, update the Trigger Map so the strategic foundation keeps pace with the product.

§ HOW The Kaizen Pipeline — six sub-workflows, one tight loop

Phase 8 is a miniature WDS pipeline for brownfield improvements. Each improvement cycles through six sub-workflows in order — Scope, Analyze, Design, Implement, Test, Deploy — with every change isolated on its own branch. The pipeline is intentionally small: one view, one scenario, one improvement per pass.

workflow-scope Scope Improvement 5 steps

Turns a vague improvement idea into a concrete, bounded scenario — one page, one flow section, a clear before/after, and measurable success criteria. Scoping happens first so that Design and Implement never chase a moving target. Borrows discipline from Phase 3 (Scenarios).

Step 1 · Load Analysis

Read the product snapshot, selected improvement target, and rationale from the Analyze activity. This is the input that keeps Scope grounded in evidence rather than assumption.

Step 2 · Define the Change

Pin down exactly which view or page changes, describe the current state and what is wrong with it, articulate the desired state the user should experience after the change, and state success criteria — measurable wherever possible.

Step 3 · Map the User Journey

For the selected view, trace the micro-journey: entry point, current step-by-step flow, the exact pain points where the experience breaks down, and the proposed flow after the change.

Step 4 · Estimate Scope

Assess the blast radius: which pages and UI components are affected, whether any API or data-model changes are needed, and the risk level — Low (visual only), Medium (behavior change), or High (structural change).

Step 5 · Write Scenario

Produce a scenario document in {output_folder}/evolution/scenarios/ covering target, current state, desired state, user journey, success criteria, and scope. Present for user approval before moving forward.

workflow-analyze Analyze Product 4 steps

Reads the existing product — its pages, navigation, tech stack, and user flows — and surfaces a prioritized list of improvement opportunities. The output is a chosen target with rationale, not a solution. Borrows discipline from Phase 3 (Scenarios).

Step 1 · Load Product Context

Gather everything available about the existing product: project configuration, WDS artifacts, live URL if accessible, codebase for tech-stack and component patterns. Produce a Product Snapshot — a factual current-state summary — before forming any opinions.

Step 2 · Identify Improvement Targets

With the user, survey five lenses: user feedback and pain points, business metrics that need movement, technical debt, visual inconsistencies, and competitor gaps. Produce a prioritized list of improvement targets with impact and effort estimates.

Step 3 · Select Target

From the prioritized list, select one target for this cycle. Kaizen discipline requires a single focus — bundling targets blurs cause and effect and expands risk. Present the choice to the user for confirmation.

Step 4 · Document Analysis

Save the product snapshot, the prioritized target list, and the selected target with rationale to {output_folder}/evolution/analysis/. This document feeds directly into the Scope workflow that follows.

workflow-design Design Solution 5 steps

Designs the smallest effective change that solves the root cause — from a quick sketch or before/after description all the way to a development-ready specification. Scope creep is actively challenged at every step. Borrows discipline from Phase 4 (UX Design).

Step 1 · Load Scenario

Read the scenario from the Scope workflow: target description, current vs desired state, user journey, and the list of pages and components affected. Design only what the scenario calls for.

Step 2 · Choose Design Approach

Pick the appropriate depth: a quick-fix for small visual or copy changes (skip straight to specification), a sketch-first approach for layout or flow changes, or Phase 6 asset generation for significant visual overhauls.

Step 3 · Design the Change

Produce a before snapshot and an after concept. Explicitly mark the diff — layout, components, content, behavior. Stress-test edge cases: mobile behavior, long text, empty states. Iterate with the user until the design is approved.

Step 4 · Write Specification

Create a mini page-spec in {output_folder}/evolution/specs/ covering change summary, before/after descriptions, component properties and behavior, responsive behavior across breakpoints, and the testable acceptance criteria from the scenario.

Step 5 · Approve Specification

Present the full specification for user sign-off. Verify it matches the scenario intent, that all acceptance criteria are objectively testable, and that the scope is still manageable. No implementation begins until sign-off is recorded.

workflow-implement Implement 5 steps

Translates the approved specification into working code on a dedicated branch, following the existing codebase's conventions. Every commit is incremental and scoped to the specification — no unrelated changes land on the branch. Borrows discipline from Phase 5 (Development).

Step 1 · Load Specification

Read the specification from the Design workflow: change summary, component specs, acceptance criteria, and the list of affected pages. This is the contract the implementation must satisfy.

Step 2 · Create Branch

Create a feature branch using the naming convention evolution/[scenario-name]. All implementation work stays on this branch until the Test and Deploy workflows are complete.

Step 3 · Understand Current Code

Before writing a line, locate the affected files, read the current component implementations, identify the framework and styling patterns, and note any design tokens. Present a brief implementation plan — which files change, what new files are needed, estimated complexity — and wait for confirmation.

Step 4 · Implement Changes

Write code that follows the specification and matches existing patterns exactly. Make only the changes the specification calls for, commit in logical increments, and verify each change works before moving on. Explain what each file change does and why.

Step 5 · Self-Review

Diff all branch changes against main. Check every acceptance criterion is addressed. Confirm no unintended side effects on adjacent pages or components. Clean the branch of debug code, commented-out blocks, and any unrelated edits before handing off to Test.

workflow-test Acceptance Test 6 steps

Validates the implementation against the specification's acceptance criteria before anything reaches production. Covers new functionality, regression, edge cases, and accessibility — and produces a test report with a clear pass/fail recommendation. Borrows discipline from Phase 5 [T] (Acceptance Testing).

Step 1 · Load Test Context

Pull together the specification, scenario, and implementation diff. Extract all acceptance criteria into a test checklist — these are the objective bars the implementation must clear.

Step 2 · Prepare Test Environment

Confirm the evolution/[scenario-name] branch is checked out, start the local development server if needed, navigate to the affected view, and note the URL and any required test data.

Step 3 · Execute Tests

Work through each acceptance criterion — steps, expected result, actual result, pass/fail — and then cover responsive breakpoints, empty and error states, and a regression sweep of pages adjacent to the change. Cross-browser testing applies if the project requires it.

Step 4 · Document Results

Create a test report in {output_folder}/evolution/test-reports/ with a summary (X of Y criteria passed), the full results table, a list of any issues with severity, and a clear recommendation: Pass, Pass with notes, or Fail — needs rework.

Step 5 · Handle Failures

Minor issues are fixed in the same branch and retested. Design issues route back to the Design workflow. Anything that amounts to scope creep is logged as a separate improvement target for the next cycle — it does not get folded into the current branch.

Step 6 · Update Delivery Status

Update the Design Delivery file (DD-XXX) to reflect the test outcome. If approved, mark status: complete and ready_for_production: true. If issues remain, mark status: in_testing and record the issue count for the next retest round.

workflow-deploy Deploy 5 steps

Packages the tested improvement as a pull request, delivers it to the development team with all artifacts linked, and sets up monitoring for post-release impact measurement. The deployment is the handshake between the kaizen cycle and the live product. Borrows discipline from Phase 4 [H] (Design Delivery).

Step 1 · Pre-Deploy Checklist

Verify all acceptance criteria pass per the test report, the branch is clean with no uncommitted changes, commits are logical and well-described, no unrelated changes are included, and documentation is updated where applicable.

Step 2 · Create Pull Request

Open a PR from the evolution/[scenario-name] branch. The PR body includes: what changed, why (linked to the scenario and analysis), how to test (steps from the test report), before/after screenshots for visual changes, and the acceptance criteria checklist.

Step 3 · Package Delivery Context

Create a delivery summary in {output_folder}/evolution/deliveries/ that links the PR and all artifacts in the chain — analysis, scenario, specification, test report — and states the expected improvement and what metrics or signals to watch after deployment.

Step 4 · Notify Team

If the project uses design-log tracking or team notifications, create a completion notification referencing all artifacts in sequence (analysis → scenario → spec → test → PR) and include any monitoring instructions the team needs to act on post-release.

Step 5 · Plan Next Cycle

Archive the completed improvement cycle, review remaining targets from the Analyze output, and propose the next target or trigger a new Analyze pass. Each deploy closes one kaizen loop and opens the possibility of the next.
