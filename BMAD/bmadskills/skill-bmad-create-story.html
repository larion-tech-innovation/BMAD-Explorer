§ WHAT Create Story · What does this Skill do?

Skill code
bmad-create-story
Trigger phrases
create the next story create story [id]
Agent
Story Context Engine — prevents LLM developer mistakes by front-loading all context the dev agent needs before a single line of code is written.
Core principle
Purpose is NOT to copy content from epics. It is to synthesize, cross-reference, and enrich — creating a single, optimized developer brief the dev agent can act on without consulting any other document.
Automation
Full auto except for initial story selection. If no story is specified, auto-discovers the first backlog story from sprint-status.yaml.
Output
{implementation_artifacts}/{story_key}.md — comprehensive story file. Sprint status updated: backlog → ready-for-dev.
Upstream
bmad-sprint-planning has run and sprint-status.yaml exists. Epic files exist in planning_artifacts.
Downstream
bmad-dev-story reads this file as its sole input. The dev agent receives no other context — everything must be in this one file.

Every Phase 4 implementation failure traces back to information the developer didn't have when they started. Create Story makes that failure impossible by building a complete developer brief from every available artifact — epics, architecture docs, UX specs, previous story learnings, and live web research — assembled at the moment it is needed, not upfront.

§ WHY Why Create Story?

Without a story context file, the dev agent starts implementation with nothing but an epic header and a list of acceptance criteria — the same information that caused every previous implementation failure.

Dimension Without Create Story With Create Story
Technical context Developer re-derives architecture patterns from scratch — may choose wrong library or pattern Architecture guardrails pre-extracted: correct libraries, naming conventions, file locations embedded in story
Continuity Each story starts with no memory of previous work — same mistakes repeated, patterns reinvented Previous story intelligence (dev notes, review feedback, established patterns) propagated forward
Library currency Developer uses whatever version they know — may hit breaking changes or security vulnerabilities Web research embeds latest API docs, deprecations, and security advisories at time of creation
Regression risk UPDATE files read mid-implementation — current state and preserved behaviors often missed Every UPDATE file read and documented upfront: current state, what changes, what must be preserved
UX compliance Developer implements functional behavior, UX requirements discovered in review UX design requirements extracted and embedded — violations caught before a line of code is written
Dev agent focus Dev agent must load and parse multiple documents mid-implementation — context spread, inconsistent One file, one context. Dev agent acts without consulting any other document.

just-in-time The Just-in-Time Principle

Why Not Bulk-Prep All Stories Upfront? Preparing 10 stories at the start of a sprint means the last 5 are written before the first 5 reveal architectural discoveries, pattern shifts, and implementation learnings. By the time dev reaches story 7, story 7's file reflects the project understanding from sprint day 1 — not from the day dev actually starts it. Just-in-time preparation means every story reflects current understanding at the moment it is needed.
Bulk-Prep vs. Just-in-Time 04-B-create-story.md
TimingBulk-Prep (all stories upfront)Just-in-Time (one story before dev)
Story freshness Stories 6–10 reflect sprint-day-1 understanding Every story reflects understanding at the moment dev starts it
Pattern propagation Patterns from story 1 not yet known when stories 2–10 are written Every story inherits learnings from the one before it
Rework cost Stale stories must be rewritten as architecture evolves No rework — stories are always written against current reality
Web research Library docs from sprint start — may miss critical updates shipped mid-sprint Research run at dev time — catches latest changes

mistake-prevention Mistake Prevention Grid 8 categories

The checklist names eight LLM developer mistakes explicitly. Create Story is designed to make each one impossible — not by reminding the developer, but by pre-loading the context that removes the opportunity for error.

Mistake 01
Reinventing wheels — creating duplicate functionality instead of reusing existing
Architecture analysis extracts every reusable component, utility, and pattern. The dev agent sees what already exists before writing a line.
Mistake 02
Wrong libraries — using incorrect frameworks, versions, or dependencies
Web research step identifies the exact library versions specified in the architecture and fetches current API docs — version mismatches caught before implementation.
Mistake 03
Wrong file locations — violating project structure and organization
Architecture guardrails extraction maps every file the story will touch to its correct path and naming convention.
Mistake 04
Breaking regressions — changes that break existing functionality
Every UPDATE file is read completely. Current behavior, what the story changes, and what must be preserved are all documented in Dev Notes before dev begins.
Mistake 05
Ignoring UX — implementing functionality without user experience requirements
UX design file is loaded and relevant requirements extracted into the story. UX violations require ignorance of explicitly stated requirements — made impossible.
Mistake 06
Vague implementations — unclear or ambiguous implementation guidance
Checklist validates for LLM optimization: clarity over verbosity, actionable instructions, unambiguous language. Every sentence must guide implementation.
Mistake 07
Lying about completion — implementing incorrectly or incompletely and marking done
Acceptance criteria are BDD-formatted and testable. Dev Agent Record section requires the agent to list every file it created or modified.
Mistake 08
Not learning from past work — ignoring previous story learnings and patterns
Previous story intelligence is loaded explicitly: dev notes, review corrections, approved patterns, and encountered problems from the prior story.

when-to-use When to Use This Skill

  • Phrase 1"create the next story" — auto-discovers first backlog story
  • Phrase 2"create story 1-2" / "create story epic 2 story 4" — targets specific story

§ HOW How to Execute · 6-Step Workflow

Step 1 · Determine Target Story

  • GoalIdentify which story to create — from user input or by scanning sprint-status.yaml.
  • User-providedIf user gives a story path (e.g. 1-2, 1-2-user-auth), parse it directly into epic_num, story_num, story_key.
  • Auto-discoverRead the FULL sprint-status.yaml from beginning to end. Find the first entry matching the pattern N-N-name with status backlog.
  • Epic activationIf this is the first story in its epic (story_num = 1) and the epic status is backlog, update it to in-progress.
  • Halt conditionsNo backlog stories found · Epic is already done · No sprint-status.yaml and no explicit story path provided.

Step 2 · Load and Analyze Core Artifacts

  • GoalBuild complete context from all available project documents. Exhaustive — no skimming.
  • DiscoveryFollow discover-inputs.md protocol: try sharded version first, fall back to whole document, load all matching files.
  • Epic analysisExtract: epic objectives, all stories (cross-story context), target story's user statement, acceptance criteria, technical constraints, dependencies.
  • Previous storyIf story_num > 1, load the prior story file. Extract dev notes, review feedback, established patterns, and problems encountered.
  • Git historyIf prior story exists and a git repo is detected, analyze last 5 commits: files modified, libraries added, testing approaches used.

Step 3 · Architecture Analysis for Developer Guardrails

  • GoalExtract everything the developer MUST follow — non-negotiable technical constraints embedded directly in the story file.
  • Architecture scanFor each architecture section: technical stack with versions, code structure and naming, API patterns and data contracts, database schemas, security requirements, testing standards, deployment patterns.
  • File readingFor every file the story will touch (marked UPDATE in architecture), read it completely. Document: current state, what this story changes, what must be preserved.
  • Non-negotiableA story implementation must leave the system working end-to-end — not just satisfy stated ACs. Behaviors required for correctness in the existing system are implicit requirements, whether or not explicitly written.

Step 4 · Web Research for Latest Technical Specifics

  • GoalPrevent outdated implementations by embedding current API knowledge at the moment of story creation.
  • ScopeFrom the architecture analysis, identify every library, framework, or external API the story will interact with.
  • ResearchFor each critical technology: latest stable version, breaking changes, security vulnerabilities, performance improvements, deprecations.
  • EmbedInclude in the story file: specific library versions, API endpoints with parameters, recent security patches, migration considerations.

Step 5 · Create Comprehensive Story File

  • GoalWrite the complete developer brief — the single file the dev agent will use for the entire implementation.
  • InitializeStart from template.md, writing to {implementation_artifacts}/{story_key}.md.
  • LLM optimizationClarity over verbosity. Every sentence guides implementation. Scannable structure. Token-efficient — no content that doesn't directly help the developer.
  • StatusSet story Status to ready-for-dev in the file header.
  • Save questionsIf clarifications arise during analysis, hold them until after the complete story is written — do not interrupt the workflow to ask.

Step 6 · Update Sprint Status and Finalize

  • GoalValidate the story file, save it unconditionally, and update sprint tracking.
  • ValidationRun the story file against checklist.md. Apply any required fixes before saving.
  • Sprint updateLoad the FULL sprint-status.yaml. Find the matching story_key entry. Update its status from backlog to ready-for-dev. Update last_updated. Preserve ALL comments and structure.
  • ReportStory ID, story key, file path, status. Next steps: review file → run dev-story → run code-review when complete.

auto-discover Auto-Discovery Logic

When no story is specified, the skill reads the entire sprint-status.yaml from top to bottom and picks the first eligible story. Order in the file is order of work.

Story Selection Flow — No User Input
1 Check if sprint-status.yaml exists. If not: offer three options — run sprint-planning, provide explicit story path, or quit.
2 Read the COMPLETE file from line 1 to end. Parse the full development_status block — do not skip any content.
3 Find the FIRST entry where: key matches N-N-name pattern (not epic-X or epic-X-retrospective) AND status equals backlog.
4 Parse the key: epic_num = first segment · story_num = second segment · story_title = remainder. Set story_id = {epic_num}.{story_num}.
5 If this is the first story in its epic (story_num = 1): check epic status. If backlog or contexted → update to in-progress. If done → HALT with error.
  • If no backlog story is found: halt with options — refresh sprint-planning, add stories via PM correct-course, or check if sprint is complete
  • Legacy status contexted is treated as equivalent to backlog for backward compatibility

validation Validation Checklist

Step 6 validates the story file against checklist.md before saving. Gaps are fixed before the file is finalized — not after.

Disaster Prevention Gaps
  • No wheel-reinvention: existing components and utilities identified and referenced in Dev Notes
  • No wrong libraries: correct library versions from architecture, validated against latest web research
  • No wrong file locations: all file paths match architecture's folder structure conventions
  • No regression risk: every UPDATE file documented with current state + what must be preserved
  • No UX violations: UX requirements from design file extracted and referenced
Acceptance Criteria Quality
  • All ACs from the epic file are present — none omitted
  • Each AC is testable and has a corresponding task or subtask
  • Tasks reference their AC by number: (AC: #1), (AC: #2)
LLM Optimization (Dev Agent Readability)
  • Clarity over verbosity: no filler sentences, no restating the obvious
  • Actionable instructions: every sentence either constrains, guides, or informs an implementation decision
  • No ambiguity: requirements have no room for multiple valid interpretations
  • Token-efficient: maximum guidance per word — no verbosity without value
Completeness
  • Story status set to ready-for-dev in the file header
  • Dev Notes cover architecture patterns, previous story learnings (if applicable), and latest tech notes (if researched)
  • All technical details cited with source path and section in References
  • sprint-status.yaml updated: story status backlog → ready-for-dev, last_updated refreshed
