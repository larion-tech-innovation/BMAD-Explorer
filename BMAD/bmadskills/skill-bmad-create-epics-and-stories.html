This skill turns PRD + Architecture (+ optional UX Design) into a fully decomposed epics.md. It extracts functional, non-functional, architecture, and UX design requirements; proposes user-value-centric epics with coverage maps; generates BDD stories with 5–7 Given/When/Then acceptance criteria; and validates the full output against 5 mandatory quality checks before closing.

principles 5 Core Principles

These principles are non-negotiable constraints that shape every decision from epic naming to AC depth. Violations are caught in Step 4 validation.

Principle 01
User Value First
Epics enable users to accomplish something meaningful. Not technical milestones. Example failures: "Database Setup", "API Development" — both deliver zero standalone user value.
Principle 02
Standalone + Enables
Each epic delivers complete functionality for its domain AND enables future epics — without requiring future epics to function. Epic 1 must be shippable independently.
Principle 03
No Forward Dependencies
Story N.2 can only depend on Story N.1. No story may reference a story not yet complete. Validated explicitly in Step 4.5 at both epic and story levels.
Principle 04
JIT Entity Creation
Database tables and entities are created only when a story needs them. Not upfront. Creating all 50 tables in Epic 1 Story 1 is an anti-pattern — it's a commitment before stories have validated what's needed.
Principle 05
Single Dev Agent Size
Each story must be completable within one dev agent context window. Stories that span too much scope (e.g., "Build entire authentication system") must be split before approval.

structure Output Document Structure

The skill writes a single epics.md file. Every section is generated sequentially during the 4-step workflow and accumulated into the final document.

epics.md
├── Frontmatter: stepsCompleted, inputDocuments
├── ## Overview              ← project name
├── ## Requirements Inventory
│   ├── ### Functional Requirements    ← FR list
│   ├── ### Non-Functional Requirements ← NFR list
│   ├── ### Additional Requirements    ← from Architecture doc
│   ├── ### UX Design Requirements     ← UX-DRs (if UX doc exists)
│   └── ### FR Coverage Map            ← FR → Epic mapping
├── ## Epic List             ← titles + FR coverage per epic
├── ## Epic 1: {title}
│   ├── ### Story 1.1: {title}
│   └── ### Story 1.2: ...
└── ## Epic 2: {title}
    └── ...
  • FrontmatterstepsCompleted: [] — updated after each step; enables exact resume from interruption without redoing work. inputDocuments: [] — all source files listed.
  • FR Coverage MapGenerated in Step 2. Every FR number mapped to the epic(s) that address it. No FR may be left unmapped.
  • Epic ListOne-line summary per epic: title, user outcome, FR numbers covered. Written before stories are generated.
  • UX-DRs SectionWritten as a separate section inside Requirements Inventory — never merged with Additional Requirements. Each UX-DR is specific enough to generate a story with testable AC.

§ WHY Design Decisions — Why is the skill built this way?

Why Epics After Architecture?

Pre-v6 antipattern: create epics and stories directly from the PRD, then architecture later. Result — stories miss technical context and require rework mid-sprint. The v6 order is fixed:

Phase Pre-v6 (wrong) v6 (correct)
Order PRD → Epics & Stories → Architecture PRD → Architecture → Epics & Stories
Result Stories have no tech context — rework when architecture is designed later Stories reflect auth approach, DB schema decisions, tech patterns
AC quality Vague AC with no architecture-specific constraints AC includes queue names, encryption spec, polling intervals, DB constraints

Why User Value Over Technical Milestones?

A "Database Setup" epic delivers zero user value standalone. If Epic 1 fails to ship, the product is dead with nothing users can see or test. Epics must ship something users can actually use — giving early feedback, validation, and product value at every stage.

  • Epic named "Database Setup" → ❌ delivers nothing if Epic 2 slips
  • Epic named "User Registration & Profiles" → ✅ shippable, testable, valuable on its own
  • Technical work still happens — it lives inside stories that deliver user-visible features, not as a standalone epic

Why UX-DRs as First-Class Input?

UX requirements have the same rigor as functional requirements. Each UX-DR must be specific enough to generate a story with testable acceptance criteria. Treating UX as supplementary documentation means design work never becomes a tracked story with AC.

  • Vague (wrong)"Create reusable components" — not actionable, cannot generate a story or testable AC.
  • Specific (right)"Implement ConfirmActions, StatusMessage, EmptyState, and FocusIndicator components" — generates a concrete story per component with AC.
  • Separate sectionUX-DRs live in their own ### UX Design Requirements section — never merged into Additional Requirements — so coverage is tracked independently.
  • Coverage checkAfter all epics are generated, the skill verifies every UX-DR is covered either in a relevant epic or a dedicated "Design System / UX Polish" epic.

Why JIT Entity Creation?

Upfront schema creation is a commitment before stories have validated what's actually needed. Creating all database tables in Epic 1 Story 1 means building against assumptions, not validated requirements. JIT keeps the schema evolving in lockstep with the stories that need each table.

  • Anti-pattern: Epic 1 Story 1 creates 50 tables → story is untestable, oversized, and commits to unvalidated design
  • Correct: the users table is created in the story that implements user registration — not before
  • Architecture starter template (if specified) is Epic 1 Story 1 by convention — this is the one exception where setup precedes user features

Why stepsCompleted Frontmatter?

Large PRDs with many FRs and UX-DRs can exhaust session context limits before all stories are generated. The stepsCompleted: [] frontmatter in epics.md enables exact resume from the interruption point — no restart, no repeated work, no lost output.

  • Session interrupted after Step 2 → resume at Step 3 with epic list already confirmed
  • Context exhaustion mid-story generation → resume at the exact epic/story boundary
  • Works across usage limit resets — the file persists even when the session does not

per-step 4 · Per-Step Preparation

The 4-step process runs sequentially. Each step builds on the previous. Having these ready prevents mid-session pivots that restart earlier steps.

Step-Level Readiness bmad-epics-stories · 4-step process
StepWhat to have ready
Step 1 · Document Discovery Know which planning artifacts apply and where they live. The agent uses a two-pass strategy to find inputs — be ready to confirm or reject paths. Also be ready to confirm the FR count, NFR count, and UX-DR count before proceeding to Step 2.
Step 2 · Epic Decomposition Know the Architecture context (if available) — this determines whether the agent writes stories in PRD-requirement terms or technical-specification terms. Know the phase boundaries so the agent decomposes only the right scope.
Step 3 · Story Writing Be ready to review AC depth: 5–7 Given/When/Then scenarios per story covering the full behavior spectrum (happy path, alternative paths, error cases, edge cases). Stories that don't have testable AC need revision before Sprint 1.
Step 4 · Finalize Verify all FRs are covered by at least one story. The agent produces a Requirements Coverage Map — review it and flag any FR with no story coverage. Stories that cover FRs not in the current phase should be moved or removed.
What Happens in Each Step
  • Step 1 · Document DiscoveryAgent scans all input documents and builds the Requirements Inventory: all Functional Requirements (numbered, with testable Consequences), all Non-Functional Requirements (performance / security / usability / reliability), Architecture requirements (starter template, infra, migrations, monitoring), and UX Design Requirements (components, tokens, a11y, responsive, interactions). User confirms counts; agent updates until confirmed before moving to Step 2.
  • Step 2 · Epic DecompositionFR clusters → natural user journeys → epics. Each epic is validated against three criteria: user-deliverable (can demo standalone), independent value (users benefit if only this epic ships), and reasonable scope (2–6 weeks). File overlap check prevents file churn epics. Requirements Coverage Map (FR → Epic) is produced. Epic-to-epic dependencies are surfaced. User gives explicit approval before Step 3 begins.
  • Step 3 · Story GenerationFor each epic in sequence: agent writes stories in As a / I want / So that format. Each story gets 5–7 Given/When/Then AC blocks covering happy path, alternative paths, error cases, and edge cases. Architecture specifics (queue names, DB constraints, encryption spec) and UX specifics (component names, color tokens, interaction mechanics) flow directly into AC. User reviews and confirms each story before it is appended to epics.md.
  • Step 4 · Finalize + ValidateFive-check gate: (4.1) FR Coverage — every FR in at least one story; (4.2) Architecture implementation — JIT entity creation, starter template story present; (4.3) Story quality — single dev agent context, testable AC, no vague "system responds appropriately"; (4.4) Epic structure — user value focus, no file churn epics; (4.5) Dependency validation — forward-dependency check at both epic and story levels. All 5 must pass. Agent invokes bmad-help for next-steps recommendation on completion.

Technique Descriptions
Story Format — Fixed, No Deviation
FieldFormat
Heading### Story {N}.{M}: {story_title} — e.g., Story 1.3: User Registration with Email
User storyAs a {user_type}, I want {capability}, so that {value_benefit}.
Acceptance criteriaGiven / When / Then BDD format — 5–7 blocks per story covering happy path, alternatives, errors, edge cases, and any performance NFR
DependenciesOnly previous stories in the same epic — Story N.2 may depend on N.1, never on a future story
Specificity ruleArchitecture specifics (queue names, encryption spec, DB constraints) and UX specifics (component names, color tokens, interaction mechanics) flow directly into AC — no vague "system responds appropriately"
AC Depth — Given/When/Then Scenarios
Coverage TypeWhat It Verifies
Happy path The primary success flow — user does the right thing with valid data. Every story must have at least one happy path scenario.
Alternative path Valid but non-primary flows — user selects optional features, completes the task a different way, or cancels partway through.
Error cases System returns expected error states — invalid input, failed API call, permissions denied. The error message must be specific and actionable.
Edge cases Boundary conditions — empty state (zero items), max size reached, concurrent access, race conditions. These are the scenarios most likely to cause bugs in production.
Performance If there is a performance NFR relevant to this story: the AC includes a measurable threshold ("completes in under 3 seconds with 10,000 rows").
UX Design Requirements (UX-DRs) — First-Class Input
PrincipleDetail
Treatment If a UX doc exists, it is treated as first-class input — the same rigor as FRs, NOT supplementary material. UX-DRs are extracted with the same discipline as FRs and NFRs.
What to find Interaction patterns that must be implemented exactly as specified · Component behavior that differs from the design system default · Motion and transition requirements · Empty, loading, and error state designs · Accessibility requirements specific to this feature
Epic coverage requirement Every UX-DR must be covered by at least one story's acceptance criteria. UX decisions with no story coverage will not be implemented — they become invisible gaps.
File Overlap Check — Epic Boundary Validation
SignalAction
High overlap detected Multiple epics repeatedly target the same core files → consider merging those epics. Parallel epic development on the same files creates merge conflicts and coordination overhead in Sprint 1.
Zero overlap Epics are well-separated — each targets a distinct area of the codebase. This is the target state for parallel development.
Shared foundation files Files touched by almost all epics (e.g., routing, auth, shared components) should be in their own foundation epic, completed before parallel epics begin.
