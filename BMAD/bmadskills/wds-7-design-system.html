§ WHAT Design System — What does this Skill do?

Skill code

Role
Living component-library and design-token curator — grows the system organically from real Phase 4 spec patterns rather than designing it upfront.
Phase
WDS Phase 7 — optional. Runs when a project is large enough that component reuse and token consistency start to matter.
Inputs
Phase 4 page specifications, Phase 6 visual assets, and any existing component library (brownfield).

This skill builds and maintains a living component library and design-token system extracted from actual page specifications. The defining principle is that the design system is not designed upfront — it grows organically, emerging from the component patterns that appear in real Phase 4 specs. Components are catalogued as they prove themselves in use, and tokens are extracted from the visual assets the project actually ships.

Use it once enough page specifications exist for genuine patterns to surface — typically after several Phase 4 pages — or when onboarding a brownfield project that already carries a component library, in which case Freya maps those existing components into the WDS spec format.

§ WHY Why a Design System? — The reasoning behind each design choice

why-not-upfront Why Not Design the System Upfront?

A design system designed before any pages exist is a system of guesses. Premature abstraction creates tokens and components that don't map to the real UI — they have to be reworked or quietly ignored once actual screens appear. WDS instead builds the system from the patterns that emerge in Phase 4 specifications: a component earns its place in the library only after it has proven itself across real pages. The system is always a description of what the product actually uses, never a speculation about what it might.

why-reuse-decision Why the Reuse Decision Is the Most Important Checkpoint

Step 6 — Present Decision — is where the human stays in the loop. Steps 1–5 do the analysis: scan, compare, score similarity, surface opportunities, and flag risks. But the call between reuse, extend, and create is a judgment about the product's coherence, not a mechanical one. Freya presents each option with its trade-offs and lets you decide. Reuse keeps the system tight but can force-fit a pattern; extending adds a variant but grows the surface; creating is clean but risks duplication. This is the decision worth investing time in — everything after it just executes the choice.

why-import-exists Why Import [I] Exists

Not every project starts from a blank library. When you join a brownfield project that already has a component library, rebuilding it from scratch would throw away working, tested UI. Import handles this: Freya maps the existing components into the WDS spec format so they live alongside spec-derived components under one catalog and token vocabulary — bringing the legacy system into the WDS pipeline rather than competing with it.

why-living-system Why a Living System Beats a Static One

A static design system is correct on the day it ships and drifts from reality every day after. Because the WDS system grows organically from real specs and assets, it stays in sync with the product as the product changes — new patterns get catalogued, conflicts get flagged at the moment of creation, and tokens trace back to the assets that defined them. The Browse activity makes that living state explorable, so the system remains a source of truth rather than a document nobody trusts.

§ HOW Sub-Workflows — How each activity works, step by step

Phase 7 routes to one of five dedicated workflow files depending on which activity you choose. Each workflow is self-contained — it reads design log state, executes its steps, and returns you to the Activity Menu. Below is a comprehensive reference for all five.

workflow-create.md Create Component 11 steps

Builds a new design system from scratch or adds new components to an existing one with automatic duplicate detection. Before writing a single component file, Freya runs a full similarity analysis — scanning, comparing, scoring, surfacing opportunities, and flagging risks — so the Step 6 reuse decision is backed by data, not guesswork. Entry routing skips the duplicate-detection phase when the design system is empty or when the required operation is already known.

Step 1 · Initialize Design System

Runs only when no design system exists yet. Creates the D-Design-System/ folder structure, writes placeholder token and config files, and sets up the component index. After initialization the workflow continues to Step 3 for the first component. Executed via step-08a-initialize-design-system.md.

Step 2a · Scan Existing Components

Reads every file in D-Design-System/components/, extracts component metadata (ID, variants, states, key styling attributes, usage count), and builds a candidate list filtered by matching component type. If no matches are found the workflow routes directly to step-08b-create-new-component.md; if the library is empty, to step-08a-initialize-design-system.md.

Step 2b · Compare Attributes

Runs a systematic four-dimension comparison of the incoming component specification against each candidate: visual attributes (size, shape, color, typography, spacing, borders), functional attributes (purpose, user action, input/output type), behavioral attributes (states, interactions, animations, keyboard, accessibility), and contextual attributes (usage pattern, frequency, journey stage). Each dimension is scored High / Medium / Low and produces a structured diff.

Step 2c · Calculate Similarity

Applies a weighted formula — Visual 30%, Functional 30%, Behavioral 25%, Contextual 15% — to derive an overall similarity percentage. Maps the result to one of six levels: Identical (95–100%), Very High (80–94%), High (65–79%), Medium (45–64%), Low (20–44%), or No Similarity (<20%). Generates an initial recommendation and highlights the key decision factors that will most influence the Step 6 choice.

Step 2d · Identify Opportunities

For each of the three decision paths — Reuse, Add Variant, Create New — enumerates the concrete benefits: consistency and maintenance gains for reuse; flexibility and scalability gains for variants; semantic clarity and independence gains for new components. Weighs these against project context (design system maturity, team size, timeline) and surfaces the strongest opportunities per path.

Step 2e · Identify Risks

Mirrors the opportunity analysis with a risk assessment: loss of semantic distinction and developer confusion for reuse; complexity growth and variant explosion for adding variants; visual inconsistency and component proliferation for creating new. Assigns severity (Low / Medium / High) to each risk, identifies deal-breakers, and proposes mitigation strategies where applicable.

Step 2f · Present Decision

Consolidates the full analysis into a structured decision panel: context summary, the three options with trade-off comparison table, a detailed analysis block for each option, and Freya's AI recommendation with reasoning. The user selects Reuse / Add Variant / Create New — or requests more information. This is the key human checkpoint; everything after it executes the chosen path.

Step 2g · Execute Decision

Routes to the appropriate component operation based on the Step 6 choice. Reuse: extracts page-specific content, creates a reference to the existing component, and increments its usage count. Add Variant: updates the component definition with variant-specific styling and states, tracks variant usage, and logs the change to version history. Create New: generates a unique component ID, extracts visual/behavioral/functional attributes, creates the component file from template, and updates the component index.

Step 3a · Initialize Design System (8a)

Creates the design system folder structure, token-category placeholders, mode-specific files (Figma mappings for custom mode, component-library config for library mode), and the component index. Runs once per project — subsequent components skip this step.

Step 3b · Create New Component (8b)

Generates a unique component ID using type-prefix convention (e.g., btn-003), categorizes the component (Interactive / Form / Layout / Content / Feedback / Navigation), extracts all attributes from the spec, populates the full component template, updates the component index, and generates a component reference for use in page specs.

Step 3c · Update Component (8c)

Handles additive or modifying changes to an existing component — new states, styling updates, behavior changes, accessibility improvements, documentation clarifications, or structural refactors. Analyzes breaking-change impact, requires confirmation for destructive changes, applies the update, increments version history, and notifies affected pages. Generates a migration checklist for breaking changes.

Step 3d · Add Variant (8d)

Loads the existing component, extracts what is shared vs. variant-specific, confirms or generates the variant name, appends the variant to the component definition with its styling and state overrides, tracks usage, checks for variant explosion (warn at 3+, review at 5+), and updates version history. Returns a component reference specifying the new variant.

Step 3e · Generate Catalog (8e)

Reads all component files and the design-tokens file, then generates D-Design-System/catalog.html — an interactive HTML page with fixed sidebar navigation, component showcases (all variants + states with live toggle demos), design token swatches (color, typography, spacing), code examples, usage guidelines, Figma links (custom mode), and a full changelog built from component version histories. Regenerated automatically after every component change.

workflow-browse.md Browse Library 4 steps

Generates and serves a disposable localhost application for exploring the design system in full — tokens, components, relationships, and natural-language intent search. The app is built fresh from the current data each time, so it always reflects the latest state. Any changes the user identifies while browsing are logged and routed to the appropriate activity on exit.

Step 1 · Load Design System Data

Reads all design system files: design-tokens.md, every component file in components/*.md, and component-library-config.md. Parses the data into structured form — tokens with categories, components with dependency lists, and a relationship graph linking tokens to the components that reference them.

Step 2 · Generate Browser Application

Builds a single-page app with four views. Token Explorer: an Airtable-style table filterable by category and sortable by name or value, with a live preview column (color swatches, spacing bars, rendered typography). Component Catalog: a grid of all components with thumbnail previews, expandable detail showing all variants and token dependencies, and category filters. Relationship Viewer: an interactive graph where clicking a component highlights its token chain and clicking a token highlights every component that references it. Intent Search: natural-language input ("I need a background for warning messages") matched against token names, descriptions, categories, and usage context — results include live previews and copy-ready token names.

Step 3 · Serve and Interact

Starts the localhost server and presents the four URL paths (/tokens, /components, /graph, /search). The user browses freely; Freya remains available to answer questions about what they find. The server stays running until the user signals exit.

Step 4 · Capture Actions

If the user identifies changes while browsing (rename a token, add a variant, fix an inconsistency), each item is logged with component name, description, and severity. On exit the action list is presented and routed to the appropriate activity: Create [C], Edit [E], or View [V]. The generated app is discarded after the session; the design system source files are unchanged.

workflow-view.md View Component 4 steps

Renders selected components in a focused localhost preview so the user can inspect them visually with all states and variants side by side. Lighter than Browse — no relationship graph or token explorer — just a clean per-component showcase. Any feedback captured routes to Edit [E] or Create [C] for follow-up.

Step 1 · Select Components

Presents the component catalog as a numbered list with variant counts. The user selects components to preview by number, comma-separated, or "all". The selection drives which component files are loaded in the next step.

Step 2 · Generate Preview App

Reads the selected component specs and the design-tokens file, then generates a minimal HTML/CSS localhost app that renders each component with: all variants displayed side by side, all interactive states (default, hover, active, disabled, focus), responsive breakpoint switching, and dark/light mode if defined in the tokens. The app is served immediately.

Step 3 · Interactive Review

With the preview running, the user inspects components visually. If changes are needed, the user can request them inline — Freya routes to Edit [E] for Figma-based changes or Create [C] (update path) for spec-level changes. Issues can also be flagged for later without blocking the preview session.

Step 4 · Capture Feedback

Any issues or desired changes noted during review are logged — component name, issue description, severity — and a next-action suggestion is offered: edit in Figma, update via Create, or defer. The localhost server is stopped and the design log updated before returning to the Activity Menu.

workflow-edit.md Edit Component 5 steps

Opens selected components in Figma for visual editing, then syncs approved changes back to the WDS design system files. Requires Figma MCP integration or a file export/import round-trip. Changes are diffed against current WDS specs and presented for approval before any files are updated, ensuring the code remains the source of truth.

Step 1 · Select Components

Presents the current component catalog and asks the user to choose which components to edit in Figma. Multiple components can be selected; each is prepared independently in the next step.

Step 2 · Prepare for Figma

For each selected component, reads the current spec and associated design tokens, generates or updates the Figma-compatible representation, and pushes it to Figma via MCP integration (or provides an export file). Confirms that the Figma file is open and ready before pausing for user edits.

Step 3 · User Edits in Figma

Pauses the workflow and waits for the user to make changes in Figma. Freya holds until the user signals completion — this is an explicit human-in-the-loop pause, not a timeout.

Step 4 · Sync Changes Back

Reads updated component data from Figma (via MCP or user export), diffs it against the current WDS specifications, and presents the delta for approval: token values changed, new variants added, properties modified. Only approved changes are written back to the WDS design system files.

Step 5 · Validate Sync

Runs a consistency check on the applied changes: verifies that altered tokens do not break other components that reference them, that all variants are complete, and that naming conventions are maintained. Reports any issues for resolution before marking the edit workflow complete. Catalog generation is then suggested to reflect the updates.

workflow-import.md Import from Spec 5 steps

Brings an existing design system — from a URL, file export, Figma project, or codebase — into the WDS format. Used for brownfield projects that already have a component library. Freya extracts tokens and components from the source, maps them into WDS spec format, and runs a validation pass to ensure consistency before the import is finalized. After import, the Browse activity is the recommended next step to explore what was brought in.

Step 1 · Identify Source

Asks the user for the design system source: a public URL (e.g., Material UI, Chakra UI, or custom documentation), an exported tokens file (JSON, CSS custom properties, or SCSS variables), a Figma design system file (via Figma MCP or export), or an existing codebase with a component library. The source type determines which extraction path is used in Step 2.

Step 2 · Extract Tokens

Reads the source and extracts design tokens across all categories: colors (primary, secondary, semantic, neutrals), typography (font families, sizes, weights, line heights), spacing (scale values, named tokens), shadows (elevation levels), borders (radius, border styles), breakpoints (responsive breakpoints), and motion (transition durations, easing curves). Extracted tokens are presented to the user for review; ambiguous mappings are flagged for clarification.

Step 3 · Extract Components

Identifies reusable components from the source — lists all found components, maps each to the WDS component template format (name, props/variants, token dependencies), and flags components that do not map cleanly. The full component list is presented for user approval before any files are written.

Step 4 · Generate Design System Files

Creates the complete WDS design system file structure: design-tokens.md with all extracted tokens in WDS format, one components/*.md file per component, and component-library-config.md with configuration and metadata. Files are written only after user approval of the extracted token and component lists.

Step 5 · Validate Import

Runs a full validation pass: checks that all tokens referenced by components exist, that no orphaned tokens remain (defined but never referenced by any component), that naming conventions are consistent throughout, and that all component variants are complete. Presents a validation report and allows issues to be fixed interactively before the import is considered finalized.
