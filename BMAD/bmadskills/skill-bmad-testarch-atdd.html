tdd-cycle TDD Red-Green-Refactor

ATDD operates exclusively in the red phase. The green and refactor phases are the developer's responsibility. The skill generates the contract; the developer fulfils it.

Red
Generate Scaffolds
ATDD generates tests that assert expected behavior. All tests wrapped in test.skip() — they don't execute. This phase produces the contract.
ATDD skill · automated
Green
Activate & Implement
Developer removes test.skip() for the current task, runs the test (sees it fail), implements the feature, sees it pass. One task at a time.
Developer · manual
Refactor
Clean Up
Improve code quality while all tests remain green. No new behavior, no skipped tests — tests act as a safety net during refactoring.
Developer · manual

What the Red Phase Produces

Every generated test must satisfy three conditions. The aggregation step (step-04c) validates all three before writing files to disk:

  • test.skip()Every test is wrapped in test.skip(). Tests do not execute and do not affect CI pass rates — they are scaffolds, not live tests.
  • Real assertionsTests assert expected behavior — real API responses, correct UI states, proper business logic outcomes. No placeholder assertions like expect(true).toBe(true).
  • expected_to_failEach test file is tagged expected_to_fail: true in the subagent output contract. Validation rejects files where this flag is missing.

Task-by-Task Activation (Developer Workflow)

After ATDD completes, the developer works through the ATDD checklist as they implement the story:

  • 1. Current taskIdentify which acceptance criterion the current implementation task covers.
  • 2. Activate testRemove test.skip() from the corresponding test file or describe block only — leave all other tests skipped.
  • 3. Run & confirm redRun npm test. The activated test should fail (feature not implemented yet). If it passes already, investigate — possible test bug or pre-existing behavior.
  • 4. ImplementWrite feature code until the activated test passes (green).
  • 5. CommitCommit passing tests together with the feature code.
  • 6. Next taskRepeat for the next acceptance criterion.

test-levels Test Level Selection

Level selection is stack-conditional. The skill avoids duplicate coverage — the same behavior tested at multiple levels is waste.

Frontend / Fullstack

Frontend / Fullstack Levels step-03-test-strategy.md
LevelUse for
E2E Critical user journeys — the flows that matter most to the business. Minimal count, maximum coverage of the happy path. High cost per test.
API Business logic and service contracts. Test at the API boundary — faster and more stable than E2E for logic testing.
Component UI component behavior: form validation, state management, user interactions, accessibility. Isolated from full stack.

Backend (No E2E)

Backend Levels step-03-test-strategy.md
LevelUse for
Unit Pure functions, business logic, edge cases, error handling branches. Fast, no I/O, no external dependencies.
Integration Service interactions, database queries, middleware, message queues. Tests real I/O with controlled state.
API / Contract Endpoint validation, request/response schemas, Pact contracts. Tests the HTTP boundary of the service.
E2E NOT used for pure backend. No browser-based testing for services with no UI.

§ WHY Why ATDD? — The case for test-first

Test-last is the industry default — write the feature, then write tests to prove it works. ATDD inverts this: the test is the specification. Implementation only begins after the contract is defined.

Dimension Without ATDD (test-last) With ATDD (this skill)
Test creation timing Tests written after implementation — validate existing behavior, not intended behavior; bias toward what was built, not what was specified Tests written before implementation — define the contract; implementation satisfies it; no survivorship bias
AC traceability Acceptance criteria exist in the story but no direct link to test coverage; easy to miss edge cases under deadline pressure Every AC mapped to at least one test scenario; checklist tracks coverage explicitly per criterion
Implementation clarity Developer interprets story ambiguously; different developers would build different things; inconsistencies found in review Red-phase tests make intended behavior concrete before coding — ambiguities surface immediately, not after implementation
Progress visibility Progress visible only in code — hard to tell what's done vs what's still missing from the acceptance criteria Checklist tracks: skip = not started, failing = in progress, passing = done — exact task-by-task visibility
Story handoff Developer reads story and guesses what to build; no automated bridge from ACs to implementation tasks ATDD artifacts embedded in story Dev Notes — paths to API tests, E2E tests, and checklist; feeds directly into dev-story workflow

when-to-use When to Use This Skill

  • Phrase 1"lets write acceptance tests" / "I want to do ATDD"
  • Phrase 2"generate red-phase tests for this story" / "create test scaffolds"
  • TimingAfter story is approved with clear acceptance criteria. Before any implementation code is written — that's the point.

steps-c Create Mode — 5 Steps + 1 Resume primary

step-01 · Preflight & Context Loading

  • Filesteps-c/step-01-preflight-and-context.md
  • GoalVerify prerequisites, detect stack, load story context, load framework patterns, and initialize knowledge base.
  • Stack detectReads test_stack_type from config or auto-detects from manifests. Sets {detected_stack} = frontend | backend | fullstack.
  • PrerequisitesStory approved with acceptance criteria · framework config present for detected stack. HALT if either is missing.
  • Story contextReads story file, extracts acceptance criteria and constraints. Derives story_key (from filename) and story_id (from metadata / H1 / filename). Stores {story_file} as tracked artifact path for later handoff.
  • Knowledge loadingCore tier always: data-factories.md, component-tdd.md, test-quality.md, test-healing-patterns.md. Frontend/Fullstack adds: selector-resilience.md, timing-debugging.md, Playwright Utils (if enabled). Backend/Fullstack adds: test-levels-framework.md, test-priorities-matrix.md.
  • Config flagsReads: tea_use_playwright_utils, tea_use_pactjs_utils, tea_pact_mcp, tea_browser_automation.
  • Output fileCreates {test_artifacts}/atdd-checklist-{story_key}.md with frontmatter: storyId, storyKey, storyFile, generatedTestFiles: [].

step-01b · Resume Workflow

  • Filesteps-c/step-01b-resume.md
  • GoalResume from last saved checkpoint. Reads lastStep from checklist frontmatter, displays progress, routes to next incomplete step.

step-02 · Generation Mode Selection

  • Filesteps-c/step-02-generation-mode.md
  • GoalChoose between AI generation (default) or browser recording (Frontend/Fullstack optional).
  • AI generationDefault for all stacks. Mandatory for Backend. Used when acceptance criteria are clear and scenarios are standard (CRUD, auth, API, navigation).
  • Recording modeFrontend/Fullstack only. Used for complex UI interactions requiring live browser verification. Tool selected from tea_browser_automation config: cli → Playwright CLI, mcp → Playwright MCP, auto → CLI for simple / MCP for complex.
  • BackendAlways AI generation. Generates from API docs, OpenAPI/Swagger specs, or source code analysis — no browser recording needed.

step-03 · Test Strategy

  • Filesteps-c/step-03-test-strategy.md
  • GoalTranslate acceptance criteria into a prioritized, level-appropriate test plan before any generation begins.
  • Criteria mappingEach acceptance criterion → one or more test scenarios. Negative and edge cases included where risk is high.
  • Level selectionAssign best test level per scenario based on detected stack. See Test Level Selection section above.
  • PriorityAssign P0–P3 using risk and business impact.
  • Red phase checkConfirms all tests are designed to fail before implementation. Tests that would trivially pass indicate a test design problem.

step-04 · Orchestrate Parallel Generation

  • Filesteps-c/step-04-generate-tests.md
  • GoalSelect execution mode, launch two parallel workers, wait for completion, verify red-phase compliance, proceed to aggregation.
  • WorkersWorker A (API red-phase) and Worker B (E2E red-phase) run in parallel via agent-team or subagent. Sequential fallback available.
  • Execution modeResolution order: explicit user request → tea_execution_mode config → capability probe. agent-team → subagent → sequential.
  • Temp filesEach worker writes to /tmp/tea-atdd-{api|e2e}-tests-{timestamp}.json. Aggregation step reads these files.
  • Red phase guardProceeds to step-04c only when BOTH workers complete successfully AND output files contain valid JSON.

step-04a/b/c Parallel Workers & Aggregation

Worker A
API Red-Phase Tests
  • Reads: story acceptance criteria + test strategy
  • Generates: API endpoint tests with real assertions
  • Wraps all tests in test.skip()
  • Marks: expected_to_fail: true
  • Covers: business logic, request/response, contracts
Output → /tmp/tea-atdd-api-tests-{ts}.json
Worker B
E2E Red-Phase Tests
  • Reads: story acceptance criteria + test strategy
  • Generates: E2E user journey tests with real assertions
  • Wraps all tests in test.skip()
  • Marks: expected_to_fail: true
  • Covers: UI flows, navigation, component behavior
Output → /tmp/tea-atdd-e2e-tests-{ts}.json

step-04c · Aggregate & Validate

  • Filesteps-c/step-04c-aggregate.md
  • GoalRead both worker outputs, validate TDD compliance, write test files to disk, generate fixtures and ATDD checklist.
  • TDD validationFor every test: must have test.skip(), must not have placeholder assertions (expect(true).toBe(true)), must be marked expected_to_fail. Any violation throws an error — does NOT proceed.
  • Write filesWrites API and E2E test files to tests/api/ and tests/e2e/ respectively.
  • FixturesGenerates minimal fixture stubs needed by the scaffolds (e.g., tests/fixtures/test-data.ts). More complete fixtures added during green phase.
  • ATDD checklistWrites {test_artifacts}/atdd-checklist-{story_key}.md with acceptance criteria coverage, test counts, and task-by-task activation guide.
  • Story handoffIf story file is writable, adds an ### ATDD Artifacts subsection to ## Dev Notes in the story — links checklist path, API test path, E2E test path. Falls back gracefully if story cannot be updated.

step-05 · Validate & Complete

  • Filesteps-c/step-05-validate-and-complete.md
  • GoalFinal validation against checklist, polish output, completion report.
  • ValidatesPrerequisites satisfied · test files created with test.skip() · checklist covers all acceptance criteria · story metadata captured · CLI sessions cleaned up · temp artifacts in {test_artifacts}/ not random paths.
  • Completion reportTest files created · checklist path · story key + story file handoff path · key risks or assumptions · recommended next workflow.

gen-modes Generation Modes

AI Generation (Default)

The standard mode for all stacks. No live browser required. The skill generates tests from story acceptance criteria, API documentation, OpenAPI/Swagger specs, or source code analysis.

  • Mandatory for Backend stacks (no UI, no browser recording needed)
  • Best for clear acceptance criteria and standard scenarios: CRUD, auth, API endpoints, navigation flows
  • Uses loaded knowledge fragments (data-factories, test-quality, selector-resilience, etc.) to produce realistic test patterns

Browser Recording (Optional — Frontend / Fullstack)

Use for complex UI interactions that benefit from live browser verification before generating tests. Controlled by tea_browser_automation config flag.

Browser Automation Config step-02-generation-mode.md
Config valueBehaviorBest for
auto CLI for simple recording (snapshot selectors); MCP for complex (drag/drop, wizards). Falls back if preferred tool unavailable. Default — let the skill decide
cli Uses Playwright CLI for all recording: playwright-cli open <url>, snapshot, screenshot, click <ref> Standard UI flows, selector capture
mcp Uses Playwright MCP tools for full browser automation with rich tool semantics Drag-and-drop, multi-step wizards, complex state
none Recording skipped entirely — falls back to AI generation from documentation No browser available, or CI-only environment

checklist Key Checklist Items

Prerequisites
  • Story approved with clear, testable acceptance criteria
  • Test framework configured for detected stack (Playwright/Cypress for FE; backend config for BE)
  • Development environment accessible
TDD Compliance
  • All generated tests use test.skip() — none are active
  • No placeholder assertions (expect(true).toBe(true))
  • All tests marked expected_to_fail: true
  • Activated tests would fail before implementation (verified by design)
Coverage
  • Every acceptance criterion has at least one test scenario
  • Negative cases included for high-risk criteria
  • No duplicate coverage across test levels
  • Test level selection appropriate for detected stack
File & Artifact Hygiene
  • All test files written at correct paths (tests/api/, tests/e2e/)
  • ATDD checklist written to {test_artifacts}/atdd-checklist-{story_key}.md
  • Story metadata (storyId, storyKey, storyFile) captured in checklist frontmatter
  • CLI sessions cleaned up — no orphaned browser processes
  • Temp files in {test_artifacts}/, not arbitrary system paths
