§ WHAT NFR Assessment — What does this Skill do?

Skill code
bmad-testarch-nfr  ·  Menu code: NF
Goal
Validate whether a system meets its Non-Functional Requirements before a release gate — gathering real evidence (test results, metrics, logs, security scans) and dispatching four parallel workers to assess each domain independently.
Role
Master Test Architect acting as a pre-release quality auditor. Looks at system behaviour — not test code quality (that is test-review). Answers: Is this system fast enough? Secure enough? Reliable enough? Does it scale?
Output
nfr-assessment.md — overall risk level, per-domain breakdown, evidence summary, remediation actions with urgency, compliance matrix, gate-ready YAML snippet
Prerequisites
Implementation must be accessible. Evidence sources must be available (test results, metrics, logs, scans). Skill HALTS if either is missing — prompts user for the missing inputs before proceeding.

4 Assessment Workers — Architecture Overview

Step 4 dispatches all four workers in parallel. Each evaluates one NFR domain independently and outputs a domain risk level (NONE/LOW/MEDIUM/HIGH) plus PASS/CONCERN/FAIL per sub-category. Step 4E aggregates all four results using the worst-case rule.

§ WHY Why NFR Assessment?

A system that passes all its unit and integration tests may still fail in production on load, expose security vulnerabilities, or go down when a dependency fails. NFR Assessment fills the gap between "tests are green" and "system is production-ready."

Dimension Without NFR Assessment With NFR Assessment (NF)
Release decision Based on unit/integration test results — green CI is assumed sufficient Evidence-based: performance metrics, security scans, burn-in results assessed before release gate
Security posture Security issues discovered via external pentest or in production — expensive and public Auth, data protection, input validation, API security assessed against evidence before any deployment
Performance evidence "Works on my machine" — no measured SLO validation against defined thresholds P95 response times, throughput, resource utilisation compared against thresholds from tech-spec
Cross-domain risks Single-domain checks — a performance PASS doesn't reveal that it degrades under load at scale Cross-domain detection: Perf+Scalability compound risk, Sec+Reliability cascade risk — explicitly flagged
Compliance status Ad-hoc — hope that requirements (SOC2, GDPR, HIPAA) were met during development Per-domain compliance matrix (SOC2/GDPR/HIPAA/PCI-DSS) automatically generated with PASS/PARTIAL/FAIL
Missing evidence Missing data is treated as "no issue found" — a silent false positive Missing evidence = automatic CONCERNS status — forces an explicit decision before release
CI integration No structured output — release decisions stay in documents and chat Gate-ready YAML snippet output — directly usable as a CI pipeline release gate condition

Why These 4 Domains?

The four domains cover all production failure modes that are invisible to functional tests:

DomainFailure mode it catchesWhy it needs its own worker
Security Auth bypass, data exposure, injection attacks, leaked secrets Security evidence (scan reports, RBAC config, HTTPS audits) requires specialist interpretation — not just "does it work"
Performance SLO breaches under real load, memory leaks, unindexed queries Functional tests use small datasets and never hit throughput limits — performance evidence requires load test results
Reliability Silent failures, missing observability, no circuit breakers, DR gaps Reliability is about what happens when things go wrong — not covered by happy-path or unit tests
Scalability Architecture that works at 100 users but fails at 10,000 Scaling requires architectural evidence (K8s config, sharding plan, read replicas) — not testable by running tests

When to Use

Run as a pre-release gate check — after implementation is complete and functional tests pass, but before deploying to production.

§ HOW Workflow Steps (5 Steps)

NFR has 5 steps. Steps 1–3 run sequentially to prepare context and evidence before evaluation begins. Step 4 fans out to 4 parallel domain workers. Step 5 aggregates all results into the final report.

STEP 01 Load Context & Knowledge Base step-01-load-context.md
  • ① Check prerequisites — confirm that the implementation is accessible and evidence sources are available. HALT and prompt the user if either is missing — do not proceed until both are confirmed.
  • ② Load knowledge base — read adr-quality-readiness-checklist.md, ci-burn-in.md, test-quality.md, playwright-config.md, error-handling.md.
  • ③ Load context artifacts — read NFR thresholds from tech-spec.md (primary) → PRD.md (secondary) → story or test-design (feature-level). Summarise which sources were found and what evidence is available.
STEP 02 Define NFR Categories & Thresholds step-02-define-thresholds.md
  • ① Select categories — use the 8 standard ADR categories plus any custom_nfr_categories configured in the workflow: Testability · Data Strategy · Scalability · Disaster Recovery · Security · Observability · Quality of Service · Deployability.
  • ② Extract thresholds — for each category, pull the threshold from the sources loaded in Step 1, in priority order: tech-spec → PRD → story/test-design. Never guess a threshold.
  • ③ Build the NFR matrix — for each category, record the threshold or mark it UNKNOWN. Any UNKNOWN threshold means that category will be reported as CONCERNS in Step 4.
STEP 03 Gather Evidence step-03-gather-evidence.md
  • ① Collect evidence per category — scan for measurable data: performance metrics (P95 response times, throughput), security scan reports (OWASP ZAP, Snyk), error rate history, burn-in run results, DR drill records.
  • ② Optional browser capture — if tea_browser_automation is cli or auto, open the target URL, capture network response times, payload sizes, and a screenshot. Store outputs in {test_artifacts}/nfr/. Close the session after capture — never use close-all.
  • ③ Flag missing evidence — any category without concrete evidence is automatically marked CONCERNS. Missing data is never treated as "no issue found."
STEP 04 Orchestrate NFR Evaluation (4 Workers in Parallel) step-04-evaluate-and-score.md
  • ① Resolve execution mode — check config to determine how workers run: agent-team (full parallel subagents) → subagent (single subagent) → sequential (fallback, one at a time).
  • ② Dispatch 4 domain workers — run in parallel: Worker A · Security (04a), Worker B · Performance (04b), Worker C · Reliability (04c), Worker D · Scalability (04d). Each worker independently assesses its domain and outputs a risk level (NONE / LOW / MEDIUM / HIGH) plus PASS / CONCERN / FAIL per sub-category.
  • ③ Sub-step 4E — Aggregate results — read all 4 worker outputs. Apply the worst-case rule: the overall risk level equals the highest risk among the 4 workers. Detect cross-domain risks (e.g. Performance + Scalability compound risk). Compile the prioritised remediation action list.
STEP 05 Generate Report & Validate step-05-generate-report.md
  • ① Write nfr-assessment.md — include: per-category results with evidence summary, overall risk level, remediation actions with urgency (URGENT / HIGH / MEDIUM / LOW), compliance matrix (SOC2 / GDPR / HIPAA / PCI-DSS), and a gate-ready YAML snippet for CI integration.
  • ② Validate the report — check against checklist.md: remove duplicates, verify consistency, fill any N/A sections. Fix all gaps before marking complete.
  • ③ Summarise and recommend next steps — state the overall NFR status, list any critical blockers or waivers required, and recommend the next workflow: run bmad-testarch-trace for coverage gate, or proceed to release gate if NFR is PASS.

Status Values (per Finding)

Each sub-category within a domain receives one of four statuses. Evidence must back every non-N/A finding.

PASS
Properly implemented and verified by evidence. No action required.
CONCERN
Partially implemented, weak, or evidence is missing. Remediation recommended before release.
FAIL
Not implemented or a critical vulnerability exists. Blocker — must fix before release.
N/A
Not applicable to this system type. Excluded from risk calculation.

Risk Levels & Aggregation

Each worker assigns a risk level to its domain. Step 4E aggregates them using the worst-case rule — the overall risk is whichever domain has the highest risk level.

NONE
All sub-categories PASS or N/A. No concerns identified.
LOW
Minor concerns only. No FAIL findings. Acceptable for release with monitoring.
MEDIUM
Some CONCERN findings. Review and plan remediation. Conditional release.
HIGH
One or more FAIL findings. Release blocked until addressed.
Aggregation Rule (Worst-Case)
  • If any domain is HIGH → overall is HIGH
  • If any domain is MEDIUM (and none HIGH) → overall is MEDIUM
  • If all domains are LOW or NONE → overall is LOW
Cross-Domain Risk Detection

Step 4E also checks for risks that span two domains. When both domains have CONCERN or FAIL findings, the combined impact is higher than either domain alone — these are elevated to the top of the remediation list.

  • Performance + Scalability → HIGH — performance problems worsen under load; an API slow at 100 users may be unusable at 10,000.
  • Security + Reliability → CRITICAL — a security breach combined with poor recovery means an attack can cascade into a prolonged outage that is hard to contain.

Gate-Ready YAML Snippet

The report includes a YAML block usable as a release gate condition in CI pipelines — listing which NFR categories passed, which have open concerns, and what waivers (if any) are required for release.

After the NFR Report — Next Actions
  • Overall LOW / NONE → proceed to release; optionally run trace for coverage gate
  • Overall MEDIUM → review CONCERNs, decide which to waive vs fix; conditional release
  • Overall HIGH → release blocked; fix all FAIL findings and re-run NFR assessment
  • FAIL in Security → treat as URGENT; must fix before any other domain concerns are addressed
