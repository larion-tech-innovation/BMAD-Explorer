Dev Story is the implementation workhorse of Phase 4. It takes a story file produced by Create Story and converts it into working, tested code — following TDD strictly (failing tests first, minimal implementation to pass, then refactor). The story file defines all boundaries; the dev agent does not invent scope, consult other documents, or pause for approval.

story-constraints Story File Write Constraints

The dev agent reads the story file completely but may only modify specific sections. Any modification outside these areas is a violation of the workflow.

Permitted Modifications to Story File
YAML Frontmatter
✓ ALLOWED: baseline_commit field only — captured from git rev-parse HEAD at story start. Written once; never overwritten.
Tasks / Subtasks
✓ ALLOWED: Checking boxes from [ ] to [x] only — after all validation gates pass for that task.
Dev Agent Record
✓ ALLOWED: Debug Log, Completion Notes, Agent Model Used. Filled progressively during implementation.
File List
✓ ALLOWED: All new, modified, or deleted files — paths relative to repo root. Updated after each task.
Change Log
✓ ALLOWED: Summary of changes per session. Review follow-up resolutions also recorded here.
Status
✓ ALLOWED: Transition to review at Step 9 completion only — after DoD validation passes.
Everything else
✗ FORBIDDEN: Story text, Acceptance Criteria, Dev Notes, Project Structure Notes, References — read-only for the dev agent.

§ WHY Why TDD and Continuous Execution?

TDD is not a quality preference — it is the mechanism that makes the dev agent's work verifiable. Tests written before code define behaviour unambiguously. Passing tests are the only objective proof that a task is done.

Dimension Without TDD (code-first) With TDD (test-first)
Behaviour definition Implementation defines behaviour — ambiguous ACs resolved however the developer interprets them Tests define behaviour before code is written — ACs become binary pass/fail contracts
Scope control Easy to over-engineer or add speculative features beyond story scope Only code needed to pass the pre-written tests gets written — nothing more
Regression safety Refactoring or neighbouring changes can break things silently Full test suite runs before every task completion — any regression is caught immediately
Completion claim Developer self-reports "done" — no objective verification Task can only be checked [x] when tests ACTUALLY exist and pass 100%
Interruption risk Stopping mid-story leaves partial work; resuming loses context Continuous execution model — no stops until complete or HALT condition
Review quality Reviewer must trace through code to understand intent Tests document intent; reviewer validates behaviour, not guesses at it

tdd-cycle Red-Green-Refactor Cycle per task

The cycle runs once per task/subtask, in order, without skipping. Each phase has a mandatory check before proceeding to the next. The cycle is the only way a task can be marked complete.

TDD Implementation Cycle Repeats for every task/subtask · Cannot be skipped
RED
Write Failing Tests
Red Phase
Write tests for the task's functionality before writing any implementation code. Run tests and confirm they FAIL — a test that passes before implementation is a broken test.
Gate: tests must fail before proceeding
→
GREEN
Implement to Pass
Green Phase
Write the minimum code required to make the tests pass. No extra features. No speculative abstractions. Handle error conditions and edge cases as specified in the task.
Gate: all tests must pass before proceeding
→
BLUE
Refactor
Refactor Phase
Improve code structure, naming, and organisation. Eliminate duplication. Apply architecture patterns from Dev Notes. Tests must remain green throughout.
Gate: all tests still pass after refactor

no-stop-rule The No-Stop Rule

Critical Execution Constraint
Do not stop because of "milestones," "significant progress," or "session boundaries." Continue in a single execution until the story is COMPLETE — meaning all ACs satisfied and all tasks/subtasks checked — unless a HALT condition is triggered or the user gives explicit instruction to stop. Do not schedule a "next session." Do not request review pauses. Only Step 9 (Completion) decides when the story is done.

This constraint exists because partial implementation leaves the codebase in an inconsistent state. A task that is 80% implemented but not passing tests is worse than not started — it is broken code with no safety net. The only valid stopping points are explicit HALT conditions listed below.

§ HOW How to Execute · 10-Step Workflow

Load & Setup — Steps 1–4

01 Find Next Ready Story and Load It

  • User-providedIf a story path is given, load it directly and proceed to task check.
  • Sprint-basedRead the FULL sprint-status.yaml from start to end. Find the first entry matching N-N-name with status ready-for-dev.
  • No sprint fileScan implementation_artifacts for story files with ready-for-dev Status section.
  • If none foundOffer four options: run create-story · validate existing stories · specify story path · review sprint status.
  • Task checkAfter loading: parse all sections, identify first incomplete task [ ]. If no incomplete tasks, jump directly to Step 9.

02 Load Project Context and Story Information

  • GoalLoad project-context.md for coding standards and project-wide patterns.
  • Dev NotesExtract architecture requirements, previous story learnings, and technical specifications from the story file's Dev Notes section. These inform all implementation decisions.

03 Detect Review Continuation

  • Fresh startNo "Senior Developer Review" section in story file → normal implementation flow.
  • Post-reviewIf "Senior Developer Review (AI)" section exists: extract outcome, count unchecked action items, list pending [AI-Review] follow-up tasks.
  • StrategyWhen continuing after review, prioritise [AI-Review] follow-up tasks before resuming regular tasks.

04 Mark Story In-Progress

  • baseline_commitIf story is ready-for-dev and no baseline_commit exists in frontmatter: run git rev-parse HEAD and write it. Written once — never overwritten on subsequent runs.
  • Sprint updateUpdate sprint-status.yaml: story status ready-for-dev → in-progress. If already in-progress, resume without status change.
Implementation Loop — Steps 5–8 (repeats per task)

05 Implement Task (Red-Green-Refactor)

  • AuthorityThe current task/subtask from the story file is the authoritative implementation guide. Follow it exactly — no deviation.
  • RedWrite FAILING tests first. Confirm tests fail before writing any implementation code.
  • GreenWrite minimal code to make tests pass. Handle error conditions and edge cases as specified in the task.
  • RefactorImprove code structure while keeping tests green. Follow architecture patterns and coding standards from Dev Notes.
  • ScopeNever implement anything not mapped to a specific task/subtask. Never proceed to the next task until the current one is complete and tests pass.

06 Author Comprehensive Tests

  • UnitBusiness logic and core functionality introduced or changed by the task.
  • IntegrationComponent interactions specified in story requirements.
  • E2ECritical user flows — only when the story requirements explicitly demand them.
  • CoverageEdge cases and error handling scenarios from story Dev Notes.

07 Run Validations and Tests

  • RegressionRun all existing tests. If any fail: STOP and fix before continuing.
  • New testsRun new tests. If any fail: STOP and fix before continuing.
  • QualityRun linting and static analysis if configured. Validate all quantitative thresholds from ACs explicitly.

08 Validate and Mark Task Complete

  • Never lieA task can be marked [x] ONLY when: tests ACTUALLY EXIST and pass 100% · implementation matches exactly what the task specifies · all related ACs are satisfied · full regression suite passes.
  • File ListUpdate File List with all new, modified, or deleted files after each task.
  • LoopIf more incomplete tasks remain, go back to Step 5. If no tasks remain, proceed to Step 9.
Completion — Steps 9–10

09 Story Completion and Mark for Review

  • VerificationRe-scan the full story: every task and subtask must be [x]. Run the full regression suite. Confirm File List is complete.
  • DoD validationExecute the full Definition of Done checklist (5 categories — see below). Any failure halts.
  • Status updateSet story Status to review. Update sprint-status.yaml: in-progress → review.

10 Completion Communication

  • SummaryReport to user: story ID, key, title, key changes, tests added, files modified, story file path, current status.
  • Skill-levelAsk if user needs explanations on what was implemented and how — depth tailored to {user_skill_level}.
  • Next stepsSuggest: review story · run code-review (preferably with a different LLM) · optionally run bmad:tea:automate for guardrail tests.

halt-conditions HALT Conditions

These are the only valid reasons for the dev agent to stop before a story is complete. Each HALT requires a specific resolution before work can continue.

HALT · Step 5
New dependencies required
"Additional dependencies need user approval" — cannot add packages beyond story specifications without explicit sign-off.
HALT · Step 5
3 consecutive implementation failures
Three failed attempts on the same task without progress. Request guidance before continuing.
HALT · Step 5
Required configuration missing
"Cannot proceed without necessary configuration files" — missing env vars, secrets, or infrastructure that cannot be inferred.
HALT · Step 7
Regression tests fail
Existing tests broken by the current implementation. Identify the breaking change and fix it before any further progress.
HALT · Step 9
Any task is incomplete
Cannot mark story for review while any task or subtask remains unchecked. Complete remaining tasks.
HALT · Step 9
File List incomplete
Every new, modified, or deleted file must be listed with its repo-relative path before handoff to review.
HALT · Step 9
DoD validation fails
One or more Definition of Done checklist items not satisfied. Address each failure and re-validate before marking complete.
HALT · Any step
Story file inaccessible
"Cannot develop story without access to story file" — file cannot be found, read, or parsed at the required path.

review-continuation Review Continuation Mode

When code-review requests changes, it appends a "Senior Developer Review (AI)" section to the story file. Dev Story detects this at Step 3 and switches into review continuation mode.

Review Continuation Behaviour SKILL.md · Step 3
AspectFresh StartReview Continuation
Detection No "Senior Developer Review" section in story "Senior Developer Review (AI)" section present
Task order Follow Tasks/Subtasks in story order Address [AI-Review] follow-ups first, then continue regular tasks
Marking complete Check [x] in Tasks/Subtasks only Check [x] in follow-up task AND find matching action item in Review section and mark it resolved

status-transitions Status Transitions

Dev Story drives a story through two status transitions. Both are written to sprint-status.yaml and to the story file's Status field.

ready-for-dev → Step 4 · start work → in-progress → Step 9 · DoD passed → review
  • ready-for-dev → in-progress: happens at Step 4. Also captures baseline_commit from git at this point.
  • in-progress → review: happens at Step 9 only after all tasks complete, DoD validated, and full test suite passing.
  • If already in-progress when Dev Story starts (resuming): no status change at Step 4 — continues from where it left off.
  • sprint-status.yaml last_updated is refreshed on every status transition.

dod Definition of Done — 5 Categories

Executed at Step 9. Every item in every category must pass before the story moves to review. A story with any open DoD item is not done — it is broken.

Category 01
Context & Requirements
  • Dev Notes contained all necessary technical requirements
  • Implementation follows all architectural requirements from Dev Notes
  • Technical specs (libraries, versions) implemented correctly
  • Previous story learnings incorporated (if applicable)
Category 02
Implementation Completion
  • Every task and subtask marked [x]
  • Implementation satisfies EVERY Acceptance Criterion
  • No ambiguous implementations — clear and unambiguous
  • Edge cases and error conditions addressed
  • Only dependencies specified in story or project-context used
Category 03
Testing & Quality
  • Unit tests for ALL core functionality added/updated
  • Integration tests for component interactions (when required)
  • E2E tests for critical flows (when story demands them)
  • ALL existing tests pass — no regressions
  • Linting and static checks pass (when configured)
  • Tests use project's testing frameworks from Dev Notes
Category 04
Documentation & Tracking
  • File List: every new/modified/deleted file (repo-relative paths)
  • Dev Agent Record: implementation notes and debug log
  • Change Log: clear summary of what changed and why
  • All [AI-Review] follow-ups resolved (if applicable)
  • Only permitted story sections were modified
Category 05
Final Status
  • Story Status set to review
  • Sprint status updated to review (when tracking active)
  • All quality checks and validations passed
  • No blocking issues or incomplete work remaining
  • Implementation summary prepared for user
