§ WHAT Quick Dev · Turn Intent into a Hardened Artifact

Skill code
bmad-quick-dev
Purpose
Implements any user intent — building, fixing, tweaking, refactoring, adding, or modifying code — by producing a clean, working, reviewable artifact that follows the project's existing architecture, patterns, and conventions.
Input types
Free-form intent · story requirement · bug description · existing spec file path (resumes) · external intent doc · tracker ticket
Step architecture
Just-in-time step loading — each step is a self-contained file, loaded only when needed. Sequential enforcement, no skipping. State tracked via spec frontmatter.
Output
spec-{slug}.md in implementation_artifacts/ — a spec that tracks implementation history, task completion, and includes a Suggested Review Order with clickable links for the human reviewer.
Epic support
Detects when intent is an epic story — loads or compiles epic-N-context.md, carries forward previous story's Code Map and Design Notes for continuity.

Quick Dev is the workhorse skill for implementation. It handles everything from a one-liner typo fix to multi-file feature additions. The routing decision — One-Shot vs. Plan-Code-Review — happens at the start of Step 1 based on a single question: is the blast radius truly zero?

scope-standard Scope Standard

Spec Scope Standard — applied in Step 1 (multi-goal check) and Step 2 (token count)
Single Goal
One cohesive user-facing deliverable. Multi-goal = two or more top-level independently shippable goals — each could be reviewed and merged as a separate PR without breaking the others. "Add validation and display errors" is ONE goal. "Add dark mode toggle AND refactor auth to JWT" are two.
Token Range
900 – 1600 tokens is the target. Below 900: ambiguity risk. Above 1600: context-rot during implementation. Neither is a hard gate — user can override either direction.
Multi-goal → Split prompt
Present detected goals as a bullet list, explain coupling risks, recommend which to tackle first. Options: [S] Split — defer others to deferred-work.md · [K] Keep all — accept the risks.
Over-token → Split prompt
Show token count. Options: [S] Split — carve off secondary goals · [K] Keep full spec. On Split: regenerate the spec for the narrowed scope — don't surgically carve sections.

§ WHY Why Spec-First? · The Problem with "Just Write the Code"

Direct implementation is fast on the first pass. It becomes slow on the second, third, and fourth — when the reviewer asks "what was the intent here?", or when a test fails six months later and no one can reconstruct the decision.

The spec file is not documentation for its own sake. It answers three questions that code alone cannot: What was the intent? (the problem and the approach), What should have happened? (the acceptance criteria, testable in Given/When/Then), and What was changed and why? (the Spec Change Log, updated on each review loop). Without these, an adversarial review has nothing to audit against — it can only find syntax errors, not intent violations.

The Suggested Review Order solves a separate problem: when a reviewer opens a diff, they see changes by file, not by concept. A concern-ordered review trail — "start here, then look at this, then peripherals" — gets the reviewer oriented in under a minute instead of ten.

ad-hoc-vs-quick-dev Ad-Hoc Coding vs. Quick Dev

Dimension Ad-hoc "just write the code" bmad-quick-dev
Intent capture In the developer's head, at best in a commit message — lost when the author leaves or forgets Captured in the spec before code is written. The frozen section locks intent on approval — can only change with explicit human edit.
Multi-goal scope Often tacitly accepts "while I'm in here…" additions that inflate blast radius and make review harder Multi-goal check at Step 1 surfaces independent goals, explains coupling risks, defers extras to deferred-work.md
Review Reviewer reads a diff without knowing what was intended — finds obvious bugs, misses intent violations Three adversarial subagents: Blind Hunter catches objective bugs; Edge Case Hunter catches boundary failures; Acceptance Auditor checks every AC
Review loop Comments → fix → re-review cycle in PR comments — unstructured, may loop indefinitely Findings classified (intent_gap / bad_spec / patch / defer) — each category has a deterministic action. Loop limit of 5 before escalation.
Navigation Reviewer navigates the diff top-to-bottom by file — starts in a random location relative to design intent Suggested Review Order: concern-grouped, entry-point-first, every stop a clickable link. Ctrl+click to jump directly to the code.

rfd-standard Ready for Development Standard

The spec must meet all four criteria before it becomes ready-for-dev and implementation begins. Checked at Step 2 self-review before presenting to the user at Checkpoint 1.

A
Actionable
Every task specifies a file path and a specific action. Nothing vague: "update the auth module" fails; "add null check in auth/middleware.ts:42" passes.
L
Logical
Tasks ordered by dependency. A task that reads a value another task writes comes after it. No circular dependencies, no implicit ordering assumptions.
T
Testable
All acceptance criteria expressed in Given/When/Then format. No "should work correctly" — the behavior, the trigger, and the expected outcome must all be explicit.
C
Complete
No placeholders, no TBDs, no "to be determined" anywhere in the spec. If something is unknown, the spec is not complete — halt and ask the user before proceeding.

§ HOW Two Routes · Blast Radius is the Deciding Factor

The routing decision is made at the end of Step 1, after intent is clarified. One question: is there any plausible path by which this change causes unintended consequences elsewhere? If the answer is definitively no — and you're certain — go One-Shot. When uncertain, default to Plan-Code-Review.

Route A
One-Shot
Criterion: zero blast radius
No plausible path by which this change causes unintended consequences elsewhere. Clear intent, no architectural decisions, no risk of touching something unexpected. Implement immediately, then review, then generate a spec trace and present.
Flow: Implement → Adversarial Review → Classify (patch/defer/reject only) → Spec Trace → Commit → Present
Route B · Default
Plan-Code-Review
Criterion: everything else
Any change that could ripple, any architectural decision, any uncertainty about blast radius. The full 5-step workflow: plan first, get user approval, implement, review adversarially, present with a Suggested Review Order.
Flow: Plan → Checkpoint → Implement → Adversarial Review → Finding Classification → Present

one-shot-detail One-Shot Path Detail

  • ImplementSync sprint-status to in-progress. Implement the clarified intent directly — no spec written first.
  • Adversarial reviewLaunch bmad-review-adversarial-general in a subagent with the changed files. Subagent gets NO conversation context. If no subagents available: write files to a review prompt file and HALT — ask user to run in a separate session.
  • Classify findingsThree categories only for One-Shot: patch (auto-fix immediately) · defer (append to deferred-work.md) · reject (drop silently). If a finding is caused by this change but too significant for a trivial patch: HALT and present to user for decision.
  • Generate spec traceWrite a minimal spec: frontmatter (status: done, route: one-shot) + Title + Intent (Problem and Approach) + Suggested Review Order. No Tasks section — the code is already done.
  • Commit and presentConventional commit message from intent. Open spec in VS Code (code -r). Display: commit hash, files changed with descriptions, review findings breakdown, navigation tip. Offer to push or create PR.

finding-cascade Finding Classification Cascade Step 4 · adversarial review

All findings from the three reviewers are deduplicated and classified in this order. The cascade is a priority waterfall: the first two tiers trigger loopbacks — lower tiers only matter if neither is present. When a loopback fires, patch findings are the only ones that survive across iterations.

review-order Suggested Review Order — Step 5 Output

Appended to the spec file after all code changes are finalized. Concern-ordered — not file-by-file. The human opens the spec in VS Code and Ctrl+clicks each stop to jump directly to the code.

  • Entry point firstLead with the single highest-leverage file:line that gives the reviewer the design intent at a glance. Usually the main changed function or the primary API entry point.
  • Order by concernGroup stops by conceptual concern — "validation logic", "schema change", "UI binding". A single file may appear under multiple concerns. Order within each concern: most important → supporting.
  • Bias toward riskHigher-risk or boundary-crossing stops (cross-language calls, IPC, host↔device, config flows) rank higher within their concern group.
  • Peripherals lastTests, config, types, and supporting changes come at the end — after the human understands the core change.
  • Clickable linksEvery stop is a spec-file-relative markdown link: [file.ts:42](../../src/path/to/file.ts#L42). Paths computed dynamically — never hardcoded depth. Each stop gets one ultra-concise framing line (≤15 words): why this approach, what it achieves.
